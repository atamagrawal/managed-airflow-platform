## 1. What the Runtime Plane is

The Runtime Plane is the actual Airflow installation — the component that schedules and executes customer DAGs. It is the only part of the system that runs customer code. Everything above it (the Control Plane and Data Plane) exists to create, configure, and manage it.

Every deployment has exactly one Runtime Plane. The Runtime Plane is isolated from all other deployments, from the Data Plane, and from the Control Plane. It has no credentials to reach the Control Plane. It does not know it is managed — it sees only the Kubernetes resources and cloud services the Data Plane has placed in front of it.

```
CONTROL PLANE (decisions, desired state, identity, billing)
      │
      │  outbound gRPC — Data Plane initiates
      ▼
DATA PLANE (reconciliation, provisioning, health monitoring)
      │
      │  Kubernetes API (in-cluster)
      ▼
RUNTIME PLANE (Airflow — schedules and runs DAGs)
  ┌────────────────────────────────────────────┐
  │  Namespace: dep-{deployment_id}            │
  │                                            │
  │  Webserver · Scheduler · Triggerer         │
  │  Workers (Celery) or Pod-per-task (K8s)    │
  │                                            │
  │  RDS (metadata DB)                         │
  │  ElastiCache Redis (broker)                │
  │  S3 / EFS (DAG storage)                    │
  └────────────────────────────────────────────┘
```

The Runtime Plane is deliberately narrow in its responsibilities:

- Schedule DAGs and tasks according to their definitions.
- Execute tasks using the configured executor.
- Serve the Airflow web UI and REST API to authenticated users.
- Write task logs to remote storage.
- Report its own health via the scheduler heartbeat and webserver health endpoint.

It does not manage its own lifecycle, provision infrastructure, route external traffic, or store secrets. All of that is handled by the layers above it.

## 2. Namespace anatomy

Each deployment is a Kubernetes namespace named `dep-{deployment_id}`. The Deployment Controller creates and owns all resources inside it via a Helm release named `airflow-{deployment_id}`.

**ResourceQuota** on the namespace is computed **for this deployment** from `desired_state.resource_profile` (and component layout — workers, scheduler, webserver, triggerer). Admission enforces that those totals stay within org/workspace ceilings for the deployment’s isolation tier; the quota is **not** a tier-wide or cluster-wide pool budget.

```
Namespace: dep-{deployment_id}
│
├── ServiceAccount: airflow-{deployment_id}
│     Annotation: eks.amazonaws.com/role-arn = arn:aws:iam::{acct}:role/{dp_id}-{dep_id}
│     Permission boundary: secretsmanager:GetSecretValue on /flowdeck/{dep_id}/* only
│     Used by: all Airflow pods (webserver, scheduler, triggerer, workers)
│
├── ResourceQuota: airflow-quota
│     requests.cpu:     {namespace_quota.requests_cpu}
│     requests.memory:  {namespace_quota.requests_memory}
│     limits.cpu:       {namespace_quota.limits_cpu}
│     limits.memory:    {namespace_quota.limits_memory}
│     count/pods:       {namespace_quota.max_pods}
│
├── LimitRange: airflow-limits
│     Container default request: 100m CPU, 256Mi memory
│     Container default limit:   2000m CPU, 4Gi memory
│     Container max:             8000m CPU, 32Gi memory
│
├── NetworkPolicy: default-deny
│     Ingress: deny all except from ingress-controller namespace
│     Egress:  allow {rds_sg_cidr}:5432        (metadata DB)
│              allow {redis_sg_cidr}:6379      (broker)
│              allow 169.254.169.254:80        (IMDS/IRSA)
│              allow s3.{region}.amazonaws.com (DAG bundle + logs)
│              allow secretsmanager.{region}.amazonaws.com (via VPC endpoint)
│     Block:   all cross-namespace pod traffic
│              Kubernetes API server (no direct access from workload pods)
│
├── Deployment: airflow-webserver
│     Replicas: 1 (HA: 2 for Pro/Enterprise)
│     image: apache/airflow:{version} or OCI bundle digest
│     Port: 8080
│     readinessProbe: GET /health → 200
│     env: AIRFLOW__DATABASE__SQL_ALCHEMY_CONN (from Secret)
│
├── StatefulSet: airflow-scheduler
│     Replicas: 1 (HA: 2 with HA scheduler for Pro/Enterprise)
│     image: apache/airflow:{version} or OCI bundle digest
│     livenessProbe: exec airflow jobs check --job-type SchedulerJob
│     volumeMount: /opt/airflow/logs (EmptyDir — logs written to S3)
│
├── Deployment: airflow-triggerer
│     Replicas: 1
│     image: apache/airflow:{version}
│
├── Deployment: airflow-worker                    ← CeleryExecutor only
│     Replicas: {min_workers} initially
│     image: apache/airflow:{version} or OCI bundle digest
│     ScaledObject: keda-worker-scaler
│     preStop hook: airflow celery worker --drain
│
├── ConfigMap: airflow-config
│     airflow.cfg overrides (non-sensitive settings)
│     AIRFLOW__CORE__EXECUTOR, AIRFLOW__CORE__PARALLELISM, etc.
│
└── Secrets: (populated by Secret Distributor — NOT by Helm)
      airflow-db-conn:       AIRFLOW__DATABASE__SQL_ALCHEMY_CONN
      airflow-broker-url:    AIRFLOW__CELERY__BROKER_URL
      airflow-fernet-key:    AIRFLOW__CORE__FERNET_KEY
      airflow-secret-key:    AIRFLOW__WEBSERVER__SECRET_KEY
      airflow-connections:   per-connection AIRFLOW_CONN_{NAME} env vars
      airflow-variables:     sensitive Airflow Variables
```

The Helm release owns all of these except the Secrets, which are managed exclusively by the Secret Distributor. This separation is intentional: Helm values are stored in the cluster (in a ConfigMap under the release name) and must not contain plaintext secrets.

## 3. Airflow components

### 3.1 Webserver

The webserver serves the Airflow UI and the REST API (`/api/v1/`). It is stateless — it reads all state from the metadata database and writes task actions (trigger, clear, pause) back to it. Multiple replicas can run behind the same ingress without coordination.

```
External user request
      │
      ▼ HTTPS → ALB → ingress controller (JWT validation at ALB)
      │
      ▼ HTTP → airflow-webserver pod :8080
      │
      ├── Static UI: served from within the image
      ├── REST API: reads from / writes to RDS metadata DB
      └── Log viewer: streams task log files from S3
```

**JWT validation at ALB:** Flowdeck's Identity Service enforces authentication before traffic reaches the webserver. The webserver's own auth (`AUTH_BACKEND`) is set to `airflow.api.auth.backend.basic_auth` as a secondary layer but is not the primary gate. This means even a misconfigured Airflow auth setting does not expose the UI to unauthenticated users.

**Session key:** `AIRFLOW__WEBSERVER__SECRET_KEY` is unique per deployment (materialised by the Secret Distributor from Secrets Manager). Without a consistent secret key across replicas, login sessions would be invalidated on each request round-robin. The same key is used by all webserver replicas.

**HA configuration (Pro/Enterprise):** Two webserver replicas run behind the ingress. The ingress uses sticky sessions (cookie-based) to route a user to the same replica for the duration of their session. Session state is in the metadata DB, not in the pod, so a pod restart does not log the user out.

### 3.2 Scheduler

The scheduler is the most critical component of the Runtime Plane. It performs three functions continuously:

1. **DAG parsing:** Reads DAG files from the DAG bundle (OCI image or sync volume), parses them using the `DagFileProcessorManager`, and writes parsed DAG definitions to the metadata DB. Parse errors are surfaced as `import_errors` in the DB and shown in the UI.

2. **Task scheduling:** On each scheduling loop, evaluates which DAG runs are due (based on schedule intervals), checks DAG run limits and concurrency, creates task instances in `scheduled` state, and hands them to the executor.

3. **Heartbeat:** Writes a heartbeat record to the `scheduler_job` table every 5 seconds. The Data Plane's Health Monitor queries this table every 30 seconds. A missing heartbeat for 2 consecutive polls triggers automated remediation.

```
Scheduler loop (runs continuously):
  ┌─────────────────────────────────────────────────────┐
  │  1. DAG parsing (subprocess pool)                   │
  │     DagFileProcessorManager scans /opt/airflow/dags │
  │     Spawns subprocesses to parse each file          │
  │     Writes SerializedDag to metadata DB             │
  │     Writes DagParsingStats (duration, error count)  │
  │                                                     │
  │  2. Scheduling loop (main thread, ~1s tick)         │
  │     Read active DAGs from metadata DB               │
  │     Evaluate next_dagrun_create_after for each DAG  │
  │     Create DagRun + TaskInstances (state: scheduled)│
  │     Check DAG concurrency limits                    │
  │     Check task pool limits                          │
  │     Hand tasks to executor queue                    │
  │                                                     │
  │  3. Executor heartbeat                              │
  │     CeleryExecutor: sync task state from Redis      │
  │     KubernetesExecutor: sync pod states from k8s    │
  │     Mark completed/failed tasks in metadata DB      │
  │                                                     │
  │  4. Scheduler heartbeat (every 5s)                  │
  │     UPDATE scheduler_job SET heartbeat_date = now() │
  └─────────────────────────────────────────────────────┘
```

**HA scheduler (Pro/Enterprise):** Two scheduler pods run simultaneously. Airflow's HA scheduler uses row-level locking in the metadata DB to prevent double-scheduling. Each scheduler processes a different subset of DAGs based on which acquired the lock first. Both write heartbeats, so the Health Monitor sees two heartbeats — it raises an alarm only if both are missing.

**DAG parsing in OCI bundle mode:** When using OCI bundles, the DAG files are baked into the image and mounted at `/opt/airflow/dags` via an `emptyDir` volume populated by an init container that extracts the DAG bundle. The scheduler parses from this path.

### 3.3 Triggerer

The Triggerer runs deferrable operators — tasks that wait for an external event (an HTTP response, a sensor condition, a timer) without occupying a worker slot. A deferrable task suspends itself, hands a trigger specification to the Triggerer, and releases its worker. The Triggerer evaluates the trigger conditions asynchronously using Python's `asyncio` event loop and resumes the task when the condition is met.

```
Deferrable task flow:
  Worker executes task
      │
      │ task calls self.defer(trigger=MyTrigger(...))
      ▼
  Task instance state → DEFERRED
  Worker slot released (task no longer occupies worker memory)
      │
      ▼
  Triggerer picks up the trigger specification from metadata DB
  Runs trigger coroutine in asyncio event loop
      │
      ├── Condition not met → keep polling (no worker used)
      │
      └── Condition met → reschedule task instance → SCHEDULED
          Worker picks it up and resumes from where it deferred
```

The Triggerer runs as a single process with an async event loop that can manage thousands of concurrent triggers without one worker per trigger. It writes its own heartbeat to `trigger_job` in the metadata DB. The Health Monitor checks this heartbeat alongside the scheduler heartbeat.

### 3.4 Workers

Workers execute tasks. Their internals differ by executor type:

**CeleryExecutor workers:** Long-running pods that register with the Celery broker (ElastiCache Redis), pick tasks from the queue, execute them in subprocesses, and report results. Each worker pod runs multiple tasks concurrently (bounded by `AIRFLOW__CELERY__WORKER_CONCURRENCY`, default 16). Workers are managed as a Kubernetes `Deployment` with KEDA autoscaling.

**KubernetesExecutor workers:** Each task gets its own pod, spawned by the scheduler directly via the Kubernetes API. The pod runs the task and terminates. There are no persistent worker pods — worker "count" is the count of active task pods in the namespace.

The executor choice affects autoscaling, startup latency, resource isolation, and operational model significantly. See Section 5 for the full comparison.

## 4. Storage layer

### 4.1 Metadata database — RDS PostgreSQL

The metadata database is Airflow's primary data store. It holds every persistent state record: DAG definitions, DAG runs, task instances, task logs references, connections, variables, pools, XCom values, users, and roles.

```
RDS instance per deployment:
  Engine:        PostgreSQL 15
  Instance type: db.t4g.micro (Starter), db.t4g.small (Pro),
                 db.r7g.large+ (Enterprise, configurable)
  Multi-AZ:      No (Starter), Yes (Pro/Enterprise)
  Storage:       gp3, auto-scaling enabled
  Encryption:    at rest with KMS (per-deployment or per-region key)
  Connection:    via RDS Proxy (PgBouncer in transaction mode for Starter/Pro)
                 direct connection (Enterprise)
  Backup:        automated daily snapshot + PITR (7-day Starter, 30-day Pro/Enterprise)
```

**RDS Proxy / PgBouncer:** Airflow components (webserver, scheduler, triggerer, workers) each maintain their own SQLAlchemy connection pool. In a deployment with 20 workers at 4 connections each, that is 80+ connections to RDS. PostgreSQL's connection limit on a t4g.micro is ~100. Without connection pooling, worker scale-up would saturate the DB. PgBouncer in transaction mode multiplexes many application connections onto a small number of actual DB connections, keeping RDS connection count bounded regardless of worker count.

**Schema management:** Airflow manages its own schema via Alembic migrations. On startup, the scheduler runs `airflow db migrate` if the DB version is behind the Airflow version. The Deployment Controller's Phase 3 validation waits for this migration to complete before marking the deployment `RUNNING`. Migrations are forward-only; a downgrade requires a DB restore.

### 4.2 Celery broker — ElastiCache Redis

The Celery broker receives tasks from the scheduler, queues them, and delivers them to workers. It also serves as the Celery result backend, storing task execution state (success, failure, retry count) that the scheduler polls.

```
ElastiCache Redis per deployment (CeleryExecutor only):
  Engine:   Redis 7
  Mode:     Single-node (Starter), cluster mode (Enterprise)
  Instance: cache.t4g.micro (Starter), configurable (Pro/Enterprise)
  Encryption: in-transit TLS, at-rest with KMS
  Persistence: not enabled — the broker is ephemeral state
               (task results are written to RDS, not Redis)

Celery queue model:
  Default queue: "celery" (all tasks unless routed elsewhere)
  Priority queues: supported via queue routing in DAG/task definition
  Dead letter: failed tasks are marked in metadata DB, not re-queued
```

**Why Redis loss is recoverable:** Redis holds tasks that are queued but not yet picked up by a worker. If Redis is lost, those queued tasks are lost too — but they are in `queued` state in the metadata DB. On Redis restart, the scheduler re-queues them on the next scheduling loop. Tasks already running in workers continue to completion (they pulled the task from Redis before it went down and hold state in memory).

### 4.3 DAG storage — S3 and OCI bundles

DAGs are delivered to the Runtime Plane primarily as OCI container images (see Section 6). The image is the DAG bundle. No separate DAG storage volume is required when using OCI bundles — the DAGs are baked into the image and extracted at pod startup.

For git-sync and S3-sync delivery paths:

```
git-sync path:
  A sidecar container in the scheduler and worker pods runs git-sync,
  cloning the configured repo branch/tag into a shared emptyDir volume
  mounted at /opt/airflow/dags.
  Pull interval: configurable (default 60s).
  Authentication: SSH key or HTTPS token from Secrets Manager.

S3 sync path:
  A sidecar container polls the configured S3 prefix, downloads changed
  .py files into a shared emptyDir volume.
  Integrity check: SHA-256 of each file verified against a manifest.
  Authentication: IRSA (deployment's IAM role, scoped to its S3 bucket).
```

**Task logs:** Task logs are written locally to `{log_folder}/{dag_id}/{run_id}/{task_id}/` and then uploaded to S3 asynchronously by the `RemoteFileTaskHandler`. The webserver streams them from S3 via pre-signed URLs. Local log storage is an `emptyDir` — it does not survive pod termination. If a worker pod is killed before the log upload completes, the partial log already uploaded is accessible; the rest is lost. This is an acceptable trade-off — log completeness is a best-effort property, not a durability guarantee.

## 5. Executor model

The executor determines how the scheduler hands tasks to compute resources. The choice is set in the deployment's desired state and cannot be changed without a deployment update (which triggers a rolling restart of all components).

### 5.1 CeleryExecutor

```
Scheduler
  │ publishes task to Redis queue
  ▼
ElastiCache Redis (broker)
  │ task claimed by a worker
  ▼
Worker pod
  │ runs task in a subprocess
  │ writes result to Redis result backend
  ▼
Scheduler (polls Redis)
  │ marks task instance SUCCEEDED / FAILED in metadata DB
  ▼
RDS metadata DB
```

**Characteristics:**
- Workers are long-running pods, pre-warmed and ready to accept tasks immediately.
- Task startup latency: ~100ms (subprocess spawn within existing worker pod).
- Multiple tasks run concurrently within one worker pod (up to `WORKER_CONCURRENCY`).
- All tasks in a pod share the same Python environment, system libraries, and network identity.
- Worker pods scale horizontally based on queue depth (KEDA).
- Scale-to-zero is supported for development deployments.

**When to use:** Default for most workloads. Suited for tasks with similar resource requirements, tasks that benefit from process-level reuse (shared connections, cached data), and workloads where startup latency matters.

### 5.2 KubernetesExecutor

```
Scheduler
  │ calls Kubernetes API: create Pod in dep-{deployment_id} namespace
  ▼
Kubernetes schedules Pod on a node
  │ pod starts → pulls image → runs task → exits
  ▼
Scheduler (watches Pod status via Kubernetes API)
  │ marks task instance SUCCEEDED / FAILED in metadata DB
  ▼
RDS metadata DB
```

**Characteristics:**
- Each task runs in its own pod. Zero shared state between tasks at the OS level.
- Task startup latency: 30–90 seconds (pod scheduling + image pull). Reduced to ~5s with pre-pulled image cache.
- Each task pod can request its own CPU/memory profile via `executor_config` in the task definition.
- Each task pod can carry its own IRSA annotation for per-task IAM scoping.
- No persistent worker Deployment — no KEDA ScaledObject. Node scaling (Karpenter) handles capacity.
- No Celery broker — Redis is not provisioned for KubernetesExecutor deployments.

**When to use:** Workloads with heterogeneous resource requirements (one task needs 1 vCPU, another needs 16 vCPU + GPU), per-task network or IAM isolation requirements, or tasks that need guaranteed clean environments with no shared library state.

### 5.3 Comparison

| Dimension | CeleryExecutor | KubernetesExecutor |
|-----------|---------------|-------------------|
| Task startup latency | ~100ms | 30–90s (cold), ~5s (warm cache) |
| Concurrency model | Many tasks per worker pod | One task per pod |
| Resource isolation | Process-level (shared pod) | Pod-level (fully isolated) |
| Per-task resource config | Not supported | Supported via `executor_config` |
| Per-task IAM role | Not supported | Supported via pod annotation |
| Dependency on Redis broker | Yes | No |
| Worker autoscaling | KEDA (queue depth) | Karpenter only (pod scheduling) |
| Scale-to-zero | Yes (worker Deployment to 0) | Always at zero between tasks |
| Best for | Most production workloads | ML pipelines, heterogeneous compute, strict isolation |

### 5.4 LocalExecutor (development only)

A third executor, `LocalExecutor`, runs tasks as subprocesses of the scheduler in the same pod. It requires no broker and no worker pods. It is supported only for development deployments (Starter tier with `dev` profile). It is not available for production workloads because it ties task execution to scheduler availability and has no horizontal scaling.

## 6. DAG delivery pipeline

How DAG code reaches a running deployment is a first-class architectural concern. The delivery path determines reproducibility, rollback capability, supply-chain integrity, and the operational model for CI/CD.

### 6.1 OCI bundle (primary path)

```
Customer CI pipeline:
  1. docker build -t {ecr_repo}/dags:{git_sha} .
       Dockerfile FROM apache/airflow:{version}
       COPY dags/ /opt/airflow/dags/
       COPY requirements.txt .
       RUN pip install -r requirements.txt --no-cache-dir

  2. cosign sign --key {signing_key} {ecr_repo}/dags:{git_sha}@{digest}

  3. POST /v1/deployments/{id}/dags
       { "image": "{ecr_repo}/dags:{git_sha}@{digest}",
         "signed_by": "{cosign_key_id}" }

Flowdeck API (Control Plane):
  4. Verify cosign signature against org's registered public key
  5. Write new desired_state: dag_image = digest
  6. Increment desired_state_version
  7. Publish UPDATE command to NATS

Data Plane:
  8. Deployment Controller receives UPDATE command
  9. Computes diff: only dag_image changed → Phase 2 only (Helm upgrade)
 10. helm upgrade airflow-{dep_id} --set image.dag_bundle={digest}

Kubernetes rolling update:
 11. New scheduler pod starts
       init container: docker pull {digest} → extract /opt/airflow/dags/
       airflow dags list --output json
       If exit code ≠ 0 → readinessProbe fails → rolling update HALTS
       Old scheduler pod stays RUNNING
 12. New worker pods start (same flow)
 13. All pods pass readinessProbe → Controller transitions to RUNNING
 14. Audit entry: { dag_image: digest, applied_at, operator_id, git_sha }
```

**Rollback on parse failure:** Helm's `--atomic` flag means a failed rolling update triggers an automatic rollback to the previous release. The deployment stays on the last known-good image. The customer sees a `DAG_PARSE_FAILED` event with the full parser stderr output.

**Image digest as truth:** Every pod in the deployment runs the same image digest. The audit trail records `dag_run_id → desired_state_version → dag_image_digest → git_sha`. This chain is exact — a customer can reconstruct which exact DAG code produced a given result at any point in time.

### 6.2 git-sync (compatibility path)

git-sync is a sidecar container that polls a Git repository and writes DAG files to a shared volume. It is supported but not recommended for production:

```
Desired state fields:
  dag_source.type: GIT
  dag_source.repo: https://github.com/org/repo.git
  dag_source.ref: main  (branch) or v1.2.3 (tag)
  dag_source.secret_ref: arn:aws:secretsmanager:...  (SSH key or token)

Deployment Controller renders Helm values:
  gitSync.enabled: true
  gitSync.repo: {repo}
  gitSync.branch: {ref}
  gitSync.credentialsSecret: airflow-git-cred

Sidecar behaviour:
  - Runs in scheduler and worker pods (shared emptyDir volume)
  - Polls every 60 seconds (configurable)
  - On change: git pull → new files appear in /opt/airflow/dags/
  - Scheduler re-parses changed files automatically
  - NO parse validation gate before files go live (risk vs OCI bundle)
  - NO atomic rollback on error
```

**Key limitation:** git-sync has no parse gate. A broken DAG pushed to the branch is immediately visible to the scheduler and will produce parse errors in the UI. There is no equivalent of the OCI rolling-update halt. CI/CD enforcement at the workspace level (`require_oci_bundles: true`) can block git-sync in production workspaces.

### 6.3 S3 sync (compatibility path)

Similar to git-sync but pulls `.py` files from an S3 prefix. The Deployment Controller provisions the deployment's S3 bucket with a dedicated prefix (`/dags/`) and an IAM policy allowing the customer's CI pipeline to write to it. The sidecar polls the prefix, downloads changed files, and verifies their SHA-256 against a manifest file at `{prefix}/manifest.json`.

## 7. Task execution lifecycle

A task moves through a well-defined state machine from creation to completion. Understanding this lifecycle is essential for diagnosing stuck or failed tasks.

```
DAG Run triggered (manual, scheduled, or via API)
      │
      ▼
TaskInstance created: NONE
      │
      ▼ Scheduler evaluates dependencies (upstream tasks, pools, concurrency)
      │
      ├── dependencies not met → UPSTREAM_FAILED or waiting
      │
      └── dependencies met
            │
            ▼
      TaskInstance: SCHEDULED
            │
            ▼ Executor picks up the task
            │
            ▼
      TaskInstance: QUEUED
      (for Celery: pushed to Redis queue)
      (for Kubernetes: pod created)
            │
            ▼ Worker begins execution
            │
            ▼
      TaskInstance: RUNNING
      (heartbeat written to metadata DB every 30s by worker)
            │
            ├── Task calls self.defer() ──► DEFERRED
            │                               (Triggerer takes over)
            │                               (resumed → SCHEDULED again)
            │
            ├── Task raises AirflowException ──► FAILED
            │         │
            │         └── retries remaining? ──► UP_FOR_RETRY ──► SCHEDULED
            │
            ├── No heartbeat for {scheduler_zombie_task_threshold}s ──► ZOMBIE
            │         │
            │         └── Scheduler marks FAILED, optionally retries
            │
            └── Task returns normally ──► SUCCESS
```

### 7.1 Zombie task detection

If a worker pod dies without reporting task completion (e.g., node failure, OOM kill), the task instance stays in `RUNNING` state indefinitely. The scheduler detects this by checking for task instances whose last heartbeat is older than `AIRFLOW__SCHEDULER__SCHEDULER_ZOMBIE_TASK_THRESHOLD` (default: 300 seconds). These are marked as `ZOMBIE` and then immediately transitioned to `FAILED`. If the task has retries remaining, it enters `UP_FOR_RETRY` and is re-queued.

### 7.2 Task pools

Airflow pools limit the number of concurrently running tasks globally or for specific task groups, independent of worker count. A pool named `database_pool` with 5 slots allows at most 5 tasks assigned to that pool to run simultaneously across all DAGs.

Pools are defined in the metadata DB and managed through the Airflow UI or REST API. They are a logical concurrency control — they do not map directly to Kubernetes resources.

### 7.3 DAG run concurrency

Three concurrency limits apply simultaneously:

| Limit | Config | Scope |
|-------|--------|-------|
| `max_active_runs_per_dag` | Per DAG definition | How many DagRun instances of one DAG can run at once |
| `max_active_tasks` | `AIRFLOW__CORE__MAX_ACTIVE_TASKS_PER_DAG` | Tasks running across all runs of one DAG |
| `parallelism` | `AIRFLOW__CORE__PARALLELISM` | Total tasks running across ALL DAGs in the deployment |

The `parallelism` setting is derived from the deployment's resource profile and is set by the Deployment Controller in the Helm values. It is not customer-configurable directly — it scales with the worker count.

## 8. Deferrable operators and the Triggerer

Deferrable operators allow a task to wait for an external condition without occupying a worker slot. This is important for sensor-heavy DAGs where traditional sensors spin-poll in a worker, consuming a worker slot for the entire wait duration.

### 8.1 How deferral works

```
Worker pod executes task
  │
  │ task.defer(
  │   trigger=TimeDeltaTrigger(timedelta(minutes=10)),
  │   method_name="execute_complete"
  │ )
  ▼
Worker:
  - Serialises trigger specification to metadata DB
  - Sets TaskInstance state → DEFERRED
  - Returns from execute() without raising an exception
  - Worker slot freed immediately

Triggerer (async event loop):
  - Polls metadata DB for DEFERRED tasks
  - Instantiates trigger object, runs coroutine in asyncio loop
  - When trigger fires:
      set TaskInstance → SCHEDULED
      set TaskInstance.next_method = "execute_complete"
      set TaskInstance.next_kwargs = {trigger_event}

Scheduler:
  - Picks up SCHEDULED task instance
  - Hands to executor

Worker pod:
  - task.execute_complete(context, event=trigger_event)
  - Task resumes from the deferred point with the trigger event
```

### 8.2 Triggerer capacity

The Triggerer is a single-process async event loop. It can manage thousands of concurrent trigger coroutines without one thread per trigger. The limiting factor is memory — each trigger coroutine holds state in memory.

The Deployment Controller sets `AIRFLOW__TRIGGERER__DEFAULT_CAPACITY` from the deployment's resource profile. For Pro/Enterprise deployments with many deferrable operators, the Triggerer pod's memory limit should be sized accordingly (typically 1–4 Gi for hundreds of concurrent triggers).

## 9. Autoscaling

The Runtime Plane has two independent autoscaling loops. They operate at different levels and answer different questions.

### 9.1 Worker autoscaling (KEDA — CeleryExecutor only)

KEDA watches the Celery queue depth in the deployment's ElastiCache Redis and scales the worker `Deployment` replica count:

```
KEDA ScaledObject (per deployment):
  scaleTargetRef:    airflow-worker Deployment
  minReplicaCount:   {desired_state.min_workers}
  maxReplicaCount:   {desired_state.max_workers}
  cooldownPeriod:    120s
  triggers:
    - type: redis
      address: {elasticache_endpoint}:6379
      listName: celery
      listLength: "5"          ← target tasks per worker
      enableTLS: "true"

Scale-up:  queue_depth > listLength × current_workers
           new_replicas = ceil(queue_depth / listLength)

Scale-down: queue_depth < listLength × current_workers
            for cooldownPeriod
            → SIGTERM to worker pod
            → preStop: airflow celery worker --drain
               (stop accepting tasks, wait for in-flight tasks)
            → pod exits after drain or tier_drain_timeout
```

KubernetesExecutor deployments have no worker Deployment and no KEDA ScaledObject. Task parallelism is bounded by the namespace `ResourceQuota` (`count/pods`).

### 9.2 Node autoscaling (Karpenter)

When new worker pods cannot be scheduled (no node capacity), Karpenter provisions nodes to fit them. This is independent of KEDA and operates at the EKS cluster level. From the Runtime Plane's perspective, nodes appear when pods need them and disappear when pods are gone — the Runtime Plane never interacts with Karpenter directly.

### 9.3 Scheduler-driven parallelism

The scheduler's `parallelism` setting (`AIRFLOW__CORE__PARALLELISM`) acts as a logical ceiling on concurrently running tasks, independent of physical worker count. This prevents a burst of queued tasks from spawning more workers than the deployment's quota allows.

```
Effective concurrency =
  min(parallelism, max_workers × worker_concurrency, pool_slots)
```

The Deployment Controller derives `parallelism` from the desired state's resource profile:

```
parallelism = max_workers × worker_concurrency × 0.9
              (0.9 headroom to avoid hitting the ceiling exactly)
```

## 10. Health model

The Data Plane's Health Monitor checks the Runtime Plane every 30 seconds. Each component has its own check, and results roll up to a deployment-level health state.

```
┌─────────────────────────────────────────────────────────┐
│  COMPONENT CHECKS (every 30s per deployment)            │
├─────────────────────────────────────────────────────────┤
│  Webserver:  GET https://{svc}/health → 200 within 5s  │
│  Scheduler:  SELECT * FROM scheduler_job                │
│              WHERE heartbeat_date > now() - '30s'       │
│  Triggerer:  SELECT * FROM trigger_job                  │
│              WHERE heartbeat_date > now() - '30s'       │
│  Workers:    kubectl get pods -l role=worker            │
│              expect ≥ min_workers in Running+Ready      │
│  RDS:        TCP connect to RDS endpoint :5432          │
│              expect SYN-ACK within 3s                   │
│  Redis:      PING to ElastiCache endpoint               │
│              expect PONG within 2s                      │
└─────────────────────────────────────────────────────────┘
         │
         ▼ Roll-up logic:
  All HEALTHY → deployment HEALTHY
  Any DEGRADED → deployment DEGRADED
  Webserver OR Scheduler UNHEALTHY → deployment UNHEALTHY
```

### 10.1 Automated remediation sequence

When the scheduler check fails, the Health Monitor follows an escalating remediation policy before alerting the customer:

```
Scheduler DEGRADED (1 consecutive failure):
  → kubectl rollout restart statefulset/airflow-scheduler
  → wait 60s, re-check

Scheduler DEGRADED (2 consecutive failures):
  → kubectl delete pod -l component=scheduler
  → wait 90s, re-check

Scheduler UNHEALTHY (3+ consecutive failures):
  → emit UNHEALTHY to Control Plane
  → notify customer via Notification Service
  → page Flowdeck SRE (internal)

Worker pods < min_replicas (2 consecutive):
  → kubectl scale deployment/airflow-worker --replicas={desired_min}

RDS unreachable:
  → do NOT auto-remediate (risk of DB restart is too high)
  → alert customer and Flowdeck SRE immediately
```

### 10.2 Tier-aware thresholds

| Tier | DEGRADED after | UNHEALTHY after |
|------|---------------|-----------------|
| Starter | 2 consecutive failed checks | 4 consecutive |
| Pro | 2 consecutive | 3 consecutive |
| Enterprise | 1 consecutive | 2 consecutive |

Enterprise deployments have tighter thresholds because they have stricter SLOs and dedicated infrastructure — a single failed check is more likely to indicate a real issue than a transient poll failure on shared infrastructure.

## 11. Networking

### 11.1 Inbound — user to webserver

```
User browser / API client
      │ HTTPS
      ▼
AWS ALB (one per EKS cluster, host-based routing)
      │ Host: {org-slug}-{dep-slug}.airflow.flowdeck.io
      │ TLS: ACM wildcard cert *.airflow.flowdeck.io
      │ Auth: Lambda authorizer validates Flowdeck JWT
      ▼
Kubernetes ingress controller (in flowdeck-system namespace)
      │ Routes by Host header to Service in dep-{deployment_id}
      ▼
Service: airflow-webserver (ClusterIP :8080)
      ▼
airflow-webserver pod
```

**JWT validation at ALB:** The Lambda authorizer calls the Flowdeck Identity Service to validate the bearer JWT. A valid JWT with a role granting access to this deployment is required before the request reaches the webserver. Airflow's own authentication is a secondary layer. The webserver receives a request header `X-Flowdeck-User-Id` injected by the authorizer, which Airflow maps to its own user model via a custom auth backend.

### 11.2 Intra-namespace communication

Airflow components communicate within the namespace using Kubernetes Services:

```
Webserver    → RDS (port 5432) via Security Group rule
Scheduler    → RDS (port 5432)
Scheduler    → Redis (port 6379, TLS) — CeleryExecutor
Worker       → Redis (port 6379, TLS) — CeleryExecutor
Worker       → RDS (port 5432) — XCom reads/writes
Triggerer    → RDS (port 5432) — trigger state
```

There is no direct pod-to-pod communication within the namespace. All coordination goes through RDS or Redis. This means the failure of one component does not directly cause another to fail — it causes them to observe stale or missing state in the shared data stores.

### 11.3 Egress — task code to external systems

Task code running in a worker pod can make outbound connections to external systems (databases, APIs, cloud services). Egress is governed by the NetworkPolicy and the deployment's IAM role:

- **Default egress:** allowed to RDS, Redis, S3 (via VPC endpoint), and Secrets Manager (via VPC endpoint). All other egress is denied by default.
- **Additional egress:** if task code needs to reach an external database or API, the customer adds the target to their Airflow Connections. The connection credentials are materialised as environment variables by the Secret Distributor. The network path (security group, VPC peering, Transit Gateway) must be approved and configured separately.
- **IAM egress:** task pods inherit the deployment's IRSA role. AWS API calls made by task code are scoped to the services the deployment's IAM role permits. Per-task IAM scoping is only available with KubernetesExecutor.

## 12. Secrets and connections

Airflow connections (database credentials, API keys) and sensitive Variables are never stored in Airflow's metadata database as plaintext. The actual secrets live in AWS Secrets Manager. The Secret Distributor materialises them as Kubernetes Secrets in the deployment namespace.

### 12.1 Connection materialisation

```
Customer creates a connection in the Flowdeck UI or API:
  POST /v1/deployments/{id}/connections
  { "name": "my_postgres",
    "type": "postgres",
    "secret_ref": "arn:aws:secretsmanager:...:secret:/flowdeck/{dep_id}/conn/my_postgres" }

Control Plane stores in Aurora:
  { connection_id, name, type, secret_ref }
  — actual password NOT stored in Aurora

Customer writes connection credentials to Secrets Manager at the ref path.

Secret Distributor reconciliation loop (every 60s):
  For each deployment's connection refs:
  1. aws secretsmanager get-secret-value --secret-id {ref}
     (using the deployment's IRSA role)
  2. Compare VersionId against last-seen version in Agent BoltDB
  3. If changed or new:
     kubectl apply secret airflow-connections
       --from-literal=AIRFLOW_CONN_MY_POSTGRES=postgres://user:pass@host/db
       --namespace dep-{dep_id}
  4. If secret was rotated AND pods are running:
     kubectl rollout restart deployment/airflow-worker
     (scheduler NOT restarted unless it directly uses the connection)
```

### 12.2 Fernet key

Airflow's Fernet key (`AIRFLOW__CORE__FERNET_KEY`) is used to encrypt connection passwords stored in the metadata DB. In Flowdeck's model, connections are stored as references (not plaintext in the DB), so the Fernet key's primary role is encrypting `Extra` fields in connections and any values stored via the deprecated connections-in-DB path.

The Fernet key is unique per deployment, generated at provisioning time, stored in Secrets Manager at `/flowdeck/{dep_id}/fernet_key`, and materialised as `airflow-fernet-key` Kubernetes Secret by the Secret Distributor. It never appears in Helm values or ConfigMaps.

**Rotation:** Fernet key rotation requires re-encrypting all encrypted values in the metadata DB with the new key, which Airflow supports via `airflow rotate-fernet-key`. This is a planned maintenance operation — it is not rotated automatically.

## 13. Task logging

### 13.1 Log flow

```
Task running in worker pod:
  │ writes log lines to local file:
  │   /opt/airflow/logs/{dag_id}/{run_id}/{task_id}/{attempt}.log
  │
  ▼ RemoteFileTaskHandler (Airflow logging plugin)
  │ uploads to S3 in real-time (line-buffered):
  │   s3://{dep_bucket}/logs/{dag_id}/{run_id}/{task_id}/{attempt}.log
  │ using deployment's IRSA role

User views logs in Airflow UI:
  │ Webserver calls S3 GetObject via pre-signed URL
  │ Streams to browser (no server-side buffering)
  │ Falls back to reading from worker pod if S3 upload not yet complete
```

### 13.2 Log retention

Task logs in S3 are retained for 90 days by default (an S3 lifecycle rule on the deployment's bucket). After 90 days, logs are deleted. The retention period is configurable per deployment in its desired state:

```yaml
log_retention_days: 90    # default
                   180    # Pro/Enterprise configurable range: 30–365
```

Logs are not archived to cheaper storage tiers by default. Customers who need longer retention export logs to their own S3 bucket using Airflow's log export configuration.

### 13.3 Structured logging

Airflow 2.6+ supports structured JSON logging. When enabled, each log line is a JSON object with fields: `timestamp`, `level`, `dag_id`, `run_id`, `task_id`, `try_number`, `message`. Flowdeck enables structured logging by default to allow the Metrics Collector to parse error patterns from task logs and surface them in the customer dashboard.

## 14. XCom

XCom (cross-communication) is Airflow's mechanism for passing small data between tasks. XCom values are stored in the metadata DB (`xcom` table) and retrieved by downstream tasks.

### 14.1 Size constraints

XCom values are stored as bytes in the RDS metadata DB. Large XCom values cause DB bloat and slow scheduling loops (the scheduler reads XCom during dependency evaluation). Flowdeck enforces a per-value size limit:

| Tier | Max XCom value size |
|------|-------------------|
| Starter | 1 MB |
| Pro | 4 MB |
| Enterprise | 16 MB (configurable) |

Values exceeding the limit raise a runtime error. Tasks that need to pass large data should use S3 (write in task A, pass the S3 path as XCom, read in task B).

### 14.2 Custom XCom backend

Airflow supports pluggable XCom backends. Flowdeck provides an optional S3-backed XCom backend that stores values in the deployment's S3 bucket and keeps only a reference in the metadata DB. This removes the size constraint entirely and keeps the metadata DB lean. The S3 XCom backend is available for Pro and Enterprise tiers and is enabled via the desired state:

```yaml
xcom_backend: S3     # default: database
xcom_s3_prefix: xcom/ # stored at s3://{dep_bucket}/xcom/{dag_id}/{run_id}/{task_id}/
```

## 15. Resource profiles

The desired state includes a `resource_profile` that the Deployment Controller uses to render Helm values for every component.

```
resource_profile:
  scheduler:
    cpu_request:    "500m"
    cpu_limit:      "2000m"
    memory_request: "1Gi"
    memory_limit:   "3Gi"
  webserver:
    cpu_request:    "250m"
    cpu_limit:      "1000m"
    memory_request: "512Mi"
    memory_limit:   "2Gi"
  triggerer:
    cpu_request:    "100m"
    cpu_limit:      "500m"
    memory_request: "512Mi"
    memory_limit:   "2Gi"
  worker:
    cpu_request:    "500m"
    cpu_limit:      "2000m"
    memory_request: "1Gi"
    memory_limit:   "4Gi"
  worker_concurrency: 16       # tasks per worker pod
  min_workers: 1
  max_workers: 10
```

Predefined profiles (Small, Medium, Large, XLarge) are offered for each tier. Custom profiles are available for Enterprise. The `ResourceQuota` on the namespace is derived from the profile: `quota.requests.cpu = (max_workers + 3) × cpu_request` (workers + scheduler + webserver + triggerer).

## 16. Airflow version management

### 16.1 Supported versions

Flowdeck supports the two most recent Airflow major versions (currently 2.x and 3.x). A version is added to the platform after it passes the internal conformance test suite. The N-2 major version is deprecated with a minimum 12-month notice.

```
Version lifecycle:
  preview:  new major version available for opt-in testing
            (limited to Enterprise customers initially)
  canary:   available for all tiers, ~5% of deployments
  stable:   default for new deployments
  deprecated: still functional, no new features, EOL date set
  EOL:      no longer supported; deployments must upgrade
```

### 16.2 Version upgrade for a deployment

Airflow version upgrades are customer-initiated via the desired state API:

```
PATCH /v1/deployments/{id}
{ "airflow_version": "2.10.0" }

Deployment Service:
  Validates version is in the supported set
  Writes new desired_state_version
  Emits UPDATE command

Deployment Controller:
  Phase 2 only (no infrastructure change)
  helm upgrade: set airflow.image.tag=2.10.0
  Kubernetes rolling update: new pods use new image
  Each new pod: airflow db migrate (on scheduler pod startup)
  Phase 3: validation (webserver health + scheduler heartbeat)
```

**DB migration during upgrade:** The scheduler pod runs `airflow db migrate` on startup. If the migration fails (schema incompatibility, locked tables), the pod fails its readiness probe and the rolling update halts — the old version keeps running. The customer sees a `DB_MIGRATION_FAILED` event with the migration log output.

### 16.3 Forced upgrades

For critical security fixes, Flowdeck may force an Airflow version upgrade with a minimum 30-day notice (except genuine zero-day emergencies). Forced upgrades are applied using the same rolling update mechanism — the customer's workload is not interrupted unless the upgrade itself introduces incompatibilities.

## 17. Operational failure modes

### 17.1 DAG parse error

A DAG file with a Python syntax error or import error causes the scheduler's `DagFileProcessor` to fail for that file. The deployment stays `HEALTHY` — parse errors do not stop the scheduler. The error appears in `import_errors` in the metadata DB and is surfaced in the Airflow UI and the Flowdeck deployment dashboard.

Parsing errors in OCI bundles are caught before deployment (rolling update halts). Parsing errors in git-sync and S3-sync paths are discovered after the file is live — this is the main operational risk of those delivery paths.

### 17.2 Scheduler restart

The scheduler can be safely restarted at any time. On restart it reads all DAG definitions from the metadata DB (parsed representations are stored there), checks in-progress DAG runs, and resumes scheduling. Tasks that were `RUNNING` at the time of restart may become `ZOMBIE` tasks (see Section 7.1) if the worker did not report completion before the scheduler restart.

### 17.3 Worker OOM kill

When a worker pod is OOM-killed by the Kubernetes OOM killer, all tasks running in that pod are marked `ZOMBIE` by the next scheduler heartbeat cycle, then transitioned to `FAILED` and retried if retries are configured. The pod is restarted by Kubernetes (restart policy: Always).

Frequent OOM kills indicate that the deployment's `worker.memory_limit` is too low for the task workload. The customer should increase the resource profile or switch to KubernetesExecutor for tasks with heterogeneous memory requirements.

### 17.4 Metadata DB connection exhaustion

If worker count grows faster than PgBouncer can multiplex connections, new worker pods fail to connect to RDS and tasks fail immediately with a connection error. The scheduler observes the failures and may backfill retries faster than workers can connect, creating a spiral.

Mitigation: `max_workers` in the desired state caps the worker count. PgBouncer's `max_client_conn` is set to `max_workers × worker_concurrency × 1.2` — it is recalculated and the PgBouncer ConfigMap updated whenever `max_workers` changes in the desired state.

### 17.5 Redis broker loss

If the ElastiCache Redis instance becomes unavailable, Celery workers cannot pick up new tasks (no broker). Tasks in `QUEUED` state remain there — they are not lost from the metadata DB. On Redis recovery:

- Workers reconnect to Redis automatically (Celery's built-in retry).
- Tasks in `QUEUED` state are re-queued by the scheduler's next heartbeat cycle.
- Tasks that were `RUNNING` when Redis went down continue to completion (the worker already has the task in memory, no ongoing Redis dependency mid-execution).

Redis loss is reported by the Health Monitor as `DEGRADED` (Redis ping fails) and escalates to `UNHEALTHY` after tier-configured consecutive failures. The Health Monitor does not attempt automated remediation for Redis (ElastiCache failover is AWS-managed for Multi-AZ instances).

## Closing note

The Runtime Plane is intentionally narrow. It does not know it is managed, does not hold its own infrastructure credentials, and does not coordinate with other deployments. Its job is to run Airflow reliably within the boundaries the Data Plane has established — namespace, quotas, network policies, secrets, and storage. The Data Plane watches it and repairs it. The Control Plane defines what it should look like. The Runtime Plane simply runs.
