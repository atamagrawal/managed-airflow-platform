## 1. What "shared data plane" means

In Flowdeck's three-plane model, the **Data Plane** is the layer that converts desired state (owned by the Control Plane) into real cloud resources and running Kubernetes workloads. In public cloud mode, Flowdeck operates the Data Plane inside its own runtime VPC.

**"Shared" is a Kubernetes compute concept, not a data or network concept.** In the Starter and Pro isolation tiers, multiple customer deployments run on the same EKS cluster. What is shared is compute capacity and the node pool. What is *never* shared — in any tier — is the metadata database, the Celery broker, DAG storage, secrets, or network traffic between deployments.

A single shared Data Plane cannot scale indefinitely — a single EKS cluster and a single Agent instance become bottlenecks at hundreds of customers. Flowdeck therefore operates a **pool** of shared Data Plane instances per region. New deployments are assigned to the least-loaded member of the pool, and the pool itself autoscales as demand grows. This is described in detail in Section 4.5.

```
SHARED (Starter/Pro tiers)            DEDICATED (every tier)
───────────────────────────────────   ──────────────────────────────────────
EKS cluster control plane             Kubernetes namespace
Karpenter node provisioner            RDS PostgreSQL (per deployment)
ALB (one per cluster)                 ElastiCache Redis (per deployment)
Data Plane Agent (one per cluster)    S3 / EFS bucket (per deployment)
Kubernetes system node pool           IAM role + IRSA binding (per deployment)
                                      Secrets Manager paths (per deployment)
                                      KEDA ScaledObject (per deployment)
                                      NetworkPolicy (per deployment namespace)
                                      ResourceQuota (per deployment namespace)
```

The architecture is deliberately designed so that shared compute provides cost efficiency while dedicated data services maintain strong isolation between customers.

## 2. Isolation tier model

The `isolation_tier` field in a deployment's desired state drives every provisioning and scheduling decision. There are three tiers, and upgrading between them is an in-place operation through the same reconciliation loop — no migration or data movement required.

```
┌────────────┬──────────────────┬───────────────────────────────────┬──────────────────────────────────┬────────────────────────────┐
│ Tier       │ Data Plane       │ Shared                            │ Dedicated                        │ Noisy-neighbour risk       │
├────────────┼──────────────────┼───────────────────────────────────┼──────────────────────────────────┼────────────────────────────┤
│ STARTER    │ Shared           │ DP Agent, EKS cluster,            │ Namespace, RDS, Redis,           │ Yes — node                 │
│            │                  │ node pool, Karpenter,             │ S3, IAM role                     │ CPU/memory level           │
│            │                  │ KEDA operator, ALB                │                                  │                            │
├────────────┼──────────────────┼───────────────────────────────────┼──────────────────────────────────┼────────────────────────────┤
│ PRO        │ Shared           │ DP Agent, EKS cluster,            │ Node pool (tainted per org),     │ Eliminated at              │
│            │                  │ Karpenter, KEDA operator, ALB     │ Namespace, RDS, Redis, S3        │ node level                 │
├────────────┼──────────────────┼───────────────────────────────────┼──────────────────────────────────┼────────────────────────────┤
│ ENTERPRISE │ Dedicated        │ Flowdeck runtime VPC              │ DP Agent, EKS cluster,           │ Zero — no other            │
│            │ (Flowdeck-ops)   │ Control Plane                     │ Karpenter, KEDA operator,        │ customer in this           │
│            │                  │                                   │ ALB, node pools,                 │ data plane                 │
│            │                  │                                   │ RDS, Redis, S3                   │                            │
└────────────┴──────────────────┴───────────────────────────────────┴──────────────────────────────────┴────────────────────────────┘
```

The `isolation_tier` is validated at admission time by the Deployment Service before any resource is created. Attempting to set a tier the organisation's plan does not include returns a deterministic error with a remediation hint.

## 3. Enterprise — dedicated data plane

### 3.1 What Enterprise isolation means

In Starter and Pro tiers, multiple customers share Flowdeck's Data Plane infrastructure. In Enterprise tier, the customer gets their own Data Plane entirely — a separate Agent instance, a separate EKS cluster, separate Karpenter and KEDA operators, and a separate ALB — all operated by Flowdeck in its public cloud runtime VPC. No other customer's workloads, Agent commands, or scaling events ever interact with the Enterprise customer's infrastructure.

**Enterprise = dedicated data plane.** A dedicated EKS cluster without a dedicated Agent is only half-isolated — the Agent is part of the control surface, and a noisy neighbour at the Agent level (command queue saturation, reconciliation delays) is just as disruptive as one at the node level.

```
FLOWDECK RUNTIME VPC (flowdeck-prod-runtime)
│
├── SHARED DATA PLANE (serves Starter and Pro customers)
│     Data Plane Agent: dp-shared-{region}-01
│     EKS cluster: flowdeck-runtime-{region}
│       ├── System node pool      (Agent, Karpenter, KEDA, ingress)
│       ├── Starter shared pool   (all Starter deployment namespaces)
│       └── Pro org pools         (one tainted pool per Pro org)
│
└── ENTERPRISE DATA PLANE (one per Enterprise customer)
      Data Plane Agent: dp-ent-{org_slug}-{region}  ← customer-exclusive
      EKS cluster: flowdeck-ent-{org_slug}-{region} ← customer-exclusive
        ├── System node pool      (this org's Agent, Karpenter, KEDA, ingress)
        └── Customer node pool(s) (all this org's deployment namespaces)
      ALB: dedicated to this org's webserver hostnames only
```

### 3.2 Registration in the Data Plane Registry

The Enterprise Data Plane is registered with `type = MANAGED_DEDICATED` and an `org_id` binding. The Control Plane's routing function checks for this binding before falling back to shared pool routing:

```
Deployment Service — routing logic:
  1. Look up MANAGED_DEDICATED row with org_id = {requesting_org}, health = HEALTHY
     → Found: route this command exclusively to that dp_id
     → Not found: fall through
  2. Look up MANAGED row in same region with capacity available
     → Found: route to shared data plane (Starter / Pro)
```

Because the NATS subject carries the `dp_id` explicitly (`deployments.commands.{dp_id}.{deployment_id}`), an Enterprise customer's commands are physically on a different subject prefix from shared customers' commands. Cross-contamination is impossible at the message bus level.

### 3.3 What is dedicated vs what remains shared

| Component | Enterprise (dedicated) | Starter / Pro (shared) |
|-----------|------------------------|------------------------|
| Data Plane Agent | Customer-exclusive instance | One Agent per region, shared |
| EKS cluster | Customer-exclusive | Shared cluster |
| Karpenter | Customer-exclusive (manages only this org's cluster) | Shared across all orgs |
| KEDA operator | Customer-exclusive | Shared across all orgs |
| Ingress controller + ALB | Customer-exclusive | One ALB shared across all deployments |
| Node pool(s) | Customer-exclusive | Shared (Starter) or per-org tainted (Pro) |
| Kubernetes system namespace | Customer-exclusive | Shared `flowdeck-system` |
| RDS per deployment | Dedicated (same as all tiers) | Dedicated (same as all tiers) |
| ElastiCache per deployment | Dedicated (same as all tiers) | Dedicated (same as all tiers) |
| S3 / IAM per deployment | Dedicated (same as all tiers) | Dedicated (same as all tiers) |
| Flowdeck runtime VPC | Shared (all modes use same VPC) | Shared |
| Control Plane | Shared (same as all cloud modes) | Shared |

The Flowdeck runtime VPC itself remains shared — Enterprise customers are in the same VPC as the shared data plane, isolated by subnet allocation and security groups rather than by separate VPC. Customers who require their data plane in their own AWS account should use Hybrid mode instead.

### 3.4 Autoscaling in Enterprise

Because Enterprise Karpenter and KEDA manage only one customer's workloads, autoscaling is completely unaffected by any other customer's demand:

- **Worker autoscaling (KEDA):** unchanged — per-deployment `ScaledObject` watching the deployment's own ElastiCache Redis.
- **Node autoscaling (Karpenter):** the NodePool CPU/memory ceiling is sized exclusively for this org's forecast. Consolidation events only affect this customer's nodes.
- **Karpenter disruption budget:** can be tuned more aggressively (e.g., `nodes: "25%"`) since consolidation events have no cross-customer impact.
- **Custom NodePool configuration:** instance type preferences, spot/on-demand ratio, and AZ affinity are configurable per Enterprise contract — this is not available in Starter or Pro.

### 3.5 Independent upgrade windows

In Starter and Pro tiers, EKS upgrades and platform component upgrades run on Flowdeck's shared maintenance calendar. In Enterprise tier, each customer's Data Plane has its own upgrade schedule entry in the platform upgrade manager. Upgrades are coordinated with the customer's designated contact and restricted to their agreed maintenance window (for example, "Sundays 02:00–06:00 UTC only").

The Control Plane is unaffected by the Enterprise cluster's upgrade. Commands queue in NATS during the brief Agent downtime (~5–10 minutes) and replay automatically on Agent reconnect once connectivity is restored.

### 3.6 Enterprise vs Hybrid — choosing the right model

| Aspect | Enterprise (dedicated data plane) | Hybrid Agent |
|--------|-----------------------------------|--------------|
| Who owns the VPC | Flowdeck | Customer |
| Who owns EC2 nodes | Flowdeck | Customer |
| Who manages node lifecycle | Flowdeck (Karpenter) | Customer |
| Customer infra team required | No | Yes |
| Data residency | Flowdeck's AWS account | Customer's AWS account |
| Primary driver | Operational isolation without customer overhead | Data must stay in customer's own account |
| Setup complexity | Low — Flowdeck provisions on contract | High — customer installs and configures Agent |

Choose Enterprise when the requirement is **full platform isolation without owning infrastructure**. Choose Hybrid when the requirement is **data residency inside the customer's own cloud account**.

## 4. Shared cluster architecture

### 4.1 EKS cluster layout

The runtime EKS cluster hosting Starter and Pro deployments is structured around function-separated node groups rather than a flat pool. This keeps platform infrastructure independent of customer workloads, which is essential for safe upgrades and capacity isolation.

```
┌──────────────────────── EKS: flowdeck-runtime-{region} ───────────────────────────┐
│                                                                                   │
│  ┌─────────────────────────────────────────────────────────────────────────────┐  │
│  │ SYSTEM NODE POOL (on-demand, m7i.large)                                     │  │
│  │ Taint: flowdeck.io/pool=system:NoSchedule                                   │  │
│  │ Hosts: flowdeck-system namespace (DP Agent, Karpenter, KEDA operator,       │  │
│  │        ingress controller, metrics collector, secret distributor)           │  │
│  └─────────────────────────────────────────────────────────────────────────────┘  │
│                                                                                   │
│  ┌─────────────────────────────────────────────────────────────────────────────┐  │
│  │ STARTER SHARED NODE POOL (Karpenter-managed, mixed on-demand + spot)        │  │
│  │ Taint: none — any Starter deployment pod is schedulable here               │  │
│  │ Hosts: all Starter deployment namespaces                                    │  │
│  │ Instance types: m7i.large, m7i.xlarge, m7i.2xlarge, c7i.xlarge            │  │
│  └─────────────────────────────────────────────────────────────────────────────┘  │
│                                                                                   │
│  ┌──── PRO ORG NODE POOL (per org, Karpenter-managed) ────────────────────────┐  │
│  │ Taint: flowdeck.io/org={org_id}:NoSchedule                                 │  │
│  │ Toleration: only pods from this org's deployment namespaces carry it        │  │
│  │ Instance types: same catalogue, separate NodeClaim budget per org           │  │
│  └─────────────────────────────────────────────────────────────────────────────┘  │
│  (one PRO node pool per Pro organisation)                                         │
│                                                                                   │
└───────────────────────────────────────────────────────────────────────────────────┘
```

### 4.2 Namespace layout per cluster

```
flowdeck-system                 ← Data Plane Agent, KEDA operator, Karpenter,
                                  ingress controller, metrics collector,
                                  secret distributor

dep-{deployment_id_A}           ← Starter customer A, deployment 1
dep-{deployment_id_B}           ← Starter customer A, deployment 2
dep-{deployment_id_C}           ← Starter customer B, deployment 1
dep-{deployment_id_D}           ← Pro org X, deployment 1
dep-{deployment_id_E}           ← Pro org X, deployment 2
...
```

Namespaces are the primary isolation boundary at the Kubernetes layer. They carry all quota, network policy, and service account configuration. The name format is deterministic from the deployment ID, which is how the Data Plane Agent maps Control Plane commands to cluster resources.

### 4.3 Single ALB with host-based routing

The runtime VPC hosts one ALB per cluster. All customer Airflow webservers are reached through this single ALB using host-based routing rules. Each deployment's hostname (`{org}-{dep}.airflow.flowdeck.io`) is registered in Route 53 by the Data Plane Controller when the deployment reaches `RUNNING`, and a corresponding ALB listener rule routes that hostname to the webserver Service in the correct namespace.

The Flowdeck Identity Service JWT validation is enforced at the ALB via a Lambda authoriser attached to the HTTPS listener. Traffic that fails authentication never reaches the Airflow webserver pod. This means each webserver pod handles only authenticated, already-authorised requests — Airflow's own auth layer is a secondary check, not the primary gate.

### 4.4 Regional topology and the Data Plane pool

Each AWS region running in public cloud mode hosts one runtime VPC containing a **pool of shared Data Plane instances**. Each pool member is an independent EKS cluster with its own Agent, Karpenter, KEDA operator, and ALB. New customer deployments are assigned to the least-loaded pool member. The pool grows automatically as demand increases.

```
REGION: us-east-1
│
└── RUNTIME VPC (flowdeck-prod-runtime)
      │
      ├── SHARED DATA PLANE POOL
      │     ├── dp-shared-use1-01   (Agent + EKS cluster, ~N deployments)
      │     ├── dp-shared-use1-02   (Agent + EKS cluster, ~N deployments)
      │     ├── dp-shared-use1-03   (Agent + EKS cluster, ~N deployments)
      │     └── dp-shared-use1-04   (NEW — provisioned by Pool Controller on scale-out)
      │
      └── ENTERPRISE DATA PLANES (one per Enterprise org)
            ├── dp-ent-acme-use1    (Agent + EKS cluster, org: Acme)
            └── dp-ent-globex-use1  (Agent + EKS cluster, org: Globex)
```

The Control Plane is not co-located in the runtime VPC. The only path between them is the outbound gRPC stream from each Data Plane Agent to `grpc.flowdeck.io`.

### 4.5 Data Plane pool autoscaling

#### 4.5.1 Capacity score

Every Data Plane Agent reports a `capacity_score` in its 15-second heartbeat. This is a composite metric computed by the Agent and stored in the Data Plane Registry:

```
capacity_score (0.0 – 1.0) =
    (active_deployments / dp_max_deployments)   × 0.40
  + (cluster_cpu_util / node_pool_cpu_ceiling)  × 0.35
  + (agent_queue_depth / agent_queue_max)        × 0.25

Where:
  dp_max_deployments   — configured cap per pool member (e.g. 300)
  cluster_cpu_util     — sum of worker pod CPU requests across all namespaces
  node_pool_cpu_ceiling — Karpenter NodePool .limits.cpu
  agent_queue_depth    — pending commands in the Agent's local BoltDB queue
  agent_queue_max      — configurable threshold (e.g. 50 concurrent operations)
```

A `capacity_score` of `0.0` means idle; `1.0` means at ceiling on all dimensions. The registry exposes this score to the Deployment Service for routing decisions.

#### 4.5.2 Deployment assignment (routing into the pool)

When the Deployment Service creates a new deployment and needs to assign it to a Data Plane, it queries the registry for the pool member with the lowest `capacity_score` in the correct region:

```
Routing function — new deployment assignment:
  1. Filter: type = MANAGED, region = {org_home_region}, health = HEALTHY
  2. Filter: capacity_score < CAPACITY_THRESHOLD  (default: 0.75)
  3. Sort by capacity_score ascending (least loaded first)
  4. Assign to dp_id of the top result
  5. If no member passes the filter → return 503 + trigger scale-out (see 4.5.3)
```

Once assigned, the deployment is permanently bound to its `dp_id`. The binding is stored in the `deployments` table alongside the desired state. All future commands for that deployment — updates, scaling, deletion — go to the same Agent. Deployments are not rebalanced across pool members after initial assignment.

#### 4.5.3 Pool scale-out — adding a new member

Scale-out is triggered by the **Pool Controller**, a service running in the Control Plane. It evaluates two conditions every 60 seconds:

```
Scale-out condition A (capacity pressure):
  IF (count of pool members with capacity_score > 0.70) / (total healthy members)
     > SCALE_OUT_RATIO_THRESHOLD  (default: 0.60)
  THEN trigger scale-out

Scale-out condition B (routing failure):
  IF any deployment creation in the last 5 min was rejected due to
     no pool member below CAPACITY_THRESHOLD
  THEN trigger scale-out immediately (bypass the 60s interval)
```

When triggered, the Pool Controller provisions a new Data Plane member:

```
Pool scale-out sequence:
  1. Pool Controller calls Terraform to provision a new EKS cluster
     (same modules as any shared cluster, no org_id binding)
     Duration: ~15–20 min (EKS control plane + node group bootstrap)

  2. Terraform outputs are used to generate:
     - Agent bootstrap token (single-use, 1-hour TTL)
     - mTLS certificate for the new Agent (issued by Vault PKI)
     - kubeconfig stored in Secrets Manager for the Pool Controller

  3. Pool Controller applies the Agent Helm chart to the new cluster
     (Agent, Karpenter, KEDA operator, ingress controller)
     Duration: ~3–5 min

  4. New Agent starts, connects to Control Plane, registers as:
       type = MANAGED
       region = {region}
       capacity_score = 0.0   ← immediately available for assignment
       health = HEALTHY

  5. DP Registry marks new member HEALTHY
     Deployment Service routing immediately includes it in the pool
     New deployments start being assigned to the new member
```

During the provisioning window (~20 min), in-flight deployment creation requests that hit capacity are queued with a retry, not rejected. The Deployment Service holds the request in a `PENDING_ASSIGNMENT` state, polls the registry every 30 seconds, and assigns as soon as a member with available capacity appears.

#### 4.5.4 Pool scale-in — removing an underutilised member

Scale-in is more conservative than scale-out. Because existing deployments cannot be transparently moved between pool members, a member can only be removed once it has no active deployments.

```
Scale-in eligibility:
  condition: capacity_score < 0.10 for > 48 hours continuously
             (low deployment count, low CPU, low queue)

Scale-in sequence:
  1. Pool Controller marks the member as DRAINING in the registry
     - DRAINING members receive no new deployment assignments
     - Existing deployments continue operating normally

  2. Pool Controller monitors active_deployments on the DRAINING member
     - Deployments leave naturally as customers delete them
     - No forced migration of live deployments

  3. If active_deployments = 0 for > 1 hour:
     Pool Controller decommissions the cluster:
       - Helm uninstall (removes Agent and system pods)
       - Terraform destroy (removes EKS cluster, node groups)
       - Registry row marked TERMINATED

  4. If active_deployments > 0 after 30 days in DRAINING state:
     SRE is alerted to evaluate whether active migration is warranted
     (migration involves metadata DB copy per deployment — same as
      Pro → Enterprise migration mechanics in Section 14)
```

Scale-in is intentionally slow. The 48-hour eligibility window prevents thrashing where a pool member goes idle during a weekend, is removed, and then needs to be re-provisioned on Monday.

#### 4.5.5 Pool size limits

The pool is bounded to prevent runaway provisioning:

```
MIN_POOL_SIZE = 2    (always at least 2 members per region for HA)
MAX_POOL_SIZE = {configured per region, reviewed quarterly}

If MAX_POOL_SIZE is reached and all members are at capacity:
  - New deployment creation returns 503 with reason POOL_AT_CAPACITY
  - A POOL_CAPACITY_CRITICAL alert pages Flowdeck SRE
  - SRE reviews and raises MAX_POOL_SIZE or investigates abnormal load
```

The minimum of 2 ensures that a single member failure or upgrade does not leave the region with no shared pool capacity.

#### 4.5.6 Pool health and failover

If a pool member's Agent goes `OFFLINE` (no heartbeat for > 5 minutes), the registry marks it `OFFLINE`. Deployments on that member:

- Continue running — the Runtime Plane (Airflow, workers) is independent of the Agent and keeps operating.
- Cannot receive new desired-state commands until the Agent reconnects.
- On reconnect, the Agent replays all missed commands in order.

If a pool member EKS cluster is lost entirely (rare — an AWS AZ failure affecting the whole cluster), the Deployment Controller for that pool member is gone. Recovery:

```
EKS cluster loss recovery:
  1. SRE promotes a standby cluster (pre-provisioned spare in the same region)
     OR provisions a new cluster through the normal scale-out flow
  2. New Agent registers with the Control Plane
  3. Control Plane re-associates the lost member's deployment bindings
     to the new dp_id (a manual SRE operation via admin API)
  4. Deployment Controller rebuilds namespaces from desired state
     (namespace is ephemeral; desired state in Aurora is the truth)
  5. Per-deployment RDS and Redis are unaffected — they are outside the cluster
```

The key resilience property: losing a pool member disrupts the Agent (command processing) but not the running Airflow environments. DAGs keep running; only new deploys and config changes are blocked until the Agent recovers.

## 5. Per-deployment dedicated resources

Even in the fully shared Starter tier, each deployment is provisioned with its own set of cloud-managed resources. These are created by the Deployment Controller in Phase 1 of the three-phase provisioning flow.

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│  PER-DEPLOYMENT RESOURCES (created in Phase 1, Terraform-managed)               │
├──────────────────────────────────────┬──────────────────────────────────────────┤
│ Resource                             │ Notes                                    │
├──────────────────────────────────────┼──────────────────────────────────────────┤
│ RDS PostgreSQL 15 cluster            │ Airflow metadata database. db.t4g.micro  │
│                                      │ for Starter, configurable for Pro.       │
│                                      │ Multi-AZ for Pro and above.              │
├──────────────────────────────────────┼──────────────────────────────────────────┤
│ ElastiCache Redis 7 cluster          │ Celery broker (if CeleryExecutor).       │
│                                      │ cache.t4g.micro for Starter.             │
│                                      │ Also used as KEDA trigger source for     │
│                                      │ this deployment's worker scaling.        │
├──────────────────────────────────────┼──────────────────────────────────────────┤
│ S3 bucket (or EFS volume)            │ DAG bundle storage. Server-side KMS.     │
│                                      │ Bucket policy: only this deployment's    │
│                                      │ IRSA role can read/write.                │
├──────────────────────────────────────┼──────────────────────────────────────────┤
│ IAM Role (IRSA)                      │ Trust policy scoped to:                  │
│                                      │   serviceaccount:dep-{id}/airflow-{id}   │
│                                      │ Permission boundary: only               │
│                                      │   /flowdeck/{dep_id}/* in Secrets Mgr   │
├──────────────────────────────────────┼──────────────────────────────────────────┤
│ Security groups                      │ Wires RDS, Redis, and EKS pods together. │
│                                      │ Only the deployment's pod CIDR can reach │
│                                      │ its RDS and Redis endpoints.             │
├──────────────────────────────────────┼──────────────────────────────────────────┤
│ Secrets Manager paths                │ /flowdeck/{dep_id}/conn/* and            │
│                                      │ /flowdeck/{dep_id}/var/* are owned       │
│                                      │ exclusively by this deployment's IAM role│
└──────────────────────────────────────┴──────────────────────────────────────────┘
```

**Why dedicated databases even in Starter?** Shared databases introduce cross-customer blast radius at the data level — a runaway migration or a misconfigured `alembic upgrade` from one deployment could corrupt another's schema. By giving each deployment its own RDS, the blast radius of any data-layer failure is one deployment, not the entire cluster. The cost of small RDS instances is low relative to the isolation guarantee.

## 6. Kubernetes isolation layer

Each deployment namespace is configured with four Kubernetes resources that enforce isolation at the compute layer.

### 6.1 ResourceQuota

Caps total CPU, memory, and pod count consumable by all pods in the namespace. Values are derived from the deployment's resource profile in desired state.

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: airflow-quota
  namespace: dep-{deployment_id}
spec:
  hard:
    requests.cpu: "{cpu_request_total}"      # e.g. "8" for a 2-worker Starter
    requests.memory: "{memory_request_total}" # e.g. "16Gi"
    limits.cpu: "{cpu_limit_total}"
    limits.memory: "{memory_limit_total}"
    count/pods: "{max_pods}"                  # webserver+scheduler+triggerer+workers
```

When KEDA attempts to scale workers beyond the quota, new worker pods enter `Pending` with reason `Exceeded quota`. KEDA observes the pending pod count (not just running pods) and does not exceed the desired replica count — the quota acts as a hard ceiling for compute within the namespace.

### 6.2 LimitRange

Sets default requests and limits for containers that do not specify them. This protects the cluster from DAG-authored tasks that launch pods without resource constraints — for example, a `KubernetesPodOperator` pod that requests unlimited memory.

```yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: airflow-limits
  namespace: dep-{deployment_id}
spec:
  limits:
    - type: Container
      default:
        cpu: "2000m"
        memory: "4Gi"
      defaultRequest:
        cpu: "100m"
        memory: "256Mi"
      max:
        cpu: "8000m"
        memory: "32Gi"
```

### 6.3 NetworkPolicy

All deployment namespaces start with a deny-all-ingress, deny-all-egress base policy. Explicit allow rules are added only for the required traffic paths.

```yaml
# Ingress: only the ALB ingress controller can reach the webserver
# Egress: only these destinations are allowed:
#   - {rds_sg_cidr}:5432       — metadata DB
#   - {redis_sg_cidr}:6379     — Celery broker
#   - 169.254.169.254:80       — IMDS endpoint for IRSA token exchange
#   - s3.{region}.amazonaws.com — DAG bundle bucket (via VPC endpoint)
#   - secretsmanager.{region}.amazonaws.com — secret reads (via VPC endpoint)
```

Critically, there is no egress rule allowing pods in namespace `dep-A` to reach anything in namespace `dep-B`. Cross-namespace pod traffic is impossible even if two deployments are on the same node.

### 6.4 Pod Security Standards

Every deployment namespace is labelled `pod-security.kubernetes.io/enforce: restricted`. This blocks:

- Privileged containers
- Host network or host PID access
- Writable root filesystems
- Containers running as root (unless the image explicitly drops to a non-root user)
- Host path volume mounts

Airflow's official images satisfy these constraints. Custom OCI bundles that violate the policy fail at admission with a clear reason code surfaced to the customer.

## 7. Autoscaling architecture

Autoscaling in the shared data plane operates through two independent control loops: **worker autoscaling** (logical demand, per-deployment) and **node autoscaling** (physical capacity, per node pool). They do not coordinate; they answer different questions.

### 7.1 Worker autoscaling — KEDA (per deployment)

KEDA is deployed as a cluster-level operator in `flowdeck-system`, but it manages per-deployment `ScaledObject` resources. Each deployment has its own `ScaledObject` pointing to its own ElastiCache Redis instance. The scaling decisions for deployment A are completely independent of those for deployment B, even when they share nodes.

```yaml
# Per-deployment ScaledObject — created by Deployment Controller in Phase 2
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: keda-worker-scaler
  namespace: dep-{deployment_id}
spec:
  scaleTargetRef:
    name: airflow-worker
  minReplicaCount: {desired_state.min_workers}   # e.g. 0 for dev, 1 for prod
  maxReplicaCount: {desired_state.max_workers}   # bounded by ResourceQuota
  cooldownPeriod: 120
  triggers:
    - type: redis
      metadata:
        address: "{elasticache_endpoint}:6379"   # this deployment's Redis only
        listName: "celery"
        listLength: "5"                          # target tasks per worker
        enableTLS: "true"
```

**Scale-up logic:**

```
poll interval: 30 seconds
trigger condition: (queue_depth / current_workers) > listLength
new replica count: ceil(queue_depth / listLength), capped at maxReplicaCount
```

When new worker pods are created and no node has capacity, they enter `Pending`. Karpenter observes the pending pods and provisions new nodes (see Section 6.2).

**Scale-down logic:**

```
condition: queue_depth < (listLength × current_workers) for cooldownPeriod
action: patch worker Deployment replicas to new lower count
        → Kubernetes sends SIGTERM to selected worker pods
        → preStop hook: airflow celery worker --drain
           (worker stops accepting tasks, waits for in-flight tasks
            up to tier_config.worker_drain_timeout_seconds)
        → pod exits cleanly after drain completes or timeout
```

The drain timeout is tier-aware:

| Tier | `worker_drain_timeout_seconds` |
|------|-------------------------------|
| Starter | 300 (5 min) |
| Pro | 600 (10 min) |
| Enterprise | 1800 (30 min, configurable) |

Longer drain timeouts for higher tiers reflect the expectation that Pro and Enterprise workloads run longer tasks that should not be interrupted by a routine scale-down event.

**Scale-to-zero:** `minReplicaCount: 0` is allowed and common for development deployments. When the queue is empty and no tasks are running, KEDA scales workers to zero after `cooldownPeriod`. The scheduler and webserver continue running (they are not KEDA-managed). On next task submission, KEDA scales workers back from zero; the first task may wait up to ~60 seconds for a worker pod to start and pull its image — warm image layers reduce this in practice.

### 7.2 Node autoscaling — Karpenter (per node pool)

Karpenter runs in `flowdeck-system` and manages EC2 node lifecycle for each Karpenter `NodePool`. In the shared tier model there is one `NodePool` per pool type (Starter shared, and one per Pro org). Karpenter watches for unschedulable pods and provisions nodes that fit them.

```yaml
# Starter shared NodePool
apiVersion: karpenter.sh/v1
kind: NodePool
metadata:
  name: starter-shared
spec:
  template:
    spec:
      requirements:
        - key: node.kubernetes.io/instance-type
          operator: In
          values: [m7i.large, m7i.xlarge, m7i.2xlarge, c7i.xlarge, r7i.xlarge]
        - key: karpenter.k8s.aws/capacity-type
          operator: In
          values: [spot, on-demand]   # Starter allows spot to reduce cost
        - key: topology.kubernetes.io/zone
          operator: In
          values: [us-east-1a, us-east-1b, us-east-1c]
      nodeClassRef:
        apiVersion: karpenter.k8s.aws/v1
        kind: EC2NodeClass
        name: starter-shared
  limits:
    cpu: {cluster_cpu_ceiling}          # hard ceiling; no new nodes beyond this
    memory: {cluster_memory_ceiling}
  disruption:
    consolidationPolicy: WhenUnderutilized
    consolidateAfter: 5m
    budgets:
      - nodes: "10%"                    # max % of nodes replaced simultaneously
```

```yaml
# Pro per-org NodePool (one per Pro organisation)
apiVersion: karpenter.sh/v1
kind: NodePool
metadata:
  name: pro-org-{org_id}
spec:
  template:
    metadata:
      labels:
        flowdeck.io/org: "{org_id}"
    spec:
      taints:
        - key: flowdeck.io/org
          value: "{org_id}"
          effect: NoSchedule
      requirements:
        - key: node.kubernetes.io/instance-type
          operator: In
          values: [m7i.large, m7i.xlarge, m7i.2xlarge, m7i.4xlarge, r7i.xlarge]
        - key: karpenter.k8s.aws/capacity-type
          operator: In
          values: [on-demand]           # Pro uses on-demand only for predictability
```

**Scale-up flow:**

```
KEDA scales worker Deployment replicas up
→ New worker pods created, enter Pending (no node capacity)
→ Karpenter's NodePool controller observes Pending pods
→ Evaluates which instance type fits all Pending pods at minimum cost
  (bin-packing: select instance type that fits the most pods per dollar)
→ Calls EC2 RunInstances for the selected type
→ Node bootstraps, joins cluster via EKS node group bootstrap
→ Pending pods scheduled, start pulling images, become Running
→ Total elapsed time from KEDA scale-up to task execution: ~60–90 seconds
  (dominated by EC2 instance boot; warm nodes reduce this to ~5 seconds)
```

**Consolidation (scale-down) flow:**

```
Workers drained and pods terminated by KEDA scale-down
→ Node CPU/memory utilisation drops
→ Karpenter observes utilisation < consolidation threshold for consolidateAfter period
→ Selects underutilised nodes for removal
→ Cordons node (no new pods scheduled)
→ Drains node (evicts remaining system pods with PodDisruptionBudget respect)
→ Calls EC2 TerminateInstances
```

The `disruption.budgets` setting (`nodes: "10%"`) prevents Karpenter from simultaneously replacing too many nodes — important in the shared cluster where a consolidation event could affect multiple customers.

### 7.3 Interaction between KEDA and Karpenter

The two loops never communicate directly. Their interaction is through Kubernetes pod scheduling:

```
Demand spike for customer A's deployment
│
├─► KEDA (customer A's ScaledObject): scale workers from 2 → 8
│     New 6 worker pods created → Pending (no node capacity)
│
├─► KEDA (customer B's ScaledObject): no change (B's queue is empty)
│
└─► Karpenter: observes 6 Pending pods in dep-{A}
      Provisions 2 × m7i.xlarge nodes
      Pods scheduled and running
      Customer B's existing pods unaffected — no eviction, no restart
```

A critical design property: **Karpenter provisions for Pending pods only**. It does not speculatively provision. If customer B's KEDA independently scales up at the same time, Karpenter evaluates all Pending pods together and provisions the minimal set of nodes that fits both sets. This bin-packing behaviour is what makes the shared tier cost-efficient.

### 7.4 Autoscaling boundaries — what is and is not per-customer

| Scaling component | Scope | Per-customer? |
|-------------------|-------|---------------|
| KEDA `ScaledObject` | Per deployment namespace | Yes — fully independent |
| KEDA Redis trigger | Each points to the deployment's own ElastiCache | Yes |
| `min_workers` / `max_workers` bounds | Set in each deployment's desired state | Yes |
| Worker `ResourceQuota` ceiling | Set per namespace | Yes |
| Karpenter `NodePool` | Per isolation tier / per org (Pro) | Partially — Starter pool is shared across orgs |
| EC2 node provisioning | Karpenter decides per NodePool | Shared for Starter, per-org for Pro |
| Node CPU/memory ceiling | Set per NodePool | Shared for Starter, per-org for Pro |

**The answer to "is autoscaling per-customer?":** Worker autoscaling (the scheduling of Airflow tasks to workers) is fully per-deployment. Node autoscaling (EC2 provisioning) is per-org in Pro tier and shared-pool for Starter tier. In both cases, the worker count decision for one deployment never affects another — Karpenter is the only shared resource, and it provisions to fit demand rather than competing for a fixed pool.

## 8. Noisy-neighbour mitigations

The shared data plane is not immune to noisy-neighbour effects. The mitigations below are defence-in-depth; each layer independently limits blast radius.

### 8.1 Node-level isolation (Pro tier)

In Pro tier, org-tainted node pools ensure that customer A's CPU-heavy scheduler or runaway workers cannot saturate nodes that customer B's pods land on. This eliminates noisy-neighbour effects at the node level. It is the primary reason Pro tier exists — customers who have predictable production workloads and cannot accept the Starter tier's shared node pool risk should use Pro.

In Starter tier, `ResourceQuota` limits total namespace resource consumption, but pods from multiple customers share nodes. A Starter deployment that hits its quota ceiling may have pods in `Pending` — it will not starve other namespaces, but its own scheduling will be blocked until the quota is freed.

### 8.2 ResourceQuota as a hard ceiling

A customer deployment in Starter tier that requests more than its quota simply cannot get it. It does not borrow from neighbours. The quota enforcement is at the Kubernetes admission level — no pod is created if it would exceed the namespace quota.

### 8.3 LimitRange defaults on unconstrained pods

DAGs running `KubernetesPodOperator` or `@task` operators may launch pods without explicit resource requests. `LimitRange` injects default requests and hard limits on those pods, preventing them from claiming unbounded node resources and evicting other pods.

### 8.4 PriorityClass differentiation

System pods in `flowdeck-system` (DP Agent, KEDA operator, ingress controller) are assigned a higher `PriorityClass` than customer workload pods. If the cluster is under memory pressure and evictions are needed, customer worker pods are evicted before platform system pods. This ensures the Data Plane Agent and ingress controller remain stable even during a heavy load spike on the shared tier.

```yaml
# Platform system pods
priorityClassName: flowdeck-system-critical    # priority: 1000000

# Customer workload pods (webserver, scheduler, workers)
priorityClassName: flowdeck-customer-default   # priority: 0
```

### 8.5 KEDA poll interval and cooldown

KEDA's 30-second poll interval and 120-second cooldown mean that a temporary spike in one deployment's queue does not instantly provision nodes that then compete with other deployments' scheduling. The cooldown prevents thrashing that would create and destroy shared nodes faster than other deployments can stabilise.

### 8.6 Karpenter disruption budgets

The `disruption.budgets: nodes: "10%"` constraint on the Starter shared NodePool prevents a large consolidation event from simultaneously evicting pods across many customer deployments. At most 10% of nodes are replaced at any point in time during consolidation.

## 9. Quota enforcement chain

Resource limits flow from the Control Plane downward, with each level enforcing a subset of constraints. A quota at a higher level cannot be exceeded by a lower level.

```
CONTROL PLANE (Aurora)
│
│  Organisation quota:
│    max_active_deployments: N
│    max_total_cpu_cores: X
│    max_total_memory_gb: Y
│    API request rate: R/s
│
├─► Workspace quota (inherits from org, optionally tighter):
│     max_deployments: n  (n ≤ N)
│     max_cpu_cores: x    (x ≤ X)
│
├─► Deployment admission check (before Phase 1 starts):
│     Redis lock: quota_lock:{org_id}  (TTL: 5s)
│     Verify: sum(existing deployment resource profiles) + new profile ≤ org quota
│     Verify: workspace deployment count < workspace max
│     If pass: write desired state, release lock
│     If fail: return 409 with deterministic error + remediation hint
│
DATA PLANE (Kubernetes)
│
└─► Per-deployment namespace:
      ResourceQuota: enforces CPU/memory/pod limits
        (values derived from the deployment's resource profile in desired state)
      LimitRange: injects defaults on unconstrained pods
      Karpenter NodePool limits: cluster-level CPU/memory ceiling
        (backs up namespace quota — even if a quota were misconfigured,
         the node pool ceiling prevents runaway EC2 provisioning)
```

The Redis distributed lock during admission is what prevents a race where two simultaneous deployment creation requests both pass the quota check and together exceed the limit. The lock is short-lived (5 seconds) and only held during the check-and-create window, so it does not create a meaningful throughput bottleneck at normal creation rates.

## 10. Data Plane Agent statefulness

### 10.1 Design decision: minimally stateful

A fully stateless Agent would be operationally simpler — pods could be freely replaced, pool members could scale without state migration concerns. But a fully stateless Agent creates a safety problem: on every reconnect after a network outage, how does it know which commands have already been executed? Without local state it must either re-run everything from scratch (potentially destructive on live deployments) or ask the Control Plane on every command (adding synchronous network dependency to the hot path).

The Agent is therefore **minimally stateful**. It carries only the state that cannot be safely or efficiently derived from the Control Plane on demand. The Control Plane's Aurora remains the authoritative source of desired state, deployment configuration, and audit records. The Agent's local state is a safety cache — it makes the Agent faster and safer, but the system can recover from its loss.

### 10.2 What the Agent stores locally (BoltDB on EBS)

The Agent uses an embedded BoltDB key-value store on a Kubernetes PersistentVolumeClaim backed by an EBS gp3 volume. The PVC is not deleted on pod termination, so the state survives pod restarts and rescheduling within the same node pool.

```
BoltDB buckets:

  applied_versions
  ────────────────
  Key:   deployment_id (UUID)
  Value: { applied_desired_state_version: uint64,
           last_applied_at: timestamp }

  Purpose: Deduplication. When a command arrives, the Agent compares its
    desired_state_version against this map. If the version is already applied,
    the command is silently acknowledged without re-execution.
  Why local: This check must happen before any outbound call. A synchronous
    Control Plane round-trip on every command receipt would add latency and
    create an availability dependency on the hot path.

  phase_checkpoints
  ─────────────────
  Key:   deployment_id (UUID)
  Value: { phase: INFRA | RUNTIME | VALIDATION,
           status: IN_PROGRESS | COMPLETE | FAILED,
           phase_output: { rds_endpoint, redis_endpoint, ... } }

  Purpose: Crash recovery. If the Agent restarts mid-provisioning, the
    Deployment Controller resumes from the last completed phase rather than
    restarting from Phase 1. This prevents re-running Terraform against a
    live RDS instance or re-issuing a Helm install on a running deployment.
  Why local: Phase outputs (RDS endpoints, Redis endpoints) are needed by
    the next phase before the Control Plane has them — they flow upward
    after provisioning completes, not before.

  credential_material
  ───────────────────
  Key:   agent (singleton)
  Value: { mtls_cert_pem, mtls_key_pem, jwt, jwt_expiry,
           last_nats_acked_sequence }

  Purpose: Reconnect without full re-registration. On pod restart the Agent
    uses its existing mTLS cert to reconnect immediately. The last acked
    NATS sequence lets it resume replay from exactly where it left off.

  buffered_status_events
  ──────────────────────
  Key:   sequence (uint64, monotonic)
  Value: { deployment_id, state, phase, reason_code, timestamp }
  Retention: ring buffer, max 1,000 events

  Purpose: Status updates emitted while the Control Plane stream is down
    are buffered here and flushed on reconnect. Without this, status
    transitions during a network outage are silently lost and the UI
    shows stale state.
```

### 10.3 What the Agent does NOT store

The following are explicitly kept out of the Agent's local state:

| Data | Where it lives | Reason |
|------|---------------|--------|
| Desired state (deployment config) | Control Plane — Aurora `deployments` table | Single source of truth; Agent fetches on command receipt |
| Deployment-to-pool-member assignment | Control Plane — Data Plane Registry | Routing decisions are made by the Deployment Service, not the Agent |
| Customer secrets | Secrets Manager (via Secret Distributor) | Secrets never touch the Agent |
| Audit records | Control Plane — Aurora `audit_log` table | Immutability and retention requirements need centralised storage |
| Billing metrics | Aggregated and pushed to Control Plane hourly | Fine to lose a partial hour; billing is not real-time |

### 10.4 Recovery when Agent state is lost

BoltDB loss (EBS volume failure, volume detachment, accidental deletion) is recoverable without data loss:

```
applied_versions lost:
  Agent fetches the current desired_state_version for each known deployment
  from the Control Plane on reconnect (via a GetDeployments RPC).
  Any command with version ≤ fetched version is treated as already applied.
  Commands with version > fetched version are reconciled in order.
  Risk: none. Slightly more conservative — may skip a command that was
    in-flight at crash time, but the Control Plane will re-send it.

phase_checkpoints lost:
  Deployment Controller re-runs all phases from Phase 1.
  Terraform: idempotent — plan shows no diff on already-provisioned resources.
  Helm:      idempotent — upgrade with same values is a no-op.
  Cost: slower (full Terraform plan + Helm diff), not destructive.
  Estimated extra time: 3–8 min per in-progress deployment.

credential_material lost:
  Agent re-registers using a new bootstrap token requested from
  the Control Plane via the admin API (SRE operation).
  Interruption: ~5–10 min until new registration completes.

buffered_status_events lost:
  Status transitions that occurred during the outage will not appear
  in the audit trail or UI history for that period.
  No operational impact — deployments keep running regardless.
```

### 10.5 Statefulness implications for pool scaling

The Agent's local state is the reason deployments are not rebalanced between pool members after initial assignment. Moving a deployment from `dp-shared-use1-01` to `dp-shared-use1-02` would require:

1. Migrating the `applied_versions` and `phase_checkpoints` BoltDB entries to the new Agent — or accepting that the new Agent starts blind (recoverable, but causes a brief reconciliation window).
2. Updating the deployment's `dp_id` binding in the Control Plane registry.
3. Ensuring no in-flight command is lost between the two Agents.

This is possible but operationally complex. The pool design avoids it entirely: new members start with an empty BoltDB and take only new deployments. Existing deployments stay on their original member until the customer deletes and recreates them. This is why pool scale-in (Section 4.5.4) waits for natural attrition rather than forcing migrations.

**For Enterprise deployments:** the Agent is also minimally stateful with the same BoltDB structure. Because the Enterprise Agent manages only one org's deployments, the BoltDB is smaller and easier to back up. SRE can snapshot the EBS volume as part of the Enterprise customer's DR policy.

### 10.6 Agent in multi-tenant mode — internal concurrency

A single Data Plane Agent instance manages all deployments on its cluster — potentially hundreds of Starter-tier deployments per cluster. This section describes how the Agent scales internally to handle this concurrency.

### 10.7 Internal concurrency model

The Agent is a Go service. Its main components use goroutines:

```
Agent process (single pod, leader-elected)
│
├── gRPC stream goroutine
│     Reads commands from Control Plane stream
│     Writes status updates and heartbeats
│
├── Command dispatcher (goroutine pool, configurable parallelism)
│     Receives dispatched commands from stream goroutine
│     Routes to Deployment Controller goroutine per deployment_id
│     Default parallelism: 20 concurrent deployment operations
│     (create/update/delete commands for different deployments run in parallel;
│      commands for the same deployment_id are serialised by the per-dep goroutine)
│
├── Deployment Controller goroutines (one per active reconciliation)
│     Runs three-phase provisioning/update for one deployment
│     Uses BoltDB to persist phase checkpoint
│     Calls Terraform CLI and Kubernetes API
│
├── Health Monitor goroutine pool
│     Polls every managed deployment every 30 seconds
│     Parallelism: all deployments polled concurrently with timeout per check
│     Does not block on command processing
│
└── Metrics Collector goroutine
      Scrapes Prometheus endpoints from all deployment namespaces
      Aggregates billable metrics hourly
      Pushes to Control Plane billing stream
```

### 10.8 BoltDB applied-version map at scale

The Agent's BoltDB stores an entry per deployment: `{deployment_id → applied_state_version, phase_checkpoint}`. This map is what allows the Agent to deduplicate replayed commands after a reconnect and to resume in-progress reconciliations after a crash. With hundreds of deployments per cluster, the BoltDB file size is bounded and predictable — each entry is small (a UUID and two integers), and the deployment count per cluster is bounded by the Starter-tier quota ceiling.

### 10.9 Terraform state partitioning

The Deployment Controller runs Terraform for Phase 1 provisioning. Each deployment has its own Terraform state file in S3 at the path `s3://flowdeck-tfstate-{region}/{dp_id}/{deployment_id}/terraform.tfstate`, with a separate DynamoDB lock key `{dp_id}-{deployment_id}`. This means concurrent Phase 1 operations for different deployments do not compete for the same Terraform state file or lock. Two deployments being provisioned at the same time proceed fully in parallel without coordination.

### 10.10 Agent resource requirements

The Agent pod's resource profile scales with the number of deployments it manages:

| Deployments managed | Recommended Agent pod resources |
|---------------------|----------------------------------|
| 1–50 | 2 vCPU, 2 Gi memory |
| 50–200 | 4 vCPU, 4 Gi memory |
| 200–500 | 8 vCPU, 8 Gi memory |

The Agent is scheduled on the system node pool with a dedicated `PodDisruptionBudget` (`minAvailable: 1`) so that Karpenter node consolidation never evicts it. The Agent's `PersistentVolumeClaim` for BoltDB is on a gp3 EBS volume that is not deleted on pod termination — this preserves the applied-version map and phase checkpoints across pod restarts.

## 11. Observability and billing in shared mode

In a shared cluster, metrics and traces must be attributable to individual deployments and organisations. This is enforced by labelling rather than physical separation.

### 11.1 Metrics attribution

Every Prometheus metric scraped from a deployment namespace carries the following labels, injected by the Metrics Collector at scrape time:

```
org_id="{org_id}"
workspace_id="{workspace_id}"
deployment_id="{deployment_id}"
isolation_tier="{STARTER|PRO|ENTERPRISE}"
```

These labels are what allow the platform to produce per-customer dashboards, per-organisation billing aggregates, and per-tier SLO tracking from a single shared AMP workspace. A query like `sum(airflow_worker_cpu_usage) by (org_id)` gives per-organisation CPU consumption across all their deployments on the shared cluster.

### 11.2 Billable metric aggregation

The Metrics Collector aggregates the following billable dimensions hourly, per deployment:

| Metric | Collection method |
|--------|-----------------|
| Worker-hours | `sum(worker_pod_running_seconds)` by deployment_id / 3600 |
| DAG run count | `sum(airflow_dagrun_duration_success_count)` |
| Task execution count | `sum(airflow_ti_finish_count)` |
| Storage GB-hours | S3 bucket `BytesStored` CloudWatch metric |

Hourly aggregates are pushed to the Control Plane Billing Service via the same gRPC stream used for status updates. The Billing Service rolls them into monthly invoices per organisation, summing across all deployments.

In a shared cluster, two customer deployments may run on the same EC2 node. Billing is based on Airflow metrics (tasks, runs, worker-hours) rather than EC2 node-hours, so shared node occupancy does not distort the per-customer invoice.

### 11.3 Distributed tracing in shared mode

Every Airflow API call and every Deployment Controller action is wrapped in an OpenTelemetry span. The span carries `deployment_id` as an attribute, which makes it possible to filter all traces for a single deployment in Jaeger or X-Ray even though the spans are emitted from shared services (the Agent, the ingress controller).

When an SRE needs to diagnose a slow provisioning event for a specific Starter-tier deployment among hundreds, they filter by `deployment_id` in the trace tool — all spans in that trace, from the Control Plane command dispatch through the Terraform apply through the Helm release through the health validation, are visible in one view.

## 12. Upgrade and maintenance lifecycle

Cluster-level upgrades in the shared data plane affect all deployments on that cluster. The upgrade process is designed to be non-disruptive but acknowledges that shared infrastructure maintenance windows have a wider blast radius than dedicated cluster maintenance.

### 12.1 EKS Kubernetes version upgrades

EKS upgrades are applied per pool member independently. Because the pool has multiple members, one member can be upgraded while others continue accepting new deployment assignments and serving existing workloads. The upgrade sequence per pool member:

```
1. Upgrade EKS control plane to target version (AWS-managed, ~15 min)
   - EKS API server rolls over; existing nodes continue running
   - No deployment disruption during control plane upgrade

2. Upgrade system node pool (flowdeck-system nodes)
   - Karpenter drains system nodes one at a time (disruption budget: 1 at a time)
   - Platform system pods (Agent, KEDA, ingress controller) are rescheduled
   - Brief Agent reconnect (< 60s); commands queue in NATS during reconnect
   - Customer deployments unaffected during system node roll

3. Upgrade Starter shared node pool
   - Karpenter drains shared nodes with disruption budget: max 10% simultaneously
   - Customer worker pods are gracefully drained (Airflow drain hook)
   - Worker pods reschedule on new nodes after drain completes
   - Scheduler and webserver pods restart on new nodes
   - Deployment health check detects restart, confirms HEALTHY after pod comes up
   - Expected per-deployment disruption: scheduler restart ~2 min, workers ~5 min

4. Upgrade Pro org node pools (one pool at a time, same drain procedure)
   - Only the org whose pool is being upgraded is affected
```

Pool members are upgraded one at a time in a rolling sequence. While one member is being upgraded, it is marked `UPGRADING` in the registry and excluded from new assignment routing — new deployments go to other healthy members. The upgrade window per member is approximately 30–45 minutes. With a 10-member pool, the full region upgrade completes in a few hours without any single moment of zero regional capacity.

### 12.2 Airflow runtime upgrades

Airflow version upgrades for a deployment are customer-initiated through the desired state API. They go through the normal `UPDATING` state machine flow (Phase 2 + Phase 3 only — no Terraform changes). Multiple customer deployments can upgrade simultaneously because the Helm chart rendering is per-deployment and the Kubernetes rolling update is per-namespace.

The Deployment Controller's command parallelism (20 concurrent operations) limits simultaneous in-flight upgrades to 20, preventing a thundering herd of rolling restarts from overloading the cluster's image pull bandwidth.

### 12.3 KEDA and Karpenter operator upgrades

KEDA and Karpenter run in the system node pool and are upgraded by the Data Plane Agent applying a new Helm release in `flowdeck-system`. During the KEDA operator upgrade (~2 minutes), `ScaledObject` reconciliations are paused — existing worker replica counts are frozen at their current value. Scaling resumes when the new KEDA pod is ready. There is no customer-visible impact unless a scale event was pending exactly during the upgrade window.

## 13. Capacity planning model

### 13.1 Inputs

Capacity planning operates at two levels: **per pool member** (how many deployments can one cluster handle) and **pool level** (how many members does a region need).

**Per pool member inputs:**

| Input | Source |
|-------|--------|
| Max deployments per member (`dp_max_deployments`) | Configured cap, derived from cluster sizing |
| Average `max_workers` per deployment | Observed desired state distribution |
| Peak concurrency factor | P95 of (active deployments / total) at peak hour |
| Worker pod CPU request (P95) | Observed per-deployment metrics |
| Node pool CPU ceiling | Karpenter NodePool `.limits.cpu` |

**Pool level inputs:**

| Input | Source |
|-------|--------|
| Total deployments in region | Current count + growth forecast |
| Target capacity utilisation per member | `SCALE_OUT_RATIO_THRESHOLD` (default: 0.70) |
| Provisioning lead time | ~20 min for a new pool member |
| Growth rate | Week-over-week deployment count delta |

### 13.2 Per pool member CPU ceiling formula

```
cluster_cpu_ceiling (per member) =
    dp_max_deployments
  × avg_max_workers
  × peak_concurrency_factor          # typically 0.3–0.5 for Starter tier
  × avg_worker_cpu_request           # typically 1–2 vCPU per worker
  + system_overhead                  # ~8 vCPU for flowdeck-system pods
  × headroom_factor                  # 1.2× to absorb simultaneous bursts
```

**Example:** 300 deployments per member × 4 max workers × 0.4 peak × 1.5 vCPU + 8 overhead × 1.2 ≈ **882 vCPU per member**. At m7i.2xlarge (8 vCPU), approximately 111 nodes per member at peak.

### 13.3 Pool size formula

```
required_pool_members =
    ceil(total_region_deployments / dp_max_deployments / target_utilisation)

target_utilisation = 1 - SCALE_OUT_RATIO_THRESHOLD  # default: 0.70 → use 70% of capacity
```

**Example:** 2,000 deployments in us-east-1, `dp_max_deployments = 300`, target utilisation 70%:
`ceil(2000 / 300 / 0.70)` = `ceil(9.5)` = **10 pool members**.

Add `MIN_POOL_SIZE = 2` as a floor, and provision one extra member as a warm spare above the formula result.

### 13.4 When a pool member approaches its ceiling

When a member's `capacity_score` crosses 0.75, the routing function stops assigning new deployments to it and the Pool Controller evaluates whether pool scale-out is needed (see Section 4.5.3). The member continues serving its existing deployments at full capacity — only new assignment is paused. No manual SRE action is required for routine growth.

### 13.5 Per-org ceiling for Pro tier

Each Pro org's Karpenter NodePool has its own `limits.cpu` and `limits.memory`. The Deployment Controller sets these limits when it creates the org's node pool, derived from the org's plan quota. The Pro org's CPU ceiling is:

```
pro_cpu_ceiling =
    sum(max_workers across all deployments in this org)
  × avg_worker_cpu_request
  × 1.2 headroom
```

This is recalculated and the NodePool limits updated whenever a Pro deployment's desired state changes its `max_workers`.

## 14. Decision guide — tier selection and migration

### 14.1 When to upgrade from Starter to Pro

| Signal | Threshold |
|--------|-----------|
| Worker pods frequently in `Pending` due to shared node saturation | More than 5% of scale-up events result in > 5 min wait for node provisioning |
| SLO violations due to noisy neighbours | Any repeated `DEGRADED` events without a clear per-deployment cause |
| Compliance or audit requirement for node-level isolation | Any such requirement — move to Pro immediately |

### 14.2 When to upgrade from Pro to Enterprise

Enterprise gives a fully dedicated data plane — Agent, cluster, Karpenter, KEDA, ALB. Move when any of the following apply:

| Signal | Description |
|--------|-------------|
| Full platform isolation required | Compliance framework requires control-layer components (Agent, Karpenter, KEDA) not shared with other organisations |
| Independent maintenance window required | Compliance or SLA requirement for Flowdeck upgrades to occur within a customer-specified window |
| Compliance: dedicated compute + data | PCI DSS, HIPAA qualifying configurations requiring dedicated database, network, and compute |
| Very high deployment count | More than ~50 deployments in one org — at this scale, a shared Agent risks command queue head-of-line blocking |
| Custom node pool configuration | Custom instance types, spot/on-demand ratio, or AZ affinity not expressible in the shared cluster's NodePool |
| BYOK KMS | Customer-managed encryption keys require Enterprise |
| Cross-AZ HA with dedicated RDS Multi-AZ | Always requires Enterprise |

### 14.3 Migration mechanics

Upgrading a deployment's `isolation_tier` is an in-place operation through the desired state API. The Deployment Controller receives an `UPDATE` command, computes the diff, and runs the appropriate phases:

```
Starter → Pro:
  Phase 1: Create Pro org node pool (if first Pro deployment in the org)
           Add toleration to deployment pods (pod spec change → Helm upgrade)
  Phase 2: Helm upgrade (add toleration annotation)
           Rolling restart of all deployment pods onto the new tainted node pool
  Phase 3: Validation (webserver health, scheduler heartbeat)
  Expected downtime: zero for scheduler/webserver (rolling update),
                     possible task delay during worker drain (~5 min)

Pro → Enterprise:
  Pre-step (SRE): Flowdeck provisions a dedicated Data Plane cluster and
                  registers it in the DP Registry as MANAGED_DEDICATED with
                  org_id binding. No customer action required.
  Phase 1: Deployment Controller receives UPDATE command targeting new dp_id
           Provisions dedicated EKS cluster via Terraform (15–30 min)
           Provisions per-deployment RDS, ElastiCache (8–12 min, parallel)
  Phase 2: Helm upgrade deploying deployment namespace into new cluster
  Phase 2b: Metadata DB migration — pg_dump from old RDS → restore to new RDS
            (deployment in UPDATING state; DAG runs paused ~5 min during cutover)
  Phase 3: Validation on new cluster
  Post-step: Old Pro node pool namespace decommissioned
  Routing change: DP Registry update → all subsequent commands for this org
    automatically route to the MANAGED_DEDICATED dp_id; no API change required
  Expected downtime: 5–10 min during metadata DB cutover
```

The `Pro → Enterprise` migration involves a metadata database copy, handled by the Deployment Controller using a pre-built migration runbook tested in the Controller's integration test suite.

## Closing note

The shared data plane is designed around one principle: **compute can be shared; data cannot**. Kubernetes namespaces, ResourceQuotas, NetworkPolicies, and Karpenter node pools handle compute isolation with acceptable trade-offs for most workloads. RDS, ElastiCache, S3, and IAM are always dedicated because the cost of sharing data — blast radius, regulatory risk, debugging complexity — far outweighs the infrastructure savings.

Worker autoscaling is fully per-deployment through KEDA's independent ScaledObjects; node autoscaling is shared at the pool level but never competes at the deployment level.

When a customer's workload demands full isolation, Enterprise tier provides a dedicated data plane — Agent, cluster, Karpenter, KEDA, and ALB — still fully operated by Flowdeck. The customer does not take on infrastructure responsibility; they simply stop sharing it. The same Control Plane, the same APIs, and the same desired-state contract work identically at every tier.
