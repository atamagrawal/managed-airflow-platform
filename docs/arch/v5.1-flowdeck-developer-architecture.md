## 1. Executive Summary

Flowdeck sells managed Apache Airflow as an enterprise service. Customers care about reliable orchestration, secure access, predictable cost, and compliance-friendly operations. Behind those outcomes sits a platform that must remain **one product** across multiple placement models: Flowdeck-hosted public cloud today, customer VPC hybrid tomorrow, and packaged private or multi-cloud later. That ambition fails if each mode becomes a separate codebase or a separate behavioral contract.

Version 5.0 establishes *how the platform is operated and secured in AWS*: accounts, networking, isolation tiers, observability, DR, and compliance posture. Version 5.1 establishes *how the platform behaves and how engineering should implement it*: what “truth” means in the system, how intent becomes infrastructure and running Airflow, and which contracts must stay stable when topology changes.

The architectural spine is deliberate separation between **intent** (what should exist), **convergence** (making reality match intent), and **execution** (running DAGs and tasks). The Control Plane records authoritative business decisions and publishes versioned intent. The Data Plane reconciles cloud and Kubernetes resources and reports what it sees. The Runtime Plane runs Airflow components and customer workloads. Communication from runtime infrastructure back to Flowdeck’s brain is **outbound-only** from the customer’s perspective in hybrid mode, which is why the same protocols serve managed and hybrid without rewriting customer-facing semantics.

This document explains those boundaries in enough depth that a product manager can reason about roadmap constraints, an architect can defend design decisions to customers, and a principal engineer can implement features without collapsing planes or breaking replay guarantees.

## 4. Customer-Visible Product Behavior

From the customer’s point of view, Flowdeck provides organisations and workspaces, invitations and SSO, quotas and billing constructs, and **deployments**—each deployment being an Airflow environment with a lifecycle customers can observe (creating, updating, scaling, stopping, failing with explained reasons). Customers integrate through the UI, CLI, and API keys used in CI. They care that **what they declare** (Airflow version, executor type, resource profile, environment variables, connections as references, DAG artifacts) becomes reality without silent divergence.

Customers also care that **audit** answers “who changed what, when, and under which policy,” that **secrets** are not stored as plaintext in vendor databases, and that **hybrid** does not mean a second product with different verbs or different semantics for the same API field.

Version 5.1 makes explicit that customer-visible reliability is not only uptime of pods; it is correctness of the **intent pipeline**. A deployment can have healthy pods yet be wrong if observed state and desired state disagree without surfacing that drift. Engineering discipline around separate persistence of desired and observed state is therefore a product requirement, not an implementation detail.

## 5. The Three Planes as a Contract, Not Just a Diagram

The Control Plane is the **authority on decisions Flowdeck is willing to record**. Identity, authorization, workspace policy, quota admission, billing-related facts, and desired deployment configuration live here. When the Control Plane accepts a write, it is saying: “This intent is valid under tenant rules and is now part of the formal record.” It does not provision RDS, apply Helm, or restart scheduler pods.

The Data Plane is the **authority on mutations and observation**. It translates accepted intent into Terraform phases, Helm releases, Kubernetes object graphs, secret materialization into namespaces, and health signals derived from probing real systems. When the Data Plane reports status, it is saying: “This is what exists and how healthy it appears from the reconciler’s vantage.” It does not reinterpret RBAC or invent quota exceptions.

The Runtime Plane is the **authority on execution**. Schedulers decide run timing subject to Airflow semantics; workers execute operators; task logs and XCom data belong to the workload realm. Runtime components should not silently enforce business rules that belong in the Control Plane (for example, deciding whether a workspace allows production pushes without CI).

This separation exists because **risk and rate of change differ**. Control Plane logic is multi-tenant and policy-rich but changes less violently than reconcilers touching cloud APIs. Data Plane code bears blast radius for provisioning mistakes. Runtime churn is dominated by customer DAG releases and autoscaling. Mixing these concerns in one service produces deployments that are hard to upgrade, hard to audit, and dangerous to expose across hybrid boundaries.

The **outbound-only** posture described in 5.0 is part of the contract: the Control Plane does not open inbound administrative paths into customer networks for reconciliation. Agents initiate streams, authenticate strongly, and consume command traffic as durable work. Principal engineers should implement features assuming **disconnects, retries, and duplicate commands** are normal, not exceptional.

## 6. Tenant Hierarchy and Why It Shapes Every Feature

An **organisation** is the commercial and governance boundary: billing, spend caps, global policies, org-wide roles. A **workspace** is how enterprises carve teams and environments inside an organisation; it carries defaults such as allowed Airflow versions, executor choices, and whether production changes must originate from automation rather than interactive users. A **deployment** is one Airflow environment instance bound to a workspace, with its own lifecycle and runtime footprint.

This hierarchy is not cosmetic naming. It defines **authorization scope** (who may mutate what), **quota scope** (consumption counted where), **audit scope** (events attributable to which container of responsibility), and **notification routing** (which Slack channel hears about a scheduler regression). When a new capability is proposed—custom connections, private Python indexes, cross-deployment promotion workflows—the first architectural question is **which scope owns the policy** and **which scope owns the secret reference**.

The Data Plane registry concept from 5.0 fits here logically but at a different dimension: it selects **which reconciler** should receive commands for a deployment based on region, health, and placement mode. Product narratives should treat registry routing as internal plumbing while preserving the customer illusion of a single consistent deployment API.

## 7. Desired State and Observed State

**Desired state** is the structured document the Control Plane persists after successful validation and admission. It describes the customer’s requested configuration: Airflow version, executor, sizing, feature flags, references to connections and variables, DAG delivery pointers (image digest or equivalent), and placement hints encoded in product-level terms. Each successful mutation advances a monotonic `desired_state_version` for that deployment.

**Observed state** is what the Data Plane knows from reality: phase progress, readiness, health rollup, endpoint information, last successful reconcile markers, and diagnostic hints. It converges toward desired state over time but **never overwrites** desired state as a shortcut. If reconciliation could mutate desired records, customers could lose the ability to answer “what did we ask for?” independently of “what broke while applying it?”—an unacceptable enterprise gap.

Drift is an operational fact. The desired versus observed split enables explicit product behavior: show both in UI for operators, drive alerts off meaningful divergence classes, and avoid pretending Kubernetes status JSON is the same as contractual intent.

Principal engineers should implement writers so that **every Control Plane mutation transaction** that bumps desired version also has a coherent command publication strategy; conversely, observed-state writers must tolerate delayed commands during outages without corrupting intent tables.

## 8. Versioning, Commands, and Why Reconciliation Must Be Idempotent

Commands are not informal RPCs like “restart scheduler.” They carry **versioned intent**: converge deployment D to desired_state_version N (with correlation metadata and typed command classification such as create, update, scale, delete). Delivery may be **at least once** due to crashes, network partitions, broker redelivery, or agent restarts.

Therefore reconciliation must be **idempotent** at the business-outcome level. Applying version N twice must yield the same end state as applying it once. Older versions must be ignored once a newer version has been accepted or applied according to defined rules. Duplicate deliveries must not create duplicate infrastructure or corrupt Helm release history.

Acknowledgment semantics matter for correctness. The Agent acknowledges command acceptance **after** internal controllers durably record that work is accepted, not after full convergence completes. That distinction prevents message loss from silently dropping intent while still allowing long-running reconciliations to proceed asynchronously. Completion is communicated through status events and observed-state snapshots.

Principal engineers should design controllers around **diff-and-apply** patterns grounded in cloud and Helm state, not around assuming “this command runs exactly once.” Tests must exercise duplicates, out-of-order older versions, and mid-flight restarts.

## 9. Deployment Lifecycle

Customers experience deployments moving through coarse lifecycle phases such as pending provisioning, configuring runtime, starting components, running healthy, updating in place, scaling, degraded health with explanation, stopped states, and terminal failure when automation cannot safely proceed without human remediation.

Architecturally, lifecycle is a **state machine** with guarded transitions. Entry into each state should imply explicit prerequisites: for example, entering running requires validation gates that prove scheduler and webserver health endpoints respond and critical dependencies connect. Transitions emit audit-facing signals suitable for enterprise change records.

Failures must be **classified**. Transient cloud quota errors differ from invalid Helm values differ from DAG parse failures. The platform should surface stable machine-readable reason codes alongside human guidance because CI systems and ITSM integrations consume those codes.

Product managers should treat lifecycle complexity as something customers pay attention to only when it is calm and truthful; architects should ensure new states are rare and justified because each state multiplies UX, support playbooks, and cross-mode parity obligations.

## 10. Control Plane

The Control Plane hosts customer-facing APIs and internal domain services described in 5.0: Identity, Organisation and Workspace management, Deployment orchestration, Data Plane registry logic, audit logging, billing ingestion interfaces, notifications, and supporting data stores.

Its architectural obligation is to remain the **single place** where policy is adjudicated before irreversible intent is recorded. Examples include enforcing workspace CI/CD-only rules for production-like workspaces, refusing deployments when quotas would be exceeded, validating that requested Airflow versions are allowed for that tenant tier, and ensuring secrets exist only as references with backend identifiers rather than cleartext.

After accepting a mutation, the Control Plane increments desired version, emits commands onto durable messaging infrastructure, and writes audit entries capturing actor identity, tenant scope, correlation identifiers, and semantic before/after snapshots suitable for later forensic reconstruction.

Rate limiting, authentication modes, and circuit breaking patterns belong here at the edge (see 5.0). Principal engineers extend behaviors by adding domain validation and transactional writes **before** cloud mutations occur, not by leaking enforcement into reconcilers where bypass becomes tempting during incidents.

---

## 11. Data Plane

The Data Plane houses the Agent that maintains authenticated streams to the Control Plane and internal controllers such as the Deployment Controller, Health Monitor, Metrics Collector, and Secret Distributor described in 5.0.

Architecturally, reconciliation should proceed in **phases** with explicit budgets and failure semantics: provision or update managed dependencies, apply Kubernetes runtime manifests, validate readiness. Phase boundaries exist because failures differ: transient AWS API errors warrant bounded backoff; invalid configuration should fail fast with actionable errors; runtime readiness failures may leave the deployment degraded with logs exposed rather than silently rolling back in ways that hide root causes.

Health monitoring continuously evaluates runtime components beyond command-driven transitions. Automated remediation (restart sequences, controlled rollouts) may occur here, but **policy decisions** about whether a customer may perform an action remain in the Control Plane.

Secret distribution exemplifies boundary discipline: the Control Plane stores references; the Data Plane reads backing stores using narrowly scoped identities and materializes Kubernetes Secrets for pods. Rotation becomes an event-driven reconcile concern rather than a surprise plaintext migration through APIs.

---

## 12. Runtime Plane

The Runtime Plane is the Airflow footprint per deployment: namespaces, workloads, metadata database, broker where applicable, DAG storage or image bundles, autoscalers, ingress routes. Isolation mechanics (quotas, network policies, IRSA-scoped identities) implement tier promises from 5.0.

DAG delivery strategy shapes engineering significantly. The preferred enterprise path treats DAGs and Python dependencies as **immutable versioned artifacts** (container image digest with verification), promoted through APIs so audit trails bind code identity to `desired_state_version`. Alternate compatibility modes (git-sync, object sync) remain secondary because they complicate provenance and enterprise approval workflows—product positioning should not pretend they are equivalent without qualification.

Executor choice (Celery versus Kubernetes) changes scaling patterns and isolation characteristics but should not change Control Plane vocabulary; reconcilers render different Helm values while preserving customer-facing configuration semantics.

## 13. End-to-End Narratives

**Creating a deployment.** A customer submits intent through UI or API. Identity establishes who they are; authorization confirms workspace role; Organisation and Workspace services enforce policy and quota atomically enough to prevent races; Deployment Service persists initial desired state at version one and emits a create command; audit records the change; UI receives progress via streaming channels as Data Plane phases execute and observed state advances until endpoints appear and health gates pass.

**Publishing DAG code.** CI authenticates with a deployer-scoped credential; API validates signatures and workspace rules; desired state advances with a new artifact pointer; update commands trigger image rollout; controllers verify parse health before declaring success; failures explain DAG parser output without leaving production on a broken revision.

**Scaling and bursts.** Autoscaling components respond to queue depth and scheduling pressure as described in 5.0. Product relevance is transparent worker growth with billing dimensions fed upstream; architecture relevance is that scaling signals are observed phenomena while scaling limits remain policy encoded in desired configuration and tier capability.

**Connectivity loss.** If an agent loses the Control Plane temporarily, already accepted intent continues to reconcile locally while new commands queue durably; replay restores ordering assumptions via versioning; customers see honest degraded connectivity states rather than silent divergence.

These narratives are the spine of demos, docs, and QA scenarios; they should remain stable across modes even when operators change underneath.

## 14. Public APIs and Contract Evolution

Public APIs are multi-year contracts with CI systems and enterprise integrations. Evolution should bias toward **additive** changes: optional fields, new endpoints, extended enums with backward-compatible defaults. Breaking removals require explicit versioning strategies and migration windows communicated as product changes, not surprises discovered by parsers.

Errors must be structured so automation can respond: stable reason codes, retry guidance where legitimate, correlation identifiers tying UI experiences to logs. Generic internal errors belong in logs; customer-facing failures should translate into understandable remediation without leaking sensitive internals.

Architects reviewing external integrations should insist that **semantic stability** travels with hybrid and private packaging; customers frequently prototype on managed cloud then migrate placement without rewriting pipelines.

## 15. Security, Secrets, and Audit from a Feature Perspective

Authorization must always be enforced server-side on mutations; UI checks improve usability but never substitute for Control Plane enforcement. Tenant scoping filters belong at the earliest query boundary to prevent cross-tenant leakage classes that audits cannot gracefully recover.

Secrets never persist as plaintext in Control Plane databases; references and rotation metadata do. Runtime reads occur through scoped identities so compromise of one deployment namespace does not imply bulk secret exposure.

Audit is more than compliance checkboxing: it is the reconstruction backbone for support and governance. Features that mutate sensitive configuration without audit artifacts should be rejected in design review.

---

## 16. Deployment Modes Without Forking the Product

Managed public cloud, hybrid agent, remote execution extensions, and private packaging differ in **where agents run and who operates underlying accounts**, not in the vocabulary customers use to describe deployments. Hybrid customers still create deployments through the same APIs; they install an agent that maintains outbound connectivity; reconciliation code stays unified.

Architects should challenge any proposal that introduces “hybrid-only endpoints” or contradicting lifecycle meanings. Placement-specific differences belong in Data Plane provider adapters and operational responsibility matrices (covered narratively in 5.0), not in forked business logic.

---

## 17. Alignment Between Product Roadmap and Technical Constraints

Roadmap items succeed when they declare upfront **which plane owns policy**, **what desired-state fields change**, **what reconciliation phases touch**, and **what audit evidence exists**. Features that look small in UX often imply schema migrations, idempotency proofs, and tier gating across Starter, Pro, Enterprise, and Hybrid.

Principal engineers should escalate when a feature asks reconcilers to reinterpret tenant rules because that coupling becomes expensive incident debt. Product managers should recognize that promising hybrid parity without honoring outbound-only and versioning contracts forces duplicate products—a strategic cost far beyond one sprint.

---

## Closing Note

Version 5.0 proves Flowdeck can be operated like an enterprise AWS platform. Version 5.1 asserts Flowdeck must also **behave like one coherent product**: intent recorded authoritatively, reality reconciled honestly, execution isolated cleanly, and contracts preserved as placement evolves. When engineering and product disagree, return to that spine—the disagreement usually clarifies quickly.
