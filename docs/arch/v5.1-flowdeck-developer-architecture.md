## 1. Why this document exists

The v5 enterprise document explains where systems run, how they are secured, and how SRE operates them.
This v5.1 document explains how we **build product features** on top of that platform.

This document is written in simple English, but it is technical by design.
It tells you what each component must do, what it must never do, and how requests move from API call to running Airflow behavior.

If v5 answers **"where things run"**, this document answers **"how things work in code and product behavior."**

## 2. One system, three planes

Flowdeck has three planes. This is not just a diagram. This is the core contract.

The **Control Plane** owns product intent and policy.  
The **Data Plane** owns infrastructure and runtime reconciliation.  
The **Runtime Plane** runs customer Airflow workloads.

If we mix these boundaries, we get hidden logic, hard incidents, and mode-specific bugs.

### 2.1 Control Plane

Control Plane is the source of business truth. It owns:

- identity and role checks
- org/workspace/deployment APIs
- policy and quota decisions
- desired state of each deployment
- audit and billing events

Control Plane must **not** directly run Terraform, Helm, or Kubernetes mutations.

### 2.2 Data Plane

Data Plane takes desired state and makes reality match it. It owns:

- command intake and dedupe
- provisioning and update steps
- health checks and observed status
- secret materialization for runtime pods

Data Plane must **not** decide user policy or tenancy rules.

### 2.3 Runtime Plane

Runtime Plane is Airflow itself:

- webserver
- scheduler
- triggerer
- workers
- metadata DB and broker integrations

Runtime Plane runs workloads. It must not become a policy engine.

## 3. Non-negotiable engineering rules

These are hard rules. Every feature must keep them true.

1. **Desired state is owned by Control Plane.**  
   Only API/domain services write desired state.

2. **Observed state is owned by Data Plane.**  
   Reconcile and health components report what is running.

3. **Desired and observed state are separate data paths.**  
   Never let reconcile code silently rewrite desired intent.

4. **Commands are versioned.**  
   Every deployment mutation increments `desired_state_version`.

5. **Reconciliation is idempotent.**  
   Same command delivered twice must not create different outcomes.

6. **Data Plane communication is outbound-only.**  
   Control Plane does not open inbound channels into customer VPCs.

7. **Every mutation is auditable.**  
   Actor, scope, before/after, reason, and correlation ID must exist.

## 4. Product domain model

We use a strict hierarchy:

**Organization -> Workspace -> Deployment**

This is more than naming:

- authorization scope follows this hierarchy
- quota checks follow this hierarchy
- audit scope follows this hierarchy
- notification routing follows this hierarchy

### 4.1 Organization

Organization is the commercial and governance boundary.
Billing ownership, high-level quotas, and org-level admin policy live here.

### 4.2 Workspace

Workspace is the team/environment boundary.
This is where we set delivery policy like "production updates must come from CI token only."

### 4.3 Deployment

Deployment is one Airflow environment lifecycle.
It has versioned desired state and continuously updated observed state.

## 5. Desired state vs observed state

This is the most important implementation concept.

### 5.1 Desired state

Desired state is what customer and policy asked for.
Examples:

- Airflow version
- executor type
- compute profile
- environment config
- connection/variable references
- DAG artifact reference (image digest, etc.)

Desired state is written by Control Plane services only.

### 5.2 Observed state

Observed state is what Data Plane sees in real systems.
Examples:

- reconcile phase progress
- component health rollups
- current endpoint status
- last successful applied version
- warnings and failure reasons

Observed state is written by Data Plane only.

### 5.3 Why separation matters

If we merge both in one path, we lose incident clarity.
Then we cannot answer:

- "What did customer ask for?"
- "What is running now?"
- "Where did failure happen?"

In enterprise systems, that is unacceptable.

## 6. State machine and lifecycle behavior

A deployment has explicit lifecycle states. We do not hide transitions.

Core path:

`PENDING -> PROVISIONING -> CONFIGURING -> STARTING -> RUNNING`

Operational branches:

- `RUNNING -> UPDATING -> RUNNING`
- `RUNNING -> SCALING -> RUNNING`
- `RUNNING -> STOPPING -> STOPPED`
- `STOPPED -> STARTING -> RUNNING`
- any state can move to `ERROR` with typed reason code

Each transition must define:

- entry condition
- completion condition
- timeout budget
- retry rules
- error reason code mapping
- audit event emission

If a new feature introduces a new state, it must define all six items above.

## 7. Request flow: from API to real infrastructure

This section describes how a normal deployment change should work.

### 7.1 API mutation path (Control Plane)

1. Request enters API.
2. Authenticate actor (user token or service token).
3. Authorize against org/workspace/deployment scope.
4. Run policy and quota checks.
5. Persist desired-state update.
6. Increment `desired_state_version`.
7. Emit command event with correlation metadata.
8. Write audit event for the mutation.

Important rule: desired-state commit and command publish must be coordinated.
If desired state is saved but no command is emitted, reconcile stalls.
If command is emitted before desired state is durable, reconcile reads inconsistent intent.

### 7.2 Reconcile path (Data Plane)

1. Agent receives command.
2. Compare incoming version with applied/accepted version.
3. Ignore duplicates and older versions safely.
4. Persist accepted work locally.
5. Run reconcile phases.
6. Emit status updates and observed-state snapshots.
7. Mark version as applied when convergence succeeds.

### 7.3 Acknowledge semantics

A command ACK means **"accepted durably for processing"**, not **"completed successfully."**
Completion is reported through status events and final observed state.

This split is required for crash safety and replay correctness.

## 8. Reconciliation model in detail

Principal-level implementation should use phased reconciliation.

### 8.1 Phase 1: dependency and infra converge

Provision or update managed dependencies and identities.
Typical resources include DB, cache, IAM roles, storage binding, security policies.
Failures here are often cloud API or quota related.

### 8.2 Phase 2: runtime converge

Render deployment config into runtime manifests/Helm values.
Apply to namespace with safe rollout strategy.
Track rollout progress and error signals at component level.

### 8.3 Phase 3: validation gate

Confirm webserver, scheduler, and required integrations pass readiness criteria.
Only then move deployment to `RUNNING`.

### 8.4 Failure strategy

Do not treat all errors the same.

- Retry transient provider errors with bounded exponential backoff.
- Fail fast on deterministic config errors.
- Keep degraded state visible when runtime boot succeeds partially.

Every failure must emit:

- stable machine reason code
- human remediation hint
- retryability classification
- correlation ID

## 9. API and event contract design

### 9.1 Public API rules

API contracts are long-lived. CI pipelines and enterprise tools depend on them.

Rules:

- prefer additive changes
- avoid removing fields without migration path
- keep semantics of existing fields stable
- make error codes deterministic

### 9.2 Command event rules

Command event is an intent envelope, not imperative script.
Minimum command metadata:

- deployment ID
- desired version
- command type
- issue timestamp
- actor context
- correlation ID

### 9.3 Status event rules

Status events are progress evidence.
They should show meaningful state movement and reason codes, not raw noisy logs.
Detailed diagnostics go to logs and traces, linked with correlation ID.

## 10. Security rules for feature implementation

### 10.1 Authorization placement

Auth checks must happen on server write paths.
Client-side role checks are UX only, never security control.

### 10.2 Secret handling

Control Plane stores secret references, not secret values.
Data Plane resolves values and materializes them into runtime namespace using scoped identity.

### 10.3 Tenant isolation

Every query and mutation must apply tenant scope at the earliest boundary.
"Filter later" patterns are not acceptable for sensitive paths.

### 10.4 Privileged paths

Emergency or break-glass operations must use explicit APIs and explicit audit fields.
No hidden admin backdoor behavior in service internals.

## 11. Developer view of major product flows

### 11.1 Create deployment

Control Plane validates request and writes desired version 1.
Data Plane reconciles infra, runtime, and validation phases.
Observed state transitions are streamed to UI/API until `RUNNING`.

### 11.2 Deploy DAG artifact

CI submits signed artifact reference.
Control Plane validates policy and records new desired version.
Data Plane rolls runtime to new artifact and validates DAG parse and startup signals.
On failure, deployment stays on last healthy behavior and reports reason clearly.

### 11.3 Scale under burst

Runtime scaling responds to queue pressure.
Data Plane reports scaling progress and health.
Billing/metering ingestion remains aligned with observed activity windows.

### 11.4 Control channel interruption

If Data Plane loses connection, accepted work continues locally where safe.
New commands queue durably.
On reconnect, replay plus version rules guarantee safe catch-up.

## 12. Multi-mode architecture without product fork

Flowdeck supports managed public cloud first, then hybrid/private extensions.
The customer-facing API and lifecycle behavior must stay the same across modes.

What changes by mode:

- where Data Plane runs
- who owns underlying cloud account operations
- network path specifics

What must not change:

- API verbs and meanings
- state machine semantics
- reason code catalog
- audit model
- desired vs observed contract

If a design requires mode-specific behavior differences at API semantics level, it is architecture debt and should be redesigned.

## 13. How a principal developer should use this document

When starting any new feature, answer these questions first:

1. Which plane owns policy for this feature?
2. Which desired-state fields change?
3. Which reconcile phase is impacted?
4. What are failure reason codes?
5. How does replay/idempotency work?
6. What audit evidence will prove correct behavior?

If any answer is unclear, the design is not ready for implementation.

This approach keeps PM goals, architecture integrity, and engineering delivery aligned.

## 14. Closing statement

Flowdeck succeeds as a platform only when it behaves like one coherent system.
That means:

- intent is recorded clearly
- reconciliation is safe under retry and replay
- runtime execution is isolated and observable
- contracts stay stable as deployment mode changes

v5 explains how we operate the platform.
v5.1 explains how we build it without losing those guarantees.
