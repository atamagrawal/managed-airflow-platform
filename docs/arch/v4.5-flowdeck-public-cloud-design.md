## 1. Executive Summary

### 1.1 Purpose

This document defines the target architecture for AirflowCloud's public cloud deployment on AWS and establishes the control model that must remain stable across future hybrid and private deployment modes. It describes how the platform is structured, how its components interact, and how it is extended to new deployment contexts without breaking the core contract between planes.

### 1.2 Architecture Goals

- Separate tenant intent, orchestration, and runtime execution into distinct planes with well-defined contracts between them.
- Keep the platform control surface centralised, auditable, and secure — always owned and operated by AirflowCloud regardless of where customer workloads run.
- Support deployment portability so that the same Control Plane API governs public cloud, hybrid, and private deployments without modification.
- Isolate each customer deployment so that failures, noisy neighbours, and security incidents do not spill across tenants.
- Provide consistent health, audit, metrics, and operational visibility across all layers and all deployment modes.

### 1.3 Three-Plane Layer Model

The platform is structured around three distinct planes, each with a clear and non-overlapping responsibility:

| Layer | Responsibility | Location |
|---|---|---|
| **Control Plane** | User identity, organisation and workspace state, billing, policies, audit logging, public APIs, and deployment intent (desired state) | AirflowCloud AWS account — always |
| **Data Plane** | Deployment orchestration and reconciliation, Helm lifecycle management, health monitoring, secret distribution, and secure outbound communication with the Control Plane | AirflowCloud VPC (Public Cloud mode) or customer VPC (Hybrid/Private mode) |
| **Runtime Plane** | Actual Airflow workloads — webserver, scheduler, workers, triggerer, metadata database, Celery broker, DAG storage, and task execution | Same environment as the Data Plane for the chosen mode |

> 💡 **Key Principle:** The Control Plane never runs Airflow workloads. The Data and Runtime planes never make decisions — they only execute what the Control Plane instructs and report back what is actually running.

### 1.4 Deployment Modes

| Mode | Data Plane Location | Runtime Plane Location | Who Operates Infrastructure |
|---|---|---|---|
| **Public Cloud** | AirflowCloud VPC | AirflowCloud VPC | AirflowCloud operates everything |
| **Hybrid** | Customer VPC | Customer VPC | AirflowCloud manages Control Plane; customer hosts execution |
| **Private** | Customer VPC | Customer VPC | Customer operates everything under AirflowCloud's platform contract |

### 1.5 Communication Model

The Control Plane holds the source of truth for desired state. It sends orchestration commands to the appropriate Data Plane, which applies them to the Runtime Plane and continuously reports actual state back.

> 🔒 **Outbound-Only Model:** All Data Plane communication is initiated outbound from the Data Plane toward the Control Plane — the Control Plane never opens inbound connections into a Data Plane. This ensures the architecture works identically for Data Planes that live in customer VPCs behind firewalls.

### 2.2 Key Design Decisions

#### Data Plane / Control Plane Separation

The Control Plane handles business logic (RBAC, billing, audit, user management). The Data Plane handles pure infrastructure orchestration. This separation allows relocating the Data Plane to a customer's VPC in Hybrid mode without changing the Control Plane. The contract between them (a gRPC API over mTLS) remains identical regardless of where the Data Plane runs.

#### Runtime Plane / Data Plane Separation

The Data Plane is the orchestrator (runs Helm, watches Kubernetes, manages lifecycle). The Runtime Plane is the workload (runs Airflow). This separation enables:
- The Data Plane to manage multiple Runtime Plane namespaces simultaneously
- Independent health-checking of each Runtime Plane
- Future Runtime Plane variations (different executors, remote execution agents) without changing orchestration logic

#### Outbound-Only Communication

Requiring the Data Plane to initiate all connections outward to the Control Plane means that in Hybrid and Private modes, customers never need to open inbound firewall rules to their VPC. This is a hard requirement for regulated enterprise customers and simplifies network security posture significantly.

#### Shared EKS Cluster per Tier (Not per Customer)

For Starter and Pro tier customers, a shared Runtime Plane EKS cluster with namespace-level isolation (ResourceQuota, NetworkPolicy, IRSA, Secrets Manager path scoping) provides sufficient isolation at dramatically lower operational cost. For Enterprise tier customers, a dedicated EKS cluster per organisation provides hard blast-radius isolation. Hybrid mode customers use their own clusters and are therefore always cluster-isolated by definition.

### 2.3 Tenant Isolation Model by Tier

| Tier | Isolation Level | Shared Resources | Dedicated Resources |
|---|---|---|---|
| **Starter** | Namespace | EKS cluster, node pools, ALB | Kubernetes namespace, ResourceQuota, IRSA role |
| **Pro** | Namespace + dedicated node pool | EKS cluster, ALB | Namespace, node pool (tainted), IRSA role |
| **Enterprise** | Dedicated cluster | Nothing | Full EKS cluster, ALB, RDS |
| **Hybrid** | Customer's own cluster | Nothing (customer owns it) | Full customer VPC and cluster |

### 2.4 AWS Account Structure

AirflowCloud operates separate AWS accounts within an AWS Organisation to enforce billing isolation, IAM blast-radius containment, and compliance boundaries:

| Account | Purpose |
|---------|---------|
| **airflowcloud-prod** | Production Control Plane and Public Cloud Data/Runtime Planes |
| **airflowcloud-staging** | Staging environment, mirrors production topology |
| **airflowcloud-dev** | Development environment for engineering |
| **airflowcloud-security** | Centralised security tooling: GuardDuty aggregator, Security Hub, CloudTrail archive, AWS Config |

Service Control Policies (SCPs) at the Organisation level enforce guardrails that cannot be overridden even by account administrators — for example, preventing creation of static IAM access keys.

### 3.3 Ingress Layer

All external traffic enters the Control Plane through a layered ingress stack designed to protect against abuse before requests reach application services.

#### AWS WAF v2

Sits at the outermost perimeter:
- Enforces OWASP Top 10 rules (SQL injection, XSS, path traversal)
- Applies rate limiting at the IP and organisation level
- Blocks traffic from known malicious IP ranges
- Processes every request before it reaches CloudFront or ALB

#### Amazon CloudFront

Serves the AirflowCloud web UI as a global CDN:
- Caches static assets (JavaScript bundles, CSS, images) at edge locations worldwide
- Terminates TLS using a wildcard certificate managed by AWS Certificate Manager
- Forwards dynamic API requests to the ALB
- Provides AWS Shield Standard for DDoS mitigation at no additional cost

#### AWS Application Load Balancer (ALB)

Handles REST API and gRPC traffic:
- Two separate target groups: HTTP/2 REST (port 8080) and gRPC (port 9090)
- Terminates TLS using ACM wildcard certificate (`*.airflowcloud.io`)
- Routes based on host header:
  - `api.airflowcloud.io` → REST
  - `grpc.airflowcloud.io` → gRPC
- Exposes WebSocket endpoint (`wss://ws.airflowcloud.io`) for real-time deployment status and log streaming

### 3.4 API Gateway Service

The API Gateway Service is the single entry point for all inbound communication. It is a stateless service deployed as a Kubernetes Deployment with three replicas on the Control Plane EKS cluster, scaled horizontally by HPA based on CPU utilisation and request rate.

#### REST API

| Property | Value |
|----------|-------|
| **Endpoint** | `https://api.airflowcloud.io/v1` |
| **Format** | JSON over HTTPS |
| **Authentication** | Bearer token or API key |
| **Validation** | OpenAPI 3.0 schema (malformed requests rejected at gateway with 400) |

#### gRPC API

| Property | Value |
|----------|-------|
| **Endpoint** | `grpcs://grpc.airflowcloud.io:443` |
| **Purpose** | Data Plane Agent lifecycle communication |
| **Authentication** | mTLS + JWT (dual-layer — neither alone is sufficient) |

The Data Plane Agent must present a valid client certificate signed by AirflowCloud's internal CA **and** supply a valid JWT in the `authorization` metadata field. This ensures that neither a stolen JWT nor a stolen certificate alone is sufficient to impersonate a Data Plane.

#### WebSocket

| Property | Value |
|----------|-------|
| **Endpoint** | `wss://ws.airflowcloud.io` |
| **Purpose** | Real-time updates: deployment status transitions, live log streaming, alert notifications |
| **Authentication** | Short-lived WebSocket token (5-minute validity, separate from API JWT) |

#### Rate Limiting

| Level | Limit | Storage |
|-------|-------|---------|
| Per IP address | 1,000 requests/minute | WAF layer |
| Per user | 300 requests/minute | API Gateway (Redis) |
| Per organisation | 5,000 requests/minute | API Gateway (Redis) |

Burst allowances are applied so that short legitimate spikes are not rejected.

#### Circuit Breaker

Applied on calls from the API Gateway to downstream services:
- **Threshold:** 50% error rate over a 10-second window
- **Action:** Circuit opens, API Gateway returns 503 with `Retry-After` header
- **Purpose:** Prevents cascade failures where one degraded service causes the entire API to queue up requests

#### API Versioning

- Managed via URL path prefixes (`/v1`, `/v2`)
- Breaking changes are never made within a version
- Previous versions maintained for minimum 12 months before deprecation

### 3.5 Core Services

#### 3.5.1 Identity Service

The Identity Service is responsible for all authentication and authorisation decisions across the platform. Every inbound request to the API Gateway is validated by the Identity Service before reaching any other service.

##### Authentication Methods

| Method | Description |
|--------|-------------|
| **Email + Password + MFA** | Passwords stored as bcrypt hashes (cost factor 12). TOTP-based MFA (RFC 6238). Access token: JWT, RS256, 1-hour expiry. Refresh token: opaque, 30-day expiry, rotated on each use. |
| **SSO (SAML 2.0 / OIDC)** | AirflowCloud acts as Service Provider. Supports Okta, Microsoft Entra ID, Google Workspace, Auth0. Group mappings auto-assign workspace roles based on IdP group membership. |
| **API Keys** | Long-lived opaque keys for CLI/CI/CD. Stored as SHA-256 hashes. Scoped to organisation (optionally to workspaces). Displayed only once at creation. |
| **Data Plane Service Tokens** | Short-lived JWTs (24-hour expiry, auto-rotated). Scoped to gRPC endpoints only — cannot call user-facing REST endpoints. |

##### Authorisation Model (Hierarchical RBAC)

| Scope | Roles |
|-------|-------|
| **Platform** | Super-administrator (internal only) |
| **Organisation** | `org:owner` (full control + billing), `org:admin` (manage workspaces/users), `org:member` (assigned workspaces only) |
| **Workspace** | `workspace:admin` (full control), `workspace:developer` (deploy, manage DAGs/connections/variables), `workspace:viewer` (read-only) |
| **Deployment** | Fine-grained access for CI/CD service accounts targeting specific deployments |

Roles are additive and inherited downward. An `org:owner` has implicit admin access to all workspaces and deployments. Role assignments are cached in Redis for 5 minutes.

##### JWT Structure

| Field | Description |
|-------|-------------|
| `sub` | User ID |
| `org_id` | Organisation ID |
| `roles` | Array of role assignments with scope |
| `scopes` | Permitted API scopes |
| `jti` | Unique token ID for revocation |
| `iat`, `exp` | Issued at, expiry timestamps |

Tokens are signed with RS256 using a 4096-bit RSA key pair. Public key exposed at `https://api.airflowcloud.io/.well-known/jwks.json`.

##### Token Revocation

Revoked tokens (on logout, password change, or security incident) are added to a Redis blocklist keyed by `jti`. Every request validates the token signature and checks the blocklist. Blocklist entries expire at the token's natural expiry time.

#### 3.5.2 Organisation Service

The Organisation Service manages the top-level tenant hierarchy. An Organisation is the billing and policy boundary — it corresponds to a company or team.

**On organisation creation, the service provisions:**
- Default workspace
- Quota record (max deployments, total resource allocation for plan tier)
- Billing record linked to Stripe customer ID

Organisation slugs are globally unique and URL-safe — they appear in API paths and as subdomain prefixes for customer Airflow UIs.

**Teams:** Logical groups of users within an organisation that can be assigned workspace or deployment roles collectively. Role changes on a team propagate to all members.

**Quotas:** Enforced by the Organisation Service and checked by the Deployment Service before any resource creation:
- Maximum number of active deployments
- Total CPU allocation across all deployments
- Total memory allocation
- Total storage

Quota checks are optimistic (check-then-act) with a distributed lock held in Redis during the check-and-create window.

#### 3.5.3 Workspace Service

The Workspace Service manages workspaces within an organisation. A workspace is a logical grouping of deployments, typically corresponding to a team or environment (e.g., "data-engineering", "ml-platform", "production").

Each workspace has:
- Its own member list and role assignments (independent of organisation)
- Shared configuration: default Airflow version, allowed executor types, CI/CD enforcement flag

**CI/CD Enforcement:** When enabled, prevents deployments from being modified through the UI or CLI by users — only service account API tokens with the `DEPLOYER` role can trigger deploys. This enforces the practice that all changes to production environments go through a pipeline.

#### 3.5.4 Deployment Service

The Deployment Service is the central orchestrator of deployment lifecycle. It holds the desired state for every deployment and translates user intent into commands dispatched to the Data Plane.

##### Desired State

A JSON document stored in the platform database describing exactly what the deployment should look like:
- Airflow version
- Executor type
- Resource profile (CPU/memory limits per component)
- Environment variables
- Connection references
- Variable references

##### Deployment State Machine

```
PENDING → PROVISIONING → CONFIGURING → STARTING → RUNNING
                                                      ↓
                                              UPDATING / SCALING
                                                      ↓
                                              STOPPING → STOPPED
                                                      ↓
                                              TERMINATING → TERMINATED

Any state → ERROR (recoverable failure, triggers alert)
```

State transitions are recorded in the audit log with timestamps and the actor (user or system) that triggered them.

##### Commands Dispatched to Data Plane

| Command | Purpose |
|---------|---------|
| `CREATE` | Provision new deployment |
| `UPDATE` | Apply configuration change |
| `DELETE` | Remove deployment |
| `RESTART` | Restart all Airflow components |
| `SCALE` | Adjust worker count |

Commands are published to the NATS `deployments.commands` stream with the deployment ID and complete desired state document.

##### Idempotency

Every command carries:
- `command_id` (UUID)
- `desired_state_version` (monotonically incrementing integer)

The Data Plane Agent tracks the last applied `desired_state_version` per deployment and ignores commands with a version equal to or lower than what it has already applied.

#### 3.5.5 Data Plane Registry

The Data Plane Registry tracks all Data Plane instances — both managed (AirflowCloud-operated) and hybrid (customer-operated). It is the routing layer that determines which Data Plane receives commands for a given deployment.

**Each Data Plane record contains:**
- Unique ID, display name
- Type (`MANAGED`, `HYBRID`, `PRIVATE`)
- AWS region
- gRPC endpoint
- Health status (`HEALTHY`, `DEGRADED`, `OFFLINE`)
- Current and maximum deployment capacity
- Last heartbeat timestamp
- Cloud account metadata

##### Routing Logic

1. Deployment Service creates a command
2. Registry resolves target Data Plane for the deployment
3. Registry verifies health status:
   - `HEALTHY` → forward command
   - `DEGRADED` → forward command + raise alert
   - `OFFLINE` → queue command in NATS until reconnection

##### Capacity Management

The registry tracks deployments per Data Plane. New deployments are assigned to the Data Plane in the appropriate region with the most available capacity.

##### Health Tracking

| Event | Action |
|-------|--------|
| Heartbeat received (every 15s) | Update health metrics |
| No heartbeat for 60s | Mark `DEGRADED` |
| No heartbeat for 5 min | Mark `OFFLINE`, notify affected customers |

#### 3.5.6 Event Bus (NATS JetStream)

AirflowCloud uses NATS JetStream as the internal event bus between the Control Plane and all Data Plane Agents. NATS JetStream provides durable, ordered, acknowledged message delivery with persistence to disk.

**Deployment:** 3-node StatefulSet on the Control Plane EKS cluster, using EBS gp3 volumes for persistence. Uses NATS Raft-based quorum protocol for leader election and log replication.

##### Key Streams

| Stream | Purpose | Retention Policy |
|--------|---------|------------------|
| `deployments.commands` | Orchestration commands from Deployment Service to Data Plane | Work-queue (deleted after acknowledgment) |
| `deployments.status` | Status updates from Data Plane Agents to Control Plane | Limits (last 100 per deployment) |
| `agents.heartbeat` | Heartbeat signals from Data Plane Agents to Registry | Limits (last heartbeat per agent) |
| `alerts.platform` | Alert notifications from Alertmanager to Notification Service | — |

Data Plane Agents connect to NATS over TLS using their mTLS client certificate. Agents always initiate the outbound connection — the NATS cluster never reaches into a customer's network.

#### 3.5.7 Billing Service

The Billing Service tracks resource consumption and generates invoices. It integrates with Stripe for payment processing.

**Usage metrics consumed from Data Plane Metrics Collector (hourly aggregated):**
- Worker-hours (sum of worker pod uptime)
- DAG run count
- Task execution count
- Storage GB-hours (EFS and S3)

**Capabilities:**
- Aggregates events into monthly invoices
- Syncs usage to Stripe for metered billing
- Enforces spend limits (auto-pause new deployments, notify owner)
- Exposes usage dashboards to organisation owners

#### 3.5.8 Notification Service

The Notification Service routes alerts and platform events to customer-configured destinations.

**Supported channels:**
- Email (AWS SES)
- Slack (webhook)
- PagerDuty (Events API v2)
- Generic HTTP webhooks

Customers configure notification channels at the workspace level. Alert routing rules determine which events trigger which channels.

#### 3.5.9 Audit Logger

Every mutating action in the platform produces an audit log entry. The Audit Logger is a middleware component that intercepts all API Gateway responses and writes audit entries to the platform database.

**Each audit entry records:**
- Organisation ID, workspace ID (if applicable)
- Actor user ID or service account ID
- Actor IP address
- Action type (e.g., `deployment.create`)
- Resource type, resource ID
- Before/after state snapshots for mutations
- Request timestamp, response status

Entries are immutable once written. Partitioned monthly for query performance. Entries older than 90 days are exported to S3 (Parquet format for Athena queries) and deleted from the database. S3 archives retained for 7 years.

### 3.6 Control Plane Data Layer

#### 3.6.1 Platform Database (RDS Aurora PostgreSQL)

Amazon RDS Aurora PostgreSQL 15 cluster deployed in Multi-AZ configuration:
- 1 writer instance (all write operations)
- 2 reader instances (read-heavy queries: deployment listing, metrics aggregation, audit log retrieval)

**Connection Pooling (PgBouncer):** Deployed as a StatefulSet alongside API services. Runs in transaction mode, allowing many application connections to share a small pool of Aurora connections. Each API Server pod has a PgBouncer sidecar with 20 server connections.

**Security:**
- All data encrypted at rest using KMS customer-managed keys (CMK)
- Automated backups retained for 7 days with PITR
- Monthly snapshots exported to S3 for long-term retention
- Enhanced Monitoring and Performance Insights enabled

**Key tables:** `organisations`, `workspaces`, `deployments`, `data_planes`, `users`, `organisation_members`, `workspace_members`, `api_keys`, `audit_logs`, `billing_events`, `notification_channels`, `alert_rules`

The `deployments` table stores desired state as JSONB (flexible schema evolution). Actual deployment status is synced from the Data Plane and stored separately (`status` column, `endpoints` JSONB column).

#### 3.6.2 Session Cache and Rate Limiting (ElastiCache Redis)

ElastiCache Redis is used for short-lived, high-throughput data:

| Use Case | Description |
|----------|-------------|
| JWT blocklist | Revoked token JTIs with TTL equal to remaining lifetime |
| Rate limit counters | Per-user and per-organisation request counts (atomic increment) |
| Session data | Server-side session state for WebSocket connections |
| Role assignment cache | RBAC decisions cached per user for 5 minutes |
| Quota check locks | Distributed locks during check-and-create window |

Deployed as a 3-node cluster in Cluster mode. Encryption in transit (TLS) and at rest (AWS-managed key) enabled.

#### 3.6.3 S3 Artifact Storage

| Use Case | Description |
|----------|-------------|
| Audit log archives | Exported monthly in Parquet format for Athena queries |
| Terraform state | Remote state for all infrastructure, versioned with DynamoDB lock table |
| CLI release artifacts | Versioned binaries for all supported platforms |
| Platform configuration snapshots | Periodic exports for disaster recovery |

## 4. Data Plane Design

### 4.1 Overview

The Data Plane is the infrastructure orchestration layer. It receives commands from the Control Plane, provisions and manages cloud resources and Kubernetes workloads for each customer deployment, monitors their health, and reports actual state back.

In Public Cloud mode it runs in AirflowCloud's VPC. In Hybrid mode it runs in the customer's VPC — the code is identical, only the deployment target changes.

```
│  └──────────────────────────────────────────────────────────────┘    │
└──────────────────────────────────────────────────────────────────────┘
```

### 4.3 Data Plane Agent

The Data Plane Agent is the bridge between the Control Plane and the Data Plane's internal components. It is deployed as a Kubernetes Deployment in the `airflowcloud-system` namespace and is the only component in the Data Plane that communicates directly with the Control Plane.

#### Outbound gRPC Connection

The Agent maintains a persistent, bidirectional gRPC stream to the Control Plane API Gateway for:
- Command delivery (Control Plane pushes commands to Agent)
- Status reporting (Agent pushes status updates to Control Plane)
- Heartbeat signals

Authentication: mTLS (client certificate) + JWT bearer token in gRPC metadata. The Agent always initiates the connection outbound.

#### Command Consumption

1. Agent subscribes to NATS `deployments.commands` stream
2. Validates `desired_state_version` against locally tracked version
3. Dispatches to Deployment Controller
4. Acknowledges NATS message only after Deployment Controller accepts (not after completion)

This ensures at-least-once delivery even if the Agent restarts mid-execution.

#### Status Reporting

The Agent publishes status updates to NATS `deployments.status` stream whenever a deployment's state changes:
- Deployment ID
- New state
- Human-readable message
- Kubernetes endpoint information (Airflow webserver URL)
- Timestamp

At startup, the Agent sends a status update for all managed deployments to re-synchronise with the Control Plane.

#### Heartbeat

Every 15 seconds, the Agent sends a heartbeat to the Data Plane Registry via `agents.heartbeat` NATS stream containing:
- Agent version
- Kubernetes version
- Total and available CPU/memory across all nodes
- Summary of each managed deployment's health

#### mTLS Certificate Lifecycle

| Event | Action |
|-------|--------|
| Registration | Agent receives initial client certificate from Vault PKI (30-day validity) |
| 7 days before expiry | Agent initiates rotation via Control Plane API |
| Rotation | New certificate issued, swapped into gRPC connection, old certificate revoked |

#### Token Rotation

| Event | Action |
|-------|--------|
| Every 24 hours | Agent requests new token from Identity Service using current valid token |
| Rotation | New token stored in memory and Kubernetes Secret |
| Overlap | 30-minute overlap period where both old and new tokens are valid |

#### Agent State Machine

```
REGISTERING → (certificate issued, NATS connected) → HEALTHY
HEALTHY → (heartbeat missed for 60s) → DEGRADED
HEALTHY → (command received) → RECONCILING → HEALTHY
DEGRADED → (reconnected) → HEALTHY
DEGRADED → (missed for 5 min) → OFFLINE → (alert fires)
```

### 4.4 Deployment Controller

The Deployment Controller receives commands from the Data Plane Agent and executes them by orchestrating infrastructure provisioning and Kubernetes resource management.

#### CREATE Command — Three-Phase Provisioning

##### Phase 1: Infrastructure (Terraform)

- Creates RDS PostgreSQL, ElastiCache Redis, EFS/S3
- State stored in S3 with DynamoDB locking
- Resources tagged with deployment/workspace/org IDs
- Runs as in-cluster Job (invoked, completes, exits)

##### Phase 2: Kubernetes Resources (Helm)

- Renders values from desired state + Terraform outputs
- Runs `helm install` (CREATE) or `helm upgrade --atomic` (UPDATE)
- `--atomic` flag: automatic rollback on failure
- Namespace: `ac-dep-{deployment-id}`

##### Phase 3: Validation

- Waits for all pods to reach Ready state
- Verifies webserver `/health` returns HTTP 200
- Confirms scheduler heartbeat in metadata database
- Extracts webserver URL from Ingress → reports `RUNNING`

#### State Management

The Deployment Controller maintains a ConfigMap in `airflowcloud-system` namespace for each deployment, recording:
- Last successfully applied `desired_state_version`
- Terraform workspace status
- Helm release revision

This ConfigMap is the local ground truth — on restart, the controller reads these to reconstruct state without querying the Control Plane.

#### Reconciliation Loop

Runs every 60 seconds per deployment:

1. Compare `desired_state_version` in ConfigMap with latest from NATS
2. If different → apply delta
3. If same → check actual Kubernetes state matches desired state
4. If drift detected (e.g., pod manually deleted) → re-apply Helm release

This makes the system self-healing without explicit reconcile commands.

#### Update Flow

For configuration changes (environment variable, resource profile):
1. Run `helm upgrade --atomic` with new values
2. Rolling update: scheduler and webserver updated one pod at a time
3. Workers drained of running tasks before replacement (graceful shutdown)
4. Typical downtime: zero for webserver, <30 seconds for scheduler

#### Delete Flow

1. Run `helm uninstall` to delete all Kubernetes resources
2. Run `terraform destroy` to delete RDS, ElastiCache, EFS
3. RDS final snapshot taken before deletion (configurable)
4. Delete namespace
5. Report status as `TERMINATED`

### 4.5 Health Monitor

The Health Monitor runs independent of the Deployment Controller. It continuously probes the health of every deployment and maintains per-deployment health status included in Agent heartbeats.

#### Probe Schedule (Every 30 seconds per deployment)

| Probe | Method | Success Criteria |
|-------|--------|------------------|
| **Webserver** | HTTP GET `/health` | HTTP 200 within 5 seconds |
| **Scheduler** | Pod status + DB heartbeat | Running + Ready + heartbeat updated within 60s |
| **Workers** | Pod status + queue depth | Running + queue not growing unboundedly |
| **Database** | `SELECT 1` | Connection successful |
| **Broker** | Redis PING | Response received |

Three consecutive failures change component status to `UNHEALTHY`.

#### Health Aggregation

| Status | Condition |
|--------|-----------|
| `HEALTHY` | All components pass |
| `DEGRADED` | At least one non-critical component failing (e.g., one worker in CrashLoopBackOff) |
| `UNHEALTHY` | Critical component failing (scheduler or webserver down) |

#### Auto-Recovery

| Failure | Recovery Action | Escalation |
|---------|-----------------|------------|
| Failed worker pod | Delete pod (Kubernetes reschedules) | — |
| Failed scheduler pod | Restart pod | If unhealthy >5 min: full `helm upgrade` |
| Persistent failure | — | If unresolved >10 min: mark `ERROR`, send alert |

### 4.6 Secret Distributor

The Secret Distributor manages the lifecycle of secrets within each deployment's Kubernetes namespace. Secrets are never stored in plaintext — always fetched from HashiCorp Vault (or AWS Secrets Manager) and written to Kubernetes Secrets with encryption at rest.

#### Secrets Managed per Deployment

| Secret | Source | Rotation |
|--------|--------|----------|
| Database credentials | Vault database secrets engine (dynamic) | Every 24 hours |
| Celery broker credentials | Redis AUTH password | Every 30 days |
| Fernet encryption key | Vault KV engine | Never (rotating requires re-encrypting all connections) |
| Customer Airflow connections | Vault KV engine (deployment-scoped path) | On update |
| Container registry credentials | ECR pull secrets | Every 12 hours |

#### Sync Process

Runs every 30 seconds:
1. Compare secret versions in Vault vs. Kubernetes Secrets
2. For any newer Vault version: fetch and update Kubernetes Secret
3. Trigger rolling restart of affected components for credential rotation

### 4.7 Metrics Collector

The Metrics Collector aggregates and forwards metrics from the Runtime Plane to:
- Control Plane (billing and platform-wide dashboards)
- Local Prometheus (operational monitoring)

#### Airflow Metrics

Emitted by Airflow's built-in StatsD client → per-deployment StatsD exporter sidecar → Prometheus format → scraped every 15 seconds.

**Key metrics:**
- Task duration by DAG and task ID
- Task failure counts
- DAG parsing time
- Scheduler heartbeat age
- Executor queue depth and slot utilisation
- Pool slot usage
- DAG run duration by status

#### Infrastructure Metrics

Collected via Prometheus Kubernetes service discovery:
- Node Exporter (CPU, memory, disk I/O per node)
- kube-state-metrics (pod status, deployment replica counts, resource usage)
- CloudWatch exporter (RDS Enhanced Monitoring metrics)

#### Billing Metrics

Computed hourly from raw Prometheus data:
- Worker-hours (integral of worker pod count over time)
- Storage usage (EFS and S3 metrics)

Published to `billing.usage` NATS stream.

#### Forwarding to Control Plane

Local Prometheus configured with remote write to Control Plane's Thanos Receive endpoint, sending curated subset of metrics for cross-deployment visibility.

### 4.8 DAG Bundle Management

AirflowCloud supports two DAG delivery strategies:

#### Image-Based DAG Bundles (Recommended)

| Step | Description |
|------|-------------|
| 1. Build | Customer builds Docker image extending AirflowCloud base image with DAGs, plugins, dependencies |
| 2. Push | Image pushed to ECR |
| 3. Deploy | Deployment Controller updates Helm values with new image tag, runs `helm upgrade` |
| 4. Restart | All Airflow components restart with new image |

**Workflow:**
```
airflowcloud deploy
  → airflowcloud dev parse (validate DAGs locally)
  → docker build
  → docker push to ECR
  → POST /v1/deployments/{id}/deploy
  → helm upgrade
```

#### Git-Sync Based DAG Delivery

A `git-sync` sidecar runs alongside scheduler and worker pods:
- Continuously polls configured Git repository
- Syncs DAG files to shared volume
- No image build required for DAG changes
- Faster iteration (~60 seconds to visibility)

**Trade-off:** Does not version Python dependencies — dependency changes still require image rebuild.

**Configuration:**
- Repository URL
- Branch
- Polling interval (default: 60 seconds)
- Deploy key (SSH private key in Kubernetes Secret)

#### EFS vs. S3 for DAG Storage (Git-Sync Mode)

| Storage | Use Case |
|---------|----------|
| **EFS** | Shared filesystem mounted by all pods |
| **S3** | Preferred for large deployments with many DAG files (eliminates EFS throughput management) |

### 5.3 Airflow Components

#### Webserver (Airflow API Server in Airflow 3.x)

| Property | Value |
|----------|-------|
| **Purpose** | Airflow UI and REST API |
| **Replicas** | 1 (Starter/Pro), 2 active-active (Enterprise) |
| **State** | Stateless (reads from metadata database) |
| **Endpoint** | `https://{deployment-slug}.airflow.airflowcloud.io` |
| **Authentication** | Airflow built-in auth or customer IdP SSO |

#### Scheduler

| Property | Value |
|----------|-------|
| **Purpose** | Parse DAGs, evaluate schedules, create task instances, submit to executor |
| **Replicas** | 1 (standard), 2 HA mode (Enterprise, database-level leadership election) |
| **Priority** | Highest pod priority class (not evicted by resource pressure) |
| **Alert** | DAG parsing time >60 seconds triggers warning |

#### Workers (CeleryExecutor)

| Property | Value |
|----------|-------|
| **Purpose** | Execute task instances via Celery message queue |
| **Concurrency** | Default 4 tasks per worker |
| **Node type** | Spot Instances (cost reduction) |
| **PodDisruptionBudget** | Max 50% unavailable simultaneously |
| **Spot interruption** | 2-minute warning → graceful finish → exit |

#### Workers (KubernetesExecutor)

Each task runs as an individual short-lived Kubernetes Pod:
- Scheduler creates pod per task, monitors, cleans up on completion
- No persistent worker pods
- Better suited for heterogeneous resource requirements per task

#### Triggerer

| Property | Value |
|----------|-------|
| **Purpose** | Handle deferrable operators (tasks pausing while waiting for external events) |
| **Replicas** | 1 per deployment |
| **Resource usage** | Minimal (asynchronous event loops) |

### 5.4 Worker Autoscaling (KEDA)

Worker autoscaling is managed by KEDA (Kubernetes Event-Driven Autoscaling) rather than standard Kubernetes HPA.

> 📊 **Scaling Formula:**
> - **Scale-up:** +1 worker per 5 pending tasks
> - **Scale-down:** Remove worker only after queue empty for 5+ consecutive minutes
> - **Minimum:** 1 worker (always ready)
> - **Maximum:** Configurable per deployment (default: 10 medium, 20 large)

#### KEDA Configuration

For CeleryExecutor deployments, KEDA creates a `ScaledObject` resource for each deployment's worker Deployment:
- Scaler type: `aws-sqs-queue` (SQS-backed Celery) or custom Redis LIST length scaler
- Scale-to-zero for development deployments (no tasks for 30 min → workers scale to 0, webserver hibernated)

#### Node-Level Autoscaling (Karpenter)

Karpenter watches for pods in `Pending` state and automatically provisions new EC2 instances:
- Prefers Spot instances for worker node groups
- Falls back to On-Demand if Spot capacity unavailable
- Consolidates cluster by removing underutilised nodes

### 5.5 Namespace Isolation

Each deployment's Kubernetes namespace is strictly isolated through:

| Control | Purpose |
|---------|---------|
| **ResourceQuota** | Limits total CPU, memory, and pod count per namespace |
| **LimitRange** | Sets default and maximum CPU/memory requests and limits for every pod |
| **NetworkPolicy** | Default-deny posture; allows only ALB ingress, same-namespace traffic, and specific egress |
| **IRSA** | Dedicated IAM role per deployment, scoped to specific Secrets Manager paths, S3 prefixes |
| **Pod Security Standards** | `restricted` profile prevents privilege escalation, running as root, host path mounts |

#### NetworkPolicy Details

| Direction | Allowed |
|-----------|---------|
| **Ingress** | ALB (webserver traffic), same-namespace pods |
| **Egress** | Deployment's RDS, ElastiCache, EFS mount target, ECR registry, external internet |
| **Blocked** | Cross-namespace traffic between customer deployments |

### 5.6 Airflow Version Management and Upgrades

AirflowCloud maintains a curated set of supported Airflow versions based on the official Apache Airflow release schedule. Each version is available as a base Docker image with security patches applied.

#### Minor Version Upgrades (e.g., 2.8.0 → 2.8.1)

- Offered as optional updates in UI and CLI
- Run `helm upgrade` with new Airflow image tag
- No database migration required (backward compatible)

#### Major Version Upgrades (e.g., 2.x → 3.x)

| Step | Description |
|------|-------------|
| 1. Notification | Customer notified 30 days before end-of-support |
| 2. Validation | Upgrade wizard validates DAG compatibility (Airflow upgrade check tool) |
| 3. Snapshot | Pre-upgrade snapshot of Airflow metadata database |
| 4. Migration | Run `airflow db upgrade` as Job |
| 5. Upgrade | Helm chart upgrade with `--atomic` flag |
| 6. Monitoring | 10-minute stability check before marking complete |
| 7. Rollback | On failure: restore database snapshot, rollback to previous version |

### 5.7 Backup and Disaster Recovery

#### Airflow Metadata Database

| Tier | Backup | Retention | Cross-Region | RPO |
|------|--------|-----------|--------------|-----|
| Standard | Daily automated | 7-day PITR | No | 1 hour |
| Enterprise | Enhanced frequency | 7-day PITR | Yes (`us-west-2`) | 5 minutes |

#### DAG Storage

| Storage | Backup Method | Durability |
|---------|---------------|------------|
| EFS | AWS Backup (daily) | — |
| S3 | Inherent | 11 nines |

Cross-region replication available for Enterprise deployments.

#### Deployment Configuration

Desired state stored in platform database (Control Plane) with PITR. Even if entire Runtime Plane is lost, all deployment configurations can be replayed to recreate all deployments.

#### RTO Targets

| Failure Scenario | RTO | Method |
|------------------|-----|--------|
| EKS node failure, pod crash | <5 minutes | Health Monitor auto-recovery, pod rescheduling |
| Full Runtime Plane EKS cluster failure | 30–60 minutes | Deploy new cluster, recreate from Control Plane desired state |
| Enterprise with warm standby cluster | <10 minutes | Failover to standby |

| `grpc.airflowcloud.io` | A | Control Plane ALB (gRPC API) |

| `{deployment-slug}.airflow.airflowcloud.io` | CNAME | Runtime Plane ALB (per-deployment) |

Private DNS hosted zones are configured within each VPC to resolve internal service names (e.g., `platform-db.internal.airflowcloud.io` → RDS writer endpoint).

### 7.3 Security Groups

Security groups enforce least-privilege network access between components. Every resource has its own security group with rules defined by role, not by CIDR.

| Security Group | Inbound | Outbound |
|----------------|---------|----------|
| **Control Plane ALB** | 443/tcp from `0.0.0.0/0` | 8080, 9090 to Control Plane App SG |
| **Control Plane App (API Server)** | 8080, 9090 from ALB SG | 5432 to DB SG, 6379 to Redis SG, 443 to VPC endpoints, 4222 to NATS SG |
| **NATS** | 4222 from App SG and Data Plane Agent SG; 6222, 8222 from NATS SG | — |
| **Data Plane Agent** | None (outbound-only) | 443 to Control Plane ALB, 4222 to NATS, all ports to Runtime Plane SG |
| **Runtime Plane (Airflow pods)** | From Data Plane Agent SG, 8080 from Runtime ALB SG | 5432 to RDS SG, 6379 to Redis SG, 2049 to EFS SG, 443 to `0.0.0.0/0` |
| **Per-deployment RDS** | 5432 from Runtime Plane SG | None |
| **Per-deployment ElastiCache** | 6379 from Runtime Plane SG | None |

### 7.4 Airflow UI Access

Customers access their Airflow UI through: `https://{deployment-slug}.airflow.airflowcloud.io`

**Traffic flow:**
1. DNS resolves to Runtime Plane ALB
2. ALB routes to Kubernetes Ingress for specific deployment namespace
3. Ingress created by Deployment Controller using AWS Load Balancer Controller
4. TLS termination at ALB using wildcard certificate `*.airflow.airflowcloud.io`
5. Internal traffic (ALB → webserver pods) uses HTTP within VPC

## 8. Security Architecture

### 8.1 Security Layers

> 🛡️ **Defense in Depth:** Security is implemented in six independent layers, so that a failure in any one layer does not expose the system.

| Layer | Components | Purpose |
|-------|------------|---------|
| **1. Perimeter** | AWS WAF v2, CloudFront, Shield | Filter malicious traffic, DDoS protection |
| **2. AuthN/AuthZ** | JWT, API keys, mTLS, RBAC | Identity verification, permission enforcement |
| **3. Network** | VPC isolation, private subnets, SecurityGroups, NetworkPolicies | Least-privilege network access |
| **4. Data** | KMS encryption (rest), TLS 1.3 (transit), Vault (secrets) | Data protection at rest and in transit |
| **5. Runtime** | Pod security standards, IRSA, Secrets envelope encryption, GuardDuty | Container and workload security |
| **6. Supply Chain** | ECR scanning, SBOM, image signing, Dependabot | Secure software delivery |

### 8.2 Secrets Management

AirflowCloud uses HashiCorp Vault as the central secrets management system. Vault is deployed in HA mode (3-node cluster with Raft-based storage backend) on the Control Plane EKS cluster. Vault data is stored on encrypted EBS volumes.

#### Vault Secret Engines

| Engine | Purpose | Examples |
|--------|---------|----------|
| **KV (Key-Value)** | Static secrets with version history | Fernet keys, customer Airflow connections, internal API keys, platform config |
| **Database** | Dynamic, short-lived credentials | RDS credentials (24-hour TTL, auto-revoked on expiry) |
| **PKI** | X.509 certificates for mTLS | Data Plane Agent certificates (30-day validity, 7-day pre-expiry rotation) |

#### Vault Authentication Methods

| Method | Use Case |
|--------|----------|
| **Kubernetes auth** | EKS pods authenticate via service account JWT token, validated against EKS OIDC endpoint |
| **AWS IAM auth** | Terraform and AWS-native processes, validated via STS `GetCallerIdentity` |
| **AppRole** | CI/CD systems outside Kubernetes/AWS; secret IDs generated per pipeline run and immediately revoked |

### 8.3 mTLS for Control Plane — Data Plane Communication

All gRPC communication between the Control Plane and Data Plane Agents uses mutual TLS (mTLS). Both sides present certificates and validate each other.

| Side | Certificate | Validation |
|------|-------------|------------|
| Control Plane | Signed by AirflowCloud's public CA | Validated by Agent's trust store |
| Data Plane Agent | Signed by AirflowCloud's internal CA | Validated by Control Plane using Vault |

Even if an agent token is compromised, an attacker cannot impersonate a legitimate Data Plane Agent without also having a valid client certificate signed by AirflowCloud's internal CA.

## 9. Infrastructure as Code

### 9.1 Approach

All AirflowCloud infrastructure is defined as code using Terraform, organised with Terragrunt for DRY multi-environment configurations. Every infrastructure change goes through a pull request review and automated plan verification before being applied. Manual console changes are blocked via AWS Config rules and SCPs.

### 9.2 Repository Structure

| Component | Description |
|-----------|-------------|
| **Modules** | Reusable building blocks (EKS cluster, RDS instance, VPC, IAM roles) |
| **Environment configs** | prod, staging, dev — reference modules with environment-specific variables |
| **Terragrunt** | Handles DRY configuration: shared remote state, common variable injection |

**Terraform state:** Stored in S3 in `airflowcloud-prod` account, encrypted with KMS, with DynamoDB table for state locking. Partitioned by environment and component (e.g., `prod/us-east-1/eks-control/terraform.tfstate`).

### 9.3 Per-Deployment Infrastructure

The infrastructure for each customer deployment (RDS, ElastiCache, EFS) is provisioned by the Deployment Controller using Terraform programmatically.

| Property | Value |
|----------|-------|
| **Workspace** | One Terraform workspace per deployment (identified by deployment ID) |
| **Isolation** | Workspaces are isolated and cannot interfere with each other |
| **State** | Stored in S3 under `deployments/{deployment-id}/` prefix |
| **Execution** | Terraform binary spawned as Kubernetes Job |
| **Outputs** | RDS endpoint, Redis endpoint, EFS mount target → used for Helm values |

### 9.4 GitOps for Platform Applications

AirflowCloud platform applications (API Server, NATS, Prometheus, Grafana, Loki, Vault) are deployed to EKS using Flux CD in a GitOps pattern:

- Flux watches infrastructure repository's `apps/` directory
- Automatically applies Helm releases and Kubernetes manifests when changes merged to `main`
- Application deployments reviewed via pull request
- Automatically tested in staging before production
- Fully auditable through git history

**Note:** Airflow deployment Helm releases are managed by the Deployment Controller (not Flux). The Deployment Controller applies Helm releases imperatively in response to commands from the Control Plane.

## 11. Extensibility

### 11.1 Extension Architecture Principle

The three-plane architecture (Control Plane, Data Plane, Runtime Plane) is the stable contract. Extensions to new deployment modes work by changing where the Data Plane and Runtime Plane run, not by changing the Control Plane.

The gRPC contract between the Control Plane and Data Plane Agent is versioned and backward-compatible — an older Agent version can still communicate with a newer Control Plane within the same major version.

### 11.2 Hybrid Mode Extension

In Hybrid mode, the Data Plane Agent Helm chart is installed by the customer in their own EKS cluster. The agent connects outbound to the Control Plane's gRPC endpoint. From the Control Plane's perspective, a Hybrid Data Plane is identical to a Managed one.

**Additional work required:**
- Agent registration flow in Control Plane UI (Section 6.6)
- AWS PrivateLink VPC Endpoint Service for private connectivity
- Documentation and Terraform module for customer IAM setup
- Secret backend pluggability in Config Syncer (customer's Vault or AWS Secrets Manager)

No changes to Control Plane services required — purely Data Plane and network configuration change.

### 11.3 Remote Execution Extension

Remote Execution is a future mode where the Airflow Scheduler and Metadata DB remain on AirflowCloud's infrastructure but task execution happens in the customer's environment. Enabled by Airflow 3's Task Execution API.

**Architecture:**
- Lightweight Worker Agent (not full Data Plane Agent) installed in customer environment
- Heartbeats to AirflowCloud API Server advertising available task slots
- Receives task assignments via Task Execution API
- Executes tasks locally (with access to customer data, secrets, network)
- Reports task results back via HTTP (task data never leaves customer environment)

**Work required:** Task Execution API proxy, Worker Agent Helm chart, DAG Processor Agent, PrivateLink support.

### 11.4 Private Cloud Extension

Private Cloud mode packages the entire AirflowCloud Control Plane for installation in a customer's own AWS account (or GCP or Azure).

**Components:**
- `airflowcloud-installer` CLI for bootstrapping Control Plane
- Offline license key (signed JWT with feature flags and expiry)
- Same Terraform modules, Helm charts, and application code as public cloud
- Air-gap support via `generate-image-bundle` command (tarball of all container images)

### 11.5 Multi-Cloud Extension

The Data Plane Agent and all Data Plane components are written without AWS SDK dependencies. AWS SDK usage is confined to the cloud provider abstraction layer (Terraform modules and Secret Distributor backends).

**To support GCP or Azure as Data Plane host:**
- Add new Terraform modules
- Add new secrets backend implementation
- No modification to core Data Plane or Control Plane logic

The Control Plane itself remains on AWS in all current extension paths. Multi-cloud Control Plane (running simultaneously on AWS, GCP, Azure for regional data residency) is a future Phase 6 capability.

### 11.6 Extension Points Summary

| Extension | What Changes | What Stays the Same |
|---|---|---|
| **Hybrid mode** | Data Plane runs in customer VPC | Control Plane, gRPC contract, all core services |
| **Remote Execution** | Thin Worker Agent in customer env | Control Plane, full Data Plane for scheduling |
| **Private Cloud** | Full platform in customer account | Same code, same Helm charts, same gRPC contract |
| **Multi-cloud Data Plane** | New Terraform modules + secrets backend | Agent code, Control Plane, gRPC contract |
| **New executor types (Dask, Ray)** | New executor Helm values + resource profiles | Data Plane reconciliation logic |
| **Custom deployment sizes** | New resource profiles in Deployment Service | Everything else |
| **New Airflow versions** | New base images in ECR | Everything else |

### 12.4 Glossary

| Term | Definition |
|---|---|
| **Control Plane** | Central management layer hosting the API, identity, billing, audit, and deployment intent (desired state). Always operated by AirflowCloud. |
| **Data Plane** | Infrastructure orchestration layer that provisions resources and manages Airflow deployment lifecycle. Runs in AirflowCloud VPC (managed) or customer VPC (hybrid/private). |
| **Runtime Plane** | The actual Airflow workload environment — webserver, scheduler, workers, triggerer, and their supporting databases and storage. Runs within the Data Plane's Kubernetes cluster. |
| **Data Plane Agent** | The component running in the Data Plane that maintains the outbound gRPC connection to the Control Plane, receives commands, and dispatches them to the Deployment Controller. |
| **Deployment Controller** | Component in the Data Plane responsible for orchestrating Terraform (infrastructure) and Helm (Kubernetes resources) for each customer Airflow deployment. |
| **Deployment** | A single Airflow environment — all Airflow components (webserver, scheduler, workers, triggerer) plus their supporting infrastructure (RDS, ElastiCache, EFS), running in a dedicated Kubernetes namespace. |
| **Workspace** | A logical grouping of deployments within an organisation, typically corresponding to a team or environment. |
| **Organisation** | The top-level tenant — corresponds to a company or team. All billing, user management, and quota enforcement happens at the organisation level. |
| **Desired State** | The JSON document stored in the Control Plane platform database describing what a deployment should look like. The source of truth for what the Data Plane should create and maintain. |
| **Actual State** | The current observed state of a deployment, reported by the Data Plane Agent from inspecting Kubernetes resources. Periodically synced back to the Control Plane. |
| **IRSA** | IAM Roles for Service Accounts — an AWS mechanism that binds a Kubernetes service account to an IAM role, allowing pods to call AWS APIs using the role's permissions without static credentials. |
| **KEDA** | Kubernetes Event-Driven Autoscaling — a Kubernetes component that scales workloads based on external event sources such as queue depth, rather than only CPU/memory metrics. |
| **Karpenter** | An AWS-native Kubernetes node autoscaler that provisions EC2 instances dynamically in response to pending pods, replacing the older Cluster Autoscaler. |
| **mTLS** | Mutual TLS — a TLS configuration where both the client and the server present certificates and verify each other's identity, rather than only the server presenting a certificate. |
