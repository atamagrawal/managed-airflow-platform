## 1. Why this structure exists

Flowdeck is organised around three kinds of thing — **Organisation, Workspace, Deployment** — plus a fourth thing called a **Project** that works differently from the others:

| Level | Problem it solves |
|-------|-------------------|
| **Organisation** | One billing account, one SSO config, one set of company-wide limits for your entire company. |
| **Workspace** | Different teams or environments should not see each other's pipelines. A workspace is the **hard line** between teams. |
| **Deployment** | One live Airflow environment. It is pre-configured with the Airflow version, Python packages, providers, connections, resource sizing, and all other runtime dependencies. |
| **Project** | A codebase unit — a repository of DAG files. A project is not a container; it is a **deploy source** that can be **linked** to one or more deployments (staging, prod, shared, etc.). |

The structure answers: **who can touch what, who pays for it, what rules apply, and which code runs where.**

---

## 2. The four levels

Flowdeck is built around **four levels**: **Organisation → Workspace → Project** and **Deployment**. Organisation, workspace, and deployment nest in a straight line. **Project** and **deployment** are not nested inside each other — they are **linked** (many-to-many) inside a workspace, as described in [section 3](#3-how-projects-and-deployments-relate).

### Rules of thumb

| Level | Rules of thumb |
|-------|----------------|
| **Organisation** | One organisation has **many** workspaces. Billing, SSO, and company-wide limits apply at this level. |
| **Workspace** | One workspace has **many** projects and **many** deployments. A workspace is the **hard boundary** between teams — people in one workspace do not see another's work unless invited. |
| **Project** | One project is **one codebase** (usually one git repo). It can be **linked** to **one or many** deployments (e.g. staging and production). |
| **Deployment** | One deployment is **one live Airflow environment**. It belongs to **exactly one** workspace. It can be **linked** to **one or many** projects (shared deployment). It is **never** shared across two workspaces. |

### Example (one org, two workspaces)

```
ORGANISATION: Acme Corp
│
├── WORKSPACE: Data engineering          ← team boundary
│     │
│     ├── PROJECT: Payments platform   ← one codebase…
│     │        └── linked to ──► DEPLOYMENT: payments-staging
│     │        └── linked to ──► DEPLOYMENT: payments-prod
│     │        └── linked to ──► DEPLOYMENT: shared-prod   ← also shares this one
│     │              (one project → several deployments)
│     │
│     ├── PROJECT: Fraud detection     ← another codebase…
│     │        └── linked to ──► DEPLOYMENT: fraud-dev
│     │        └── linked to ──► DEPLOYMENT: fraud-prod
│     │        └── linked to ──► DEPLOYMENT: shared-prod   ← same shared env as Payments
│     │
│     └── DEPLOYMENT: shared-prod      ← one Airflow; both projects above deploy here
│
└── WORKSPACE: Analytics                 ← another team, fully separate
       └── PROJECT: Reporting
                └── linked to ──► DEPLOYMENT: reporting-prod
```

This example shows **both** patterns: Payments is **one project → many deployments** (its own staging and prod, plus the shared production pool). `shared-prod` is **many projects → one deployment** (Payments and Fraud both linked there — a shared deployment).

## 3. How projects and deployments relate

Projects and deployments are **peers inside a workspace** — they are not nested. The link between them is **many-to-many**:

- **One project → many deployments** — the same codebase (same git repo) can deploy DAG bundles to staging *and* production, each a different Airflow environment.
- **One deployment → many projects** — several projects can deploy DAG bundles into the same Airflow environment (a **shared** deployment).

**Typical pattern (one codebase, staging + prod):**

```
WORKSPACE: Data engineering
│
├── DEPLOYMENT: payments-staging  ◄──┐
├── DEPLOYMENT: payments-prod     ◄──┴── PROJECT: Payments platform
│                                        (same repo, two targets)
├── DEPLOYMENT: fraud-dev         ◄── PROJECT: Fraud detection
└── DEPLOYMENT: fraud-prod        ◄── PROJECT: Fraud detection
```

Each deployment is still its own environment (packages, sizing, connections). The Payments project pushes the same DAG bundle twice — once per target — usually from a CI pipeline that picks the environment from a branch or manual approval gate.

**Shared (many projects → one deployment):**

```
WORKSPACE: Data engineering
│
├── DEPLOYMENT: shared-prod  ◄── PROJECT: Payments platform
│                            ◄── PROJECT: Fraud detection
│                            (both teams deploy DAGs here)
│
└── DEPLOYMENT: reporting    ◄── PROJECT: Analytics
```

One deployment runs DAGs from both Payments and Fraud. The deployment is pre-configured upfront with all packages and dependencies both teams need.

**Rules of thumb:**

| Question | Answer |
|----------|--------|
| How many deployments can one project be linked to? | **Many** — e.g. dev, staging, and production each as a separate deployment. |
| How many projects can be linked to one deployment? | **One** (that deployment is dedicated to a single codebase) or **many** (shared deployment). |
| Does a project contain deployments? | No. A project deploys *to* linked deployments. |
| Can a deployment be shared across workspaces? | Never. Every link stays inside one workspace. |

### What a project contains

A project lives in a **git repository** (or a path in a monorepo). It contains:

- DAG files (`dags/`)
- An optional dependency manifest (`project.yaml`) — see below
- CI/CD pipeline config that pushes DAG bundles to each linked deployment (staging, prod, etc.)

A project does **not** contain: Python packages, `requirements.txt`, Airflow version, resource sizing, connections, or variables. All of those belong to the deployment.

### The project.yaml manifest (V1: informational)

A project can include a `project.yaml` file in its repository that declares what each linked deployment must provide (informational in V1):

```yaml
# project.yaml — lives in the project repo, not deployed to Airflow
# Optional labels help humans match CI jobs to environments
target_deployments:
  - deployment: payments-staging
    label: staging
  - deployment: payments-prod
    label: production

requires:
  airflow_version: ">=2.9"
  python_packages:
    - pandas>=1.5
    - my-internal-lib>=2.0
  providers:
    - apache-airflow-providers-postgres>=5.0
  connections:
    - payments_db
    - payments_s3
```

In **V1 this manifest is informational only** — Flowdeck does not automatically enforce it before accepting a DAG deploy. Its purpose in V1 is to make the dependency contract visible in the repository and give the deployment owner a clear checklist when configuring a shared environment. V2 will use it to gate DAG deploys: if a required package is not installed on the target deployment, the deploy is rejected at CI time with a clear error.

### V1 deploy mechanism: DAG-only

In V1, the only supported deploy mechanism is **DAG-only deploy**:

- A project's CI pipeline builds a DAG bundle containing only DAG files.
- Each deploy call names **which linked deployment** receives the bundle (e.g. staging or production).
- The deployment's Airflow environment (packages, providers, connections, variables) is **not changed** by a project deploy.
- Environment changes require a separate action by the **deployment owner** (workspace admin or whoever is designated to manage that deployment's settings).

This is a deliberate constraint for V1. It keeps the environment stable — one project's deploy cannot break another project's DAGs by changing a shared package version.

## 4. Deciding how to structure your team

This is the most important decision a PM makes in Flowdeck. Use these questions:

### Should I create a new workspace or a new project?

| Situation | Use a new… |
|-----------|------------|
| A different team who should **never** see our pipelines | **Workspace** — hard boundary, independent membership, own alert channels |
| Same team, different codebase / programme deploying to its own environment | **Project** linked to one or more dedicated deployments |
| Two teams want to share one Airflow environment | Two **projects** linked to the same shared deployment |
| Just another environment (dev/staging/prod) for the same codebase | Another **deployment**, same project |

**Rule:** Workspace = team or org boundary. Project = one deployable codebase unit.

### Can one workspace hold all our work?

Yes. Small organisations often use a single workspace with several projects — each project may link to one deployment or to several (for example staging and production). The workspace settings (who gets alerted, which Airflow versions are allowed, whether production requires automation) apply to everyone inside it.

### Can a deployment be shared across projects?

Yes — and this is a first-class concept in Flowdeck. When two or more projects are **linked** to the same deployment, that deployment is **shared**. In V1 this works through DAG-only deploys: each project ships only its DAG files; the deployment environment (packages, connections, sizing) is managed separately by the deployment owner.

A deployment can never be shared across two different **workspaces**. All projects targeting the same deployment must be in the same workspace.

See [section 9](#9-common-scenarios) for when sharing makes sense and what the trade-offs are.

## 5. Who can do what — roles

Every person or automation account has a **role** at one or more levels. Higher-level roles automatically carry equivalent access to everything below them in the same branch.

### Roles at a glance

| Role | What this person can do |
|------|-------------------------|
| **Org owner** | Billing, SSO setup, all workspace admin rights everywhere, delete org. |
| **Org admin** | Create/remove workspaces, manage members and limits — but not billing. Full workspace admin everywhere. |
| **Org member** | Has an identity in your org but cannot touch anything until given a workspace/project role. |
| **Workspace admin** | Manages this workspace: members, production guardrails, defaults, alerts. Effectively a project lead in every project here. |
| **Workspace developer** | Creates and configures environments and projects. Cannot change workspace-wide settings or membership. |
| **Workspace viewer** | Read-only across this workspace. |
| **Project lead** | Manages this project: which deployments are linked, signing keys, project members, deploy history. Cannot touch other projects or workspace settings. |
| **Project developer** | Deploys DAG bundles to deployments **already linked** to this project. Cannot add or remove deployment links (that is a lead / workspace admin action). |
| **Project viewer** | Read-only for this project. |
| **Deployment deployer** | Ships new code **to this one environment**. The CI/CD pipeline role — cannot change how the environment is configured. |
| **Deployment operator** | Triggers runs, clears failed tasks, adjusts runtime settings. For on-call and analysts who run pipelines but don't ship code. |
| **Deployment viewer** | Sees run history and logs only. |

### Key rule: access only flows downward

An org admin is effectively an admin in every workspace. A workspace admin is effectively a project lead in every project. But a project developer cannot see other projects — and cannot see the workspace settings at all.

**Roles are additive.** A person who is "workspace viewer" overall but "project lead" on one project has read-only access everywhere else and full control in that project.

**There are no deny rules.** To keep someone out of a project, simply do not invite them to it.

### Who should get each role? (quick guide)

| Person | Suggested role |
|--------|---------------|
| Platform or infrastructure admin | Org admin |
| Team lead responsible for a workspace | Workspace admin |
| Data engineer building pipelines | Workspace or project developer |
| Analyst who runs reports but does not write code | Deployment operator |
| Security reviewer or exec stakeholder | Org member + workspace viewer |
| External contractor on one programme | Project developer on that project only |
| CI/CD pipeline | Deployment deployer (scoped key) |

## 6. Controlling who deploys to production

A workspace can be set to **require all code changes to go through an automated pipeline** rather than allowing engineers to push manually.

When this is on for a workspace:

- **CI/CD automation** (service accounts with a deployer key) can still ship code.
- **Human users** — even workspace admins — cannot push pipeline code directly. They can still trigger runs, read logs, and clear tasks.
- **Platform admins** can still change environment-level settings (like Airflow version or sizing), but that is separate from shipping DAG code.

**This is the main governance switch for production.** Turning it on means production only changes when your review process approves it. Turning it off means anyone with developer access can push directly.

## 7. Onboarding and offboarding people

### Inviting someone

- For **full team access**: invite them to the workspace with the right role.
- For **one programme only**: invite them directly to the project — they get the **Workspace Accessor** level automatically (minimal workspace visibility) and see only the deployments you assign.
- For **one environment only**: give them a deployment-level role directly.

### SSO and directory groups

If your company uses Okta, Microsoft Entra, or another identity provider, IT can map directory groups to Flowdeck roles. When a user joins or leaves a group, their access updates automatically on next sign-in — no manual Flowdeck steps required.

Example mapping:

| Company directory group | Flowdeck role |
|-------------------------|---------------|
| data-eng-all | Workspace developer — "Data engineering" |
| data-eng-leads | Workspace admin — "Data engineering" |
| payments-team | Project developer — "Payments platform" |
| audit-access | Workspace viewer — all production workspaces |

### When someone leaves

If directory sync is on: HR removes them from the IdP → their Flowdeck sessions and keys are disabled automatically, typically within minutes.

Without directory sync: an org admin removes them manually. The result is the same — sessions and keys stop working immediately. **Running environments are not affected** and keep running under automation.

## 8. Automation and CI/CD pipelines

Pipelines, scripts, and monitoring tools use **automation accounts** with **keys** instead of passwords. You assign them a role just like a person.

**How a project CI/CD pipeline works in V1:**

1. The pipeline builds a DAG bundle from the project's `dags/` directory.
2. It pushes the bundle to **one linked deployment** at a time (e.g. staging on merge to `develop`, production on tag) using a **Deployment deployer** key scoped to that deployment.
3. The deployment's Airflow environment does not change — only the DAG files are updated.
4. If the deployment is shared, each project's pipeline pushes only its own bundle to that deployment. Other projects' DAGs are not touched.

**Best practice: scope the key tightly.**

| Pipeline | Role | Key scope |
|----------|------|-----------|
| Project CI pushing DAGs to one deployment | Deployment deployer | That one deployment |
| Workspace admin updating deployment environment | Workspace admin | That workspace |
| Read-only health check | Deployment viewer | That deployment |

A key scoped to a deployment cannot accidentally affect other deployments or change the environment configuration — even if the automation account's broader role would permit it.

Treat keys like passwords: store them in a secrets manager, rotate when team membership changes, revoke promptly when no longer needed.

## 9. Common scenarios

### Giving a contractor access to one programme

The project lead invites the contractor **directly to the project** with a developer role. The contractor:

- Can deploy DAG bundles to every deployment **linked** to that project.
- **Cannot** see other projects in the workspace.
- **Cannot** change workspace-wide settings or membership.

If the workspace requires pipeline-only code pushes, the contractor must use a CI/CD key — not direct pushes.

### Two projects sharing one deployment

Sometimes two project teams within the same workspace want to run DAGs on a single Airflow environment — typically to reduce infrastructure cost or share compute for closely related pipelines.

In Flowdeck both projects are **linked** to the same deployment. Neither project "owns" the deployment in a parent-child sense — ownership is a role: whoever holds the **deployment owner** or **workspace admin** role controls the environment configuration.

**How it works in V1 (DAG-only deploy):**

- Each project's CI pipeline pushes its own DAG bundle to the shared deployment using a **Deployment deployer** key.
- A deployer key can push DAGs but **cannot** change the deployment environment — no package installs, no connection changes, no resource resizing.
- The deployment owner (workspace admin or a designated person) is the only one who can change the environment.
- Each project's pipeline is independent — Project A deploying does not affect Project B's DAGs.

**DAG ID uniqueness is enforced by the platform:**

Airflow requires all DAG IDs to be unique within a deployment. If two projects both define a DAG called `etl_pipeline`, Airflow silently discards one — whichever file the scheduler happened to parse last. No error, no alert; one team's production DAG just disappears.

To prevent this, Flowdeck maintains a **DAG ID registry** for every shared deployment — a record of which project owns which DAG IDs. When a project pushes a DAG bundle, the platform parses all DAG IDs in the bundle and checks them against this registry:

- If a DAG ID is **not yet registered** → allowed, registered to this project.
- If a DAG ID is **already registered to this project** → allowed (it is an update).
- If a DAG ID is **registered to a different project** → **deploy rejected** with a clear error: `DAG ID "etl_pipeline" is already owned by project "fraud-detection"`.

This means teams do not need to coordinate naming conventions — the platform catches conflicts at deploy time, before anything reaches the running Airflow environment.

**The shared deployment must be pre-configured with everything upfront:**

Because no project team can modify the environment after the fact, the deployment must already have every package, provider, connection, and variable any project's DAGs will need before those DAGs are deployed. This is a key responsibility of the deployment owner:

| Dependency type | Who sets it | When |
|-----------------|-------------|------|
| Python packages and Airflow providers | Deployment owner | Before any project team ships DAGs |
| Connections and variables | Deployment owner | Before DAGs reference them |
| Resource sizing (worker count, memory) | Deployment owner | Must account for the combined load of all projects |
| Airflow version | Deployment owner | Must be compatible with all projects' DAG code |

If a project team needs a new dependency, they raise it with the deployment owner. They cannot add it themselves. Use `project.yaml` to make these requirements explicit and visible in the project repo.

**Residual trade-offs:**

| Risk | What it means in practice |
|------|--------------------------|
| **Shared outage** | A runaway DAG from one project slows the scheduler and workers for all projects sharing that deployment |
| **Shared version lock** | All projects run on the same Airflow version — a version upgrade requires all teams to agree |
| **Dependency bottleneck** | If a project needs a new Python package, they must ask the deployment owner — they cannot add it themselves |
| **Unclear on-call** | Agree upfront who is paged when the shared deployment goes down |
| **Compliance** | Regulators may require sensitive data pipelines to be isolated even within the same workspace |

**When sharing makes sense:** related pipelines with compatible dependencies, shared infrastructure cost is the priority, and all teams commit to DAG-only deploys with one clear environment owner.

**When each project should have its own deployment:** different release cadences, compliance or data isolation requirements, or any team that needs independent control over environment-level settings.

### Promoting an engineer to workspace admin

The current workspace admin changes their role. The promotion takes effect immediately. The change is recorded in the audit trail with who made it and when.

### Splitting a team into two workspaces

When two sub-teams need genuinely independent governance — separate alert channels, different production rules, or compliance reasons — they should be in separate workspaces. Once a deployment exists under a workspace, it cannot be "moved" to another workspace directly; the practical path is to create new deployments in the new workspace and retire the old ones.

### Adding a new product line inside an existing team

Add a new **project** inside the existing workspace and **link** it to a new dedicated deployment, to several deployments (staging + prod), or to an existing shared deployment. The team's workspace-wide membership, defaults, and alert routing all apply automatically. Invite anyone who needs narrower access directly at project level.

## 10. How Flowdeck compares to Astronomer Astro

[Astronomer Astro](https://www.astronomer.io/docs/astro/overview) is a widely used managed Airflow product. If your team is familiar with it, here is how the models map.

### Structure

Astro uses **Organization → Workspace → Deployment** — three levels. An Astro "project" (created with `astro dev init`) is a **local development artefact** — a folder with DAG files, a `requirements.txt`, a `Dockerfile`, and CI/CD config. It is not a platform entity; there is no "Project" resource in the Astro API or UI.

Flowdeck's **Project** is similar in spirit to an Astro project: it is a codebase unit that deploys to deployments. The difference is that Flowdeck tracks it as a platform entity — giving it its own member list, deploy history, and **formal links** to one or many deployments (see [section 2](#2-the-four-levels)). That makes the relationship explicit and auditable rather than something that only exists in CI configuration.

Both products agree: **a deployment is never shared across two workspaces.**

### Access

| Topic | Astro | Flowdeck |
|-------|------|----------|
| Narrow access within a workspace | **Workspace Accessor** role + deployment-level roles | Project-level roles or deployment-level roles |
| Teams / groups | Teams apply a shared workspace role to a group of users | Directory groups map to roles at any level via SSO |
| CI/CD enforcement | Set as a workspace default; can be overridden per deployment | Set at workspace level; applies to all deployments in that workspace |
| Custom roles | Custom deployment roles defined at org level | Fixed named roles per scope in V1 |
| DAG-only deploy | Supported via OCI image CI/CD | V1 only supported mechanism for projects |

### The main difference

In Astro, an Astro project is an infrastructure + code unit — the project's `Dockerfile` and `requirements.txt` define the environment, and the full image is built and deployed together. Environment and DAGs travel as one.

In Flowdeck V1, environment and DAGs are deliberately separated: the deployment owns the environment, the project owns the DAGs. This makes shared deployments practical (multiple projects share one environment without rebuilding it) at the cost of requiring the deployment to be pre-configured before any project deploys to it.

## Closing note

The **four levels** are not bureaucracy — they are decisions. **Organisation** decides billing and company-wide rules. **Workspace** decides which teams are independent of each other. **Project** is one codebase and **which deployments it is linked to**. **Deployment** is where Airflow actually runs and where the environment is configured.

Get the workspace boundaries right first ([section 2](#2-the-four-levels)). Then wire projects to deployments — one codebase to many environments, or many codebases to one shared environment, or both. The deploy mechanism in V1 is DAG-only: the environment is always the deployment's responsibility, never the project's.
