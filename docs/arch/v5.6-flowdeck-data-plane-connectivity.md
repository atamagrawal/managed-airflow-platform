## 1. Why this document exists

The baseline Flowdeck architecture describes the Data Plane Agent as **always initiating** connectivity to the Control Plane — an **outbound-only** pattern for command delivery and status (`v5.1`, `v5-flowdeck-enterprise-architecture.md`). That design is strongest when the Data Plane might live in a **customer-controlled network** (hybrid), where inbound holes from Flowdeck are unacceptable.

If **all** Data Planes run in **Flowdeck-operated** AWS accounts and VPCs, outbound-only is **not** the only viable design. This document spells out **what changes** if you relax it: network topology, security boundaries, component responsibilities, and operational trade-offs.

Nothing here replaces the shared-data-plane or runtime content in **v5.3** — it only addresses **how bytes move** between Control Plane and Data Plane.

---

## 2. What “outbound-only” actually constrains

“Outbound-only” is not one switch. It usually means all of the following together:

| Constraint | Meaning |
|------------|---------|
| **TCP direction for the control channel** | The first SYN on the long-lived control session comes from the Data Plane side (Agent → Control Plane endpoint). |
| **No listener on the customer / runtime side for administration** | Flowdeck does not require `ingress :443` from `0.0.0.0/0` into the runtime cluster for platform administration. |
| **Command delivery semantics** | Commands are **pulled** by the Agent (over a stream backed by NATS on the Control Plane side) rather than **pushed** by the Control Plane into an arbitrary URL inside the runtime VPC. |

Telemetry (metrics to billing, audit) may still be described as “outbound” from the Data Plane even when commands use a different pattern — see **§6**.

---

## 3. Baseline model — Control Plane never opens inbound to customer networks

**When hybrid / customer VPC Data Planes are in scope:** keep the baseline:

- Agent initiates **mTLS gRPC** to `grpc.flowdeck.io` (or regional equivalent).
- **NATS JetStream** lives with the Control Plane; the Agent’s connection is outbound through the customer’s NAT/firewall (allow-list destination only).
- No inbound agent listener required on the customer side.

**Properties:**

- Strong enterprise narrative (“no inbound from vendor into our VPC”).
- Single registration pattern for every Data Plane type.
- Control Plane does not need routable addresses into each regional runtime VPC.

**Costs:**

- Long-lived streams or polling from many regions into one (or few) Control Plane clusters.
- All command traffic funnels through public or shared endpoints unless you add **PrivateLink for egress** from customer VPC (still Agent-initiated).

This remains the **default recommendation** whenever the Data Plane network is not fully owned by Flowdeck.

## 4. Alternative A — Private ingress within Flowdeck’s estate

**Scope:** Data Plane EKS clusters live **only** in Flowdeck AWS accounts. No customer-hosted Agent.

**Idea:** The Control Plane (or a dedicated **command relay** service in the control account) opens **outbound** connections **inside AWS private networking** to a **regional endpoint** that fronts each Data Plane — for example:

- **NLB** or **internal ALB** in the runtime VPC, reachable via **VPC peering**, **Transit Gateway**, or **AWS PrivateLink** interface endpoint from the Control Plane VPC.
- Agent still runs in-cluster, but now exposes a **private** gRPC or HTTP service (`flowdeck-agent.dp-region.internal:8443`) that accepts **connections initiated from the Control Plane side**.

**Command flow variant:**

1. Deployment Service publishes command to NATS as today.
2. **Regional relay** (running in control or edge account) consumes NATS and **pushes** gRPC `ApplyCommand` to the Agent’s private listener **or** opens a **bidirectional stream** where the TCP SYN originates from the relay (still **not** from the public internet into runtime — only from trusted private CIDRs).

**Trade-offs:**

| Benefit | Cost |
|---------|------|
| Lower reliance on long-lived **reverse** streams from runtime → central CP | Control Plane (or relay) must maintain **routing tables**, **security groups**, and **health checks** per region / per pool member |
| Faster perceived command latency for urgent applies | **More moving parts**: relay HA, regional failover, duplicate NATS consumers |
| Easier to reason about “push deploy now” if CP initiates | Agent must implement **authenticated inbound** — mTLS from relay, never anonymous LB |

**Security:** Runtime NLB/ALB must **not** be public. Bind to **private subnets**; SGs allow **only** Control Plane / relay security groups. Identity is **mTLS** or **SigV4** + IAM auth on private API Gateway.

## 5. Alternative B — Queue-based command delivery

**Idea:** Replace (or supplement) the streaming channel with **per–Data Plane queues** in AWS (**SQS**) or similar:

1. Control Plane writes **command envelope** to `sqs://deployments-commands-{dp_id}` (with KMS, DLQ, visibility timeout).
2. Agent **polls** its queue from inside the runtime VPC (outbound HTTPS to SQS via VPC endpoint — still “Agent initiates TCP” at the transport level).
3. Agent applies command, deletes message, acks back via status path.

**Semantics:**

- Still **pull-based** from the Agent’s perspective — similar security story to outbound-only streams.
- **No** persistent gRPC stream required; good for flaky networks (at-least-once + idempotent reconcile).

**Trade-offs:**

| Benefit | Cost |
|---------|------|
| Operational familiarity (SQS ops, DLQ replay) | **Ordering** across deployments needs careful design (FIFO per deployment key, or strict single consumer per `deployment_id`) |
| Natural backoff when Agent down | **Latency floor** = poll interval unless hybrid long-poll |
| Clear separation of command store from stream infra | **Cost** at very high message volume; must tune batching |

NATS can remain for **real-time status WebSockets** to the UI while commands use SQS — **split path** (see **§6**).

## 6. Alternative C — Split paths for commands vs telemetry

You do **not** have to use one mechanism for everything.

| Path | Typical baseline | Relaxed option |
|------|------------------|----------------|
| **Commands** (desired state apply) | NATS + Agent stream (outbound) | Private push (**§4**) or SQS (**§5**) |
| **Status / phase** | Agent → CP on same stream | Same, or **direct Postgres write** via narrow relay (higher risk — usually avoid) |
| **Heartbeats / capacity_score** | Agent → CP | **CloudWatch** custom metrics + scraper in CP, or regional **API Gateway** ingest |
| **Billing metrics** | Agent → Billing path | **Kinesis Firehose** / **S3** batch files polled by Billing Service |

**Rule:** Anything that lets the **runtime VPC accept arbitrary inbound from the internet** should remain off the table. Split paths are fine **inside** Flowdeck-owned AWS with tight SGs and IAM.

---

## 7. Security and blast radius

Relaxing outbound-only **does not** mean opening the Data Plane cluster to the world.

| Control | Apply to every alternative |
|---------|----------------------------|
| **No public listener** for Agent admin API | NLB internal-only or PrivateLink only |
| **mTLS or IAM** between CP components and runtime |
| **Least privilege** — relay role can only call Agent endpoint for registered `dp_id` |
| **Audit** — every pushed command logged with `desired_state_version` and `correlation_id` |
| **Blast radius** — compromise of CP relay must not equal shell on customer namespaces (RBAC in-cluster for Agent is still minimal) |

If hybrid returns later, you can run **regionally**: hybrid keeps pure outbound-only Agent; Flowdeck-hosted regions use private push — **two connectivity profiles** behind one **logical** Deployment Service API.

---

## 8. Decision guide

| Your situation | Prefer |
|----------------|--------|
| Customer-hosted Data Plane (hybrid) possible | **Baseline outbound-only** stream (§3) |
| Flowdeck-only AWS; want simplest ops story | **Baseline** is still valid — one protocol everywhere |
| Flowdeck-only AWS; want lower central fan-in / faster regional apply | **§4 Private ingress** from CP relay to regional Agent listener |
| Intermittent networks; queue replay matters more than sub-second apply | **§5 Queue-based** delivery |
| Heavy billing / metrics volume thrashing the stream | **§6 Split paths** — keep commands on stream or SQS; move metrics to pipelines |

## 9. Implementation notes for NATS and the Agent

**If you keep NATS on the Control Plane** (baseline):

- Agent remains a **NATS client** (or consumes via gRPC gateway that bridges NATS). No change.

**If Control Plane pushes to Agent (§4):**

- Option **A:** NATS stays in CP; **relay** subscribes to NATS and forwards to Agent — Agent may **stop** subscribing to NATS directly (simpler Agent; relay is critical tier).
- Option **B:** Deploy **regional NATS** next to runtime (adds ops burden — usually avoid unless scale demands).

**If you use SQS (§5):**

- NATS can be narrowed to **status fan-out for UI only**, not command durability — command durability lives in SQS + Aurora `desired_state_version`.

**Idempotency:** Regardless of path, reconciliation remains **idempotent** on `desired_state_version`. Push vs pull does not change the Data Plane reconciliation loop described in **v5.1** / **v5.3**.

## 10. End-to-end flows

The steps below are **logical** — they omit retries, idempotency keys, and exact API shapes. They all share the same **start** (user or CI changes desired state) and the same **end** (Deployment Controller reconciles in-cluster). Only the **transport** between Control Plane and Data Plane Agent differs.

### 10.1 Baseline — outbound gRPC stream (Agent initiates)

| Step | Where | What happens |
|------|--------|--------------|
| 1 | User / CI | `POST` to Control Plane API (e.g. deploy DAG, scale workers). |
| 2 | API Gateway → Deployment Service | AuthN/Z, validate workspace policy and quota, begin Aurora transaction. |
| 3 | Deployment Service | Write new `deployments.desired_state` JSONB, increment `desired_state_version`, commit **transactional outbox** so a `CREATE` / `UPDATE` command is published to **NATS** `deployments.commands.{dp_id}.{deployment_id}` in the same commit. |
| 4 | Agent (already running) | **Previously** opened long-lived **bidirectional gRPC** to `grpc.flowdeck.io`; API Gateway’s stream writer reads NATS and **forwards** the command on the open stream **toward** the Agent. |
| 5 | Agent | Receives `Command`, verifies incoming `desired_state_version` is newer than the locally applied version, **persists** accept to BoltDB, sends `CommandAck` on the stream. |
| 6 | Control Plane NATS consumer | Receives ack, **+ACK** the JetStream message (command is durably accepted by Agent). |
| 7 | Agent | Dispatches to **Deployment Controller** (in-cluster gRPC / channel). |
| 8 | Deployment Controller | Terraform / Helm / health as in **v5.1**; publishes **status** events (back on the same stream or to `deployments.status` subject). |
| 9 | Control Plane | Persists observed state, **WebSocket** / `deployments.status` consumers update UI. |
| 10 | Heartbeat (every ~15s) | Agent → Control Plane on same or parallel path; **Data Plane Registry** updates `last_heartbeat_at` and `capacity_score`. |

**Note:** The **first TCP SYN** for the control session is **Agent → Control Plane**. Commands still flow **Control Plane → Agent** *on the established stream* (server push over client-opened connection).

### 10.2 Alternative A — private push (regional relay → Agent listener)

| Step | Where | What happens |
|------|--------|--------------|
| 1–3 | Same as baseline | Desired state written; command published to **NATS** in Control Plane (unchanged). |
| 4 | **Command relay** (e.g. in control VPC or regional edge) | **Subscribes** to NATS (or an internal bus). For each message, resolves `dp_id` → **private runtime endpoint** (internal NLB DNS for that Data Plane’s Agent service). |
| 5 | Relay | Opens **mTLS gRPC** **outbound** from relay → **runtime VPC** (first SYN is **relay → Agent listener**). Path is **private** (peering / TGW / interface endpoint) — not the public internet. |
| 6 | Agent | Accepts `ApplyCommand`, authenticates relay cert, persists to BoltDB, returns ack **on the same RPC** (or over a short callback channel). |
| 7 | Relay | Confirms to NATS consumer → **+ACK** JetStream (same durability rule as today: do not ack NATS until Agent has persisted). |
| 8 | Agent | Dispatch to Deployment Controller **as in baseline** (step 7 onward same). |
| 9 | Status / billing | **Options:** (a) Agent still **pushes** status to Control Plane on a **separate** outbound gRPC to `grpc.flowdeck.io`, or (b) relay also opens **duplex** and streams status back. UI path unchanged. |

**Flow shape:** *NATS in CP → relay (private) → Agent in runtime VPC → controllers → K8s.* The Control Plane does not need a public route **into** the cluster; it needs **private** routability from relay to internal NLB.

### 10.3 Alternative B — SQS (durable queue, Agent long-poll or poll)

| Step | Where | What happens |
|------|--------|--------------|
| 1–2 | Same as baseline | API accepts change. |
| 3 | Deployment Service | Instead of (or in addition to) NATS, writes a **command envelope** to **SQS** `deployments-commands-{dp_id}` (KMS-encrypted, **FIFO** per `deployment_id` if strict ordering is required). Aurora transaction still records `desired_state_version`. |
| 4 | Agent | **Polls** SQS from the runtime VPC over **HTTPS** to the **SQS VPC endpoint** (outbound from private subnets). Receives message, **visibility timeout** holds it while processing. |
| 5 | Agent | Verifies version, dispatches to Deployment Controller. On **successful** handoff to durable local state, **DeleteMessage**; on failure, message becomes visible again after timeout (**at-least-once**). |
| 6 | Idempotency | If the same `desired_state_version` is redelivered, Agent **dedupes** (same as stream model) and deletes the duplicate message. |
| 7 | Status | Agent POSTs / streams status to Control Plane (could use **same** outbound gRPC as baseline, or **§6** split). |
| 8 | No persistent stream | If Agent is down, messages **accumulate** in SQS; no lost JetStream delivery while disconnected (DLQ for poison messages). |

**Flow shape:** *Aurora + SQS in CP account (or shared) → Agent pull via VPC endpoint → apply → delete message.* Control Plane never opens a connection into the runtime; the **queue** is the buffer.

### 10.4 Alternative C — split path (example: SQS commands + stream status + Firehose billing)

| Path | Flow |
|------|------|
| **Commands** | As **§10.3** — SQS only for `ApplyDeploymentCommand`. |
| **Status / UI** | Agent maintains **lightweight outbound gRPC** (or periodic **POST** batch) only for `deployments.status` so the WebSocket hub and deployment page stay real-time. |
| **Heartbeats** | Agent writes **CloudWatch** `PutMetricData` in the runtime account; a **Lambda** or **Fargate** poller in the Control Plane account consumes metrics and updates **Data Plane Registry** (no heartbeat on gRPC). |
| **Billing aggregates** | Agent sidecar or Metrics Collector writes **hourly Parquet** to **S3**; Billing Service **EMR** / **Athena** / batch job ingests — zero billing traffic on the command channel. |

**End-to-end for one user action:** steps 1–2 unchanged → command lands in **SQS** → Agent applies → status on **stream** → UI updates; billing path **decoupled** on its own schedule.

---

## Closing

**Outbound-only** is the right default when the Data Plane network is not yours alone. When **every** Data Plane runs inside **Flowdeck-operated** AWS, you may introduce **private, authenticated inbound** to the Agent or **queue-based pull** without contradicting the three-plane model — you are only changing **transport**, not **who owns desired state** (Control Plane) or **who executes it** (Data Plane).

For tiering, pooling, and Agent state, continue to use **v5.3**. For Control Plane services and APIs, use **v5.5**.
