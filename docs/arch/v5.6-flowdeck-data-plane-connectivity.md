## 1. The idea in one minute

The **Control Plane** decides *what should happen* (desired state). The **Data Plane** does the real work (Kubernetes, Terraform, health checks). They have to exchange messages: **commands** (“please apply this”) and **updates** (“here is status / capacity / billing-related data”).

The usual Flowdeck story is: **the Data Plane Agent is the one that phones home**. It opens the connection out to Flowdeck. Flowdeck does **not** require an open hole **into** the customer’s network just so we can push commands. That pattern is called **outbound-only**, and it is ideal when a Data Plane might run **inside a customer VPC** one day.

If **every** Data Plane always runs in **Flowdeck’s own AWS accounts**, you have more choices. You could instead **call into** the runtime network over **private** links inside AWS, or put commands in a **queue** that the Agent reads. This document explains those choices in readable language and shows the **flow** for each.

---

## 2. What “outbound-only” means in plain English

It is **not** a single checkbox. Usually it means:

1. **Who opens the phone line first** — The Agent starts the long-lived connection **to** Flowdeck’s API endpoint. Flowdeck does not rely on “you must open port 443 from the internet into my cluster.”

2. **No public admin port on the runtime cluster** — Customers do not need “allow anyone on the internet to hit my Kubernetes API for Flowdeck.”

3. **How commands travel on that phone line** — Once the line is open, Flowdeck can send work **down** that same connection (like talking on a call you dialled). So “outbound-only” still allows commands to flow **from Flowdeck to the Agent** — it just means **your side placed the call**.

Billing and metrics can still travel **out** from the Data Plane on separate paths; that does not break the idea above.

## 3. Option 1 — Baseline (best when the Data Plane might live in a customer network)

**How it feels:** The Agent **keeps a line open** to Flowdeck (encrypted). Work is queued inside Flowdeck (using NATS JetStream behind the scenes). When something changes, Flowdeck sends the next command **over that existing line**.

**Why people like it**

- Sales and security can honestly say: “Flowdeck does not need an inbound firewall rule from our servers into your VPC.”
- One pattern works for **managed cloud** and **future hybrid**: same Agent behaviour.

**Downsides**

- Many regions all maintain connections into the central Control Plane — you need to scale that entry point well.
- If you want **private egress** from the customer VPC to Flowdeck, you add PrivateLink on the **customer** side — the Agent still **starts** the connection.

**When to use:** Default whenever the runtime network is **not** 100% under Flowdeck’s control, or you want one story everywhere.

---

## 4. Option 2 — Flowdeck calls the Agent over private AWS networking

**When it applies:** Every Data Plane runs in **Flowdeck-operated** VPCs. There is **no** customer-hosted cluster.

**How it feels:** Instead of only “the Agent calls Flowdeck,” you add a **private door** on the inside of your AWS network. A small **relay service** in Flowdeck’s control side reads the same work queue as today, then **places a call** over **private** networking (VPC peering, Transit Gateway, or PrivateLink) to an **internal** load balancer that sits in front of the Agent. The internet never sees that door — only trusted Flowdeck networks.

**Good**

- Commands can feel snappier because Flowdeck can **push** toward the region without waiting on a particular streaming layout.
- Less dependence on one giant “everyone dial home” pattern.

**Trade-offs**

- You must **operate** the relay: routing, health checks, high availability per region.
- The Agent must **trust inbound** calls — but only from Flowdeck’s relay, using strong TLS identity (not a public anonymous endpoint).

**Security in one line:** The load balancer stays **private**; security groups only allow Flowdeck’s relay — never “open to the world.”

---

## 5. Option 3 — Commands sit in a queue; the Agent picks them up

**How it feels:** Think **mailbox** instead of **phone call**. Flowdeck drops a letter (the command) into an AWS **SQS** queue dedicated to that Data Plane. The Agent **checks the mailbox** on a schedule (or long poll). When it sees work, it applies it and **throws the letter away** so it is not processed twice.

**Good**

- If the Agent is offline, messages **wait** in the queue instead of being lost.
- Teams already know how to run SQS (dead-letter queues, replay, alarms).

**Trade-offs**

- There is a **minimum delay** between “command posted” and “Agent polls” unless you tune aggressive polling.
- If many deployments share patterns, you must design **ordering** (often a FIFO queue **per deployment**).

The Agent still uses **outbound HTTPS** to reach SQS through a **VPC endpoint** — so you are not opening random inbound internet to the cluster.

---

## 6. Option 4 — Use different channels for different jobs

You do not have to pick **one** pipe for everything.

| Kind of data | Simple baseline | Another way |
|--------------|-----------------|-------------|
| **Commands** (apply desired state) | Long-lived connection + internal queue | Private push (**§4**) or SQS (**§5**) |
| **Status** (what phase, healthy or not) | Same connection, or a second lightweight outbound stream | Short batches to an HTTP API |
| **“Are you alive?” / capacity** | Heartbeat on the stream | Ship **metrics to CloudWatch**; Control Plane reads them indirectly |
| **Billing usage** | Sent with other telemetry | **Hourly files to S3** or a pipe like Firehose — keeps billing traffic off the hot path |

Rule of thumb: **never** expose a **public** “admin API” on the customer cluster. Split paths are fine **inside** Flowdeck’s AWS with proper security groups and IAM.

---

## 7. Safety rules (no matter which option)

- **Private beats public** for anything that can change infrastructure.
- **Prove who is calling** — TLS client certs, IAM, or both between Flowdeck components.
- **Log every command** with version numbers so you can audit “who asked for what.”
- **Repeating the same command should be safe** — the Data Plane already reconciles by version; that stays true.

---

## 8. Quick “which option should we use?”

| Situation | Sensible default |
|-----------|------------------|
| You might sell **hybrid** (Data Plane in customer VPC) later | **Option 1** baseline |
| Flowdeck-only AWS and you want the **simplest** single story | **Option 1** is still fine |
| Flowdeck-only AWS and you want **regional push** with less central fan-in | **Option 2** |
| Networks are messy; you care about **queue replay** and durability | **Option 3** |
| Billing and metrics **flood** the same pipe as commands | **Option 4** — split traffic |

## 9. Short note on NATS and keeping work safe when messages repeat

**NATS** (in the baseline) is the durable queue behind the scenes that makes sure commands are not dropped before the Agent accepts them.

If you add **SQS**, NATS might shrink to “real-time UI updates only,” while **SQS + the database** hold the truth for commands.

If the **same** command shows up twice (network glitch, redelivery), the Agent already knows how to say **“already applied”** using the monotonic **desired state version**. That idea does not change when you switch transport.

## 10. What actually happens, step by step

Every path below starts the same way: **someone changes intent** (UI or CI). It ends the same way: **Deployment Controller** inside the cluster does Terraform / Helm / checks until reality matches desired state. Only the **middle** — how the command reaches the Agent — changes.

---

### 10.1 Baseline flow (Agent opened the connection first)

1. User or pipeline saves a change through the Flowdeck API.
2. The Deployment Service writes the new desired state to the database and bumps a **version number**.
3. In the **same database transaction**, Flowdeck records that a **command** must be sent (so you never update state without also intending to deliver work).
4. A background worker reads that intent and puts the command on an internal queue (**NATS**).
5. The Agent — which **already** opened an encrypted long-lived connection **to** Flowdeck — receives the command **over that connection**.
6. The Agent saves “I have accepted this version” locally, tells Flowdeck “OK,” and only then does Flowdeck consider the message fully delivered.
7. The Agent hands off to the **Deployment Controller**, which does the real provisioning work inside the cluster.
8. Status flows back to Flowdeck; the UI updates.
9. On a timer, the Agent sends **heartbeat / capacity** so Flowdeck knows this Data Plane is healthy.

**Remember:** The Agent **dialled Flowdeck first**. After that, commands travel **from Flowdeck to the Agent** on the open line — like caller ID showing who started the call, but both sides can speak.

---

### 10.2 Option 2 flow (private push from a relay)

Steps **1–3** are the same: API → database → command queued inside Flowdeck.

4. A **relay service** (running in Flowdeck’s trusted network) reads the queue.
5. The relay looks up **which runtime cluster** should receive this command.
6. The relay opens a **private** connection **into** the runtime VPC — **not from the public internet** — to an internal endpoint in front of the Agent.
7. The Agent verifies the relay is allowed, saves acceptance locally, answers OK.
8. The relay tells the queue “done” only after the Agent has persisted — same safety rule as today.
9. From here on, **same as baseline**: Deployment Controller runs, status goes back to Flowdeck (either another outbound Agent connection or the same private channel — your choice).

---

### 10.3 Option 3 flow (SQS mailbox)

1–2. Same: API validates and saves desired state.

3. Flowdeck writes the command into an **SQS queue** for that Data Plane (often encrypted). The database still holds the version number.

4. The Agent **polls** SQS using **outbound** HTTPS through an AWS **VPC endpoint** (traffic stays inside Amazon’s network from the cluster’s perspective).

5. The Agent locks a message while working (**visibility timeout**). If it crashes mid-way, the message **reappears** for another try.

6. When the Agent is sure the work is handed to the Controller safely, it **deletes** the message from the queue.

7. Status returns to Flowdeck on whatever path you prefer (stream, HTTP batch, etc.).

**If the Agent is down for an hour:** messages **pile up** in the queue instead of disappearing — that is the point.

---

### 10.4 Option 4 flow (split paths — example)

Imagine you combine **§10.3** with lighter paths for everything else:

- **Commands:** SQS (**§10.3**).
- **Live status for the UI:** a small **outbound** stream or frequent REST batches — only status, not heavy billing.
- **Heartbeats:** publish simple numbers to **CloudWatch**; Flowdeck reads them without loading the main command pipe.
- **Billing:** batch files to **S3** once an hour; the Billing service imports them separately.

**One user click** still ends as: database updated → command delivered → cluster reconciled → UI shows progress; only the **pipes** differ.

---

## Closing

**Outbound-only** (the Agent phones Flowdeck first) remains the best default when you care about **customer networks** and a single global story.

If **only Flowdeck** ever runs the Data Plane, you may add **private inbound** to the Agent or **queue-based** delivery — you are changing **how mail gets delivered**, not **who owns the truth** (still the Control Plane) or **who executes** (still the Data Plane).

For cluster layout and capacity, keep using **v5.3**. For APIs and services in the Control Plane, use **v5.5**.
