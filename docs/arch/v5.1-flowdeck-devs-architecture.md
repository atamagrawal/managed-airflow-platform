## 1. System Overview

Flowdeck separates what the platform *decides* from what it *executes*. The Control Plane is the decision layer — it owns identity, policy, desired state, and audit. The Data Plane is the execution layer — it turns desired state into real cloud resources and running Kubernetes workloads. The Runtime Plane is the workload itself — the actual Airflow environment that customers interact with.

The three planes are connected by a single outbound-only, mTLS-authenticated gRPC stream from the Data Plane to the Control Plane. The Control Plane never opens an inbound connection into a Data Plane. This one constraint is what allows the same architecture to support a managed cloud deployment, a customer-hosted hybrid agent, and an air-gapped private install without changing any protocol or code.

## 2. Control Plane — Components and Internal Wiring

The Control Plane runs as a set of Kubernetes Deployments on an EKS cluster in `flowdeck-prod-control`. Every component is stateless at the process level — all durable state lives in Aurora PostgreSQL, ElastiCache Redis, or NATS JetStream. Horizontal scaling is safe for all services because no local state survives a pod restart.

### API Gateway

The API Gateway is the single entry point to all Control Plane application logic. It is a stateless Go service (three replicas minimum) that does four things: terminates the protocol, authenticates the caller, enforces rate limits, and proxies to the correct downstream service.

REST requests carry either a bearer JWT (from a logged-in user) or an API key (from CLI/CI). The gateway verifies the JWT signature against the Identity Service's public key, checks the `jti` against the Redis blocklist to catch revoked tokens, and then extracts `org_id`, `user_id`, and role assignments from the claims before forwarding. For API keys, the gateway hashes the presented key with SHA-256 and looks it up in Aurora to resolve the same identity context.

gRPC requests from a Data Plane Agent go to a separate ALB target group on the same hostname (`grpc.flowdeck.io`). The gateway validates both the mTLS client certificate (CN and SAN must match the registered Data Plane ID) and a JWT carried in the `:authorization` gRPC metadata header. Both checks must pass; the certificate alone proves network identity, the JWT proves logical identity.

WebSocket connections use a short-lived 5-minute token issued by the Identity Service specifically for the session. This keeps the WebSocket token scope narrow — it cannot be replayed against the REST API — and avoids the need to re-validate a long-lived credential on every message.

Rate limiting is layered: AWS WAF enforces per-source-IP limits at the edge (prevents DDoS before it hits application code); the API Gateway enforces per-user and per-organisation limits using atomic INCR operations in Redis with a sliding window; a circuit breaker on every downstream call opens at 50% error rate over a 10-second window and returns `503 Retry-After` until the downstream recovers.

### Identity Service

The Identity Service is the only component allowed to mint or validate credentials. Every other service's authentication path goes through it. It owns four data types: user records, service account records, the JWT signing key pair, and the API key hash table.

Human authentication: email + bcrypt password (cost 12) with TOTP second factor. Enterprise SSO via SAML 2.0 or OIDC; group memberships from the IdP are mapped to Flowdeck workspace roles by a configurable attribute mapping table. Successful authentication issues a JWT (RS256, 1-hour expiry, claims: `sub`, `org_id`, `roles`, `scopes`, `jti`) plus an opaque refresh token (256-bit random, SHA-256 hashed at rest, rotated on every use, 30-day window).

Token revocation works through a Redis blocklist keyed by `jti`. Entries are written on explicit logout, password change, or security incident, and expire automatically at the token's natural TTL. The blocklist is checked on every request; it is the mechanism that makes the 1-hour token expiry a ceiling, not a minimum.

The RBAC decision cache sits in Redis: role assignments are cached per `(user_id, org_id)` with a 5-minute TTL. Role changes and user deactivations explicitly invalidate the cache entry via a publish to a Redis pub/sub channel that all API Gateway instances subscribe to.

### Organisation, Workspace, and Deployment Services

These three services share an Aurora schema and treat the `org → workspace → deployment` hierarchy as a single aggregate. They use a serialisable transaction plus a short-lived Redis distributed lock (key: `quota_lock:{org_id}`, TTL: 5 seconds, acquired with `SET NX`) around every resource-creation operation to prevent quota races.

The Deployment Service is the orchestrator of deployment lifecycle. It holds the desired state for each deployment as a JSONB document in the `deployments` table alongside normalised columns for the fields that carry foreign-key constraints (workspace ID, Airflow version enum, executor type). When desired state changes, the service atomically increments `desired_state_version`, writes the new JSONB, writes an audit entry, and publishes a command to NATS JetStream — all in one Aurora transaction. There is no scenario in which the desired state is updated but no command is published, because the NATS publish is inside the transaction boundary using NATS' transactional outbox pattern.

Each deployment moves through a state machine. The states and the transitions that cause them are:

```
PENDING
  └─► PROVISIONING   (command accepted by Data Plane Agent)
        └─► CONFIGURING  (Phase 1 complete, Phase 2 running)
              └─► STARTING     (Phase 2 complete, Phase 3 running)
                    └─► RUNNING      (Phase 3 complete, healthy)
                          ├─► UPDATING   (new desired state received while RUNNING)
                          │     └─► RUNNING
                          ├─► SCALING    (quota/resource change only)
                          │     └─► RUNNING
                          ├─► STOPPING   (stop command issued)
                          │     └─► STOPPED
                          └─► ERROR      (unrecoverable failure in any phase)

STOPPED
  └─► STARTING   (restart command)
  └─► TERMINATING

TERMINATING
  └─► TERMINATED
```

Every transition is written to the `deployment_events` table with: `deployment_id`, `from_state`, `to_state`, `actor_id`, `reason_code`, `correlation_id`, `timestamp`.

### Data Plane Registry

The registry is a table in Aurora that stores the identity and routing address of every Data Plane the platform manages. Each row holds: `dp_id` (UUID), `type` (`MANAGED | HYBRID | PRIVATE | REMOTE`), `region`, `grpc_endpoint`, `health` (`HEALTHY | DEGRADED | OFFLINE`), `capacity_score`, `last_heartbeat_at`, and cloud account metadata.

When the Deployment Service emits a command, it calls the registry's routing function: "give me the healthiest `MANAGED` Data Plane in `us-east-1` with capacity for one more deployment". The registry returns a `dp_id`. If the target later becomes `OFFLINE`, the command stays durably in the NATS stream until the Data Plane reconnects — the registry does not redirect it to a different Data Plane, because desired state is per-deployment and a deployment cannot be transparently migrated between Data Planes without a coordinated handoff.

Heartbeats arrive from each Data Plane Agent every 15 seconds on the `agents.heartbeat` NATS stream (last-value-per-agent semantics). A background process in the registry evaluates them: if `now - last_heartbeat_at > 60s`, mark `DEGRADED`; if `> 300s`, mark `OFFLINE`. The transition to `OFFLINE` triggers a notification to affected customers via the Notification Service.

### NATS JetStream

NATS JetStream is the internal event bus for the Control Plane and the command delivery channel to Data Planes. It runs as a 3-node Raft StatefulSet on the Control Plane EKS cluster with EBS gp3 volumes.

Three streams are critical:

`deployments.commands` is a work-queue stream (each message is delivered to exactly one consumer). Subject structure: `deployments.commands.{dp_id}.{deployment_id}`. Retention: until acknowledged. A command is removed from the stream only after the Data Plane Agent sends a `+ACK`. If the agent crashes before acknowledging, the message is redelivered on reconnect with an exponential backoff up to 30 minutes. Payload schema: `{command_type, deployment_id, desired_state_version, desired_state_json, idempotency_key}`.

`deployments.status` is a limits-based stream, keeping the last 100 status events per deployment. Subject: `deployments.status.{deployment_id}`. The API Gateway's WebSocket hub subscribes to this stream and pushes updates to the browser over the WebSocket connection. Payload schema: `{deployment_id, state, phase, reason_code, timestamp, correlation_id}`.

`agents.heartbeat` keeps only the most recent event per agent (last-value consumer). Subject: `agents.heartbeat.{dp_id}`. The registry process subscribes and updates `last_heartbeat_at` in Aurora.

## 3. Control Plane to Data Plane Communication

The communication between the Control Plane and Data Plane is the most architecturally important channel in the system. It is a single bidirectional gRPC stream, always initiated by the Data Plane, always outbound from the Data Plane's network. The Control Plane never opens a connection into the Data Plane.

```
  CONTROL PLANE                          DATA PLANE AGENT
       │                                        │
       │         ◄──── TCP SYN (outbound) ──────│  Agent initiates
       │         ◄──── TLS ClientHello ─────────│
       │         ──── TLS ServerHello ──────────►│
       │         ◄──── mTLS ClientCert ──────────│  Agent presents cert
       │         ──── TLS Finished ─────────────►│
       │                                        │
       │  (gRPC stream established)             │
       │                                        │
       │  ◄─── RegisterDataPlane(dp_id, token) ─│  First message on startup
       │  ──── RegisterDataPlane(ack) ──────────►│
       │                                        │
       │  ──── CommandStream.Send(command) ─────►│  Control Plane writes commands
       │  ◄──── CommandStream.Recv(ack) ─────────│  Agent acks after local persist
       │                                        │
       │  ◄──── StatusStream.Send(status) ───────│  Agent sends status updates
       │  ──── StatusStream.Recv(ack) ──────────►│
       │                                        │
       │  ◄──── Heartbeat(dp_id, metrics) ───────│  Every 15 seconds
       │  ──── Heartbeat(ack) ───────────────────►│
       │                                        │
```

The stream is implemented as a bidirectional streaming gRPC call (`/flowdeck.v1.DataPlaneService/Connect`). Both sides write onto it independently using goroutines; there is no request/response pairing within the stream. The Control Plane's writer goroutine reads from the `deployments.commands` NATS subject for the Data Plane and writes each command onto the stream. When the Data Plane Agent has durably accepted the command (written it to its local BoltDB state file), it writes an acknowledgement back on the stream, and only then does the Control Plane's NATS consumer call `+ACK` on the NATS message.

This two-hop ack — agent acknowledges to Control Plane, Control Plane acks to NATS — is what makes the system at-least-once safe. The NATS message is retained until the agent has physically persisted the work. If the stream drops before the agent acks, the NATS message is redelivered to the agent on reconnect.

If the gRPC stream breaks (network loss, agent restart, TCP timeout), the Control Plane marks the stream as disconnected and begins a backoff reconnect window. Commands published to NATS during this window accumulate. When the agent reconnects, it re-registers, the stream is re-established, and the Control Plane resumes writing commands from the NATS stream. The agent deduplicates by comparing each incoming `desired_state_version` against its local applied-version map and silently acks any command with a version it has already processed.

## 4. Data Plane — Agent and Internal Controllers

The Data Plane runs in the `flowdeck-system` namespace on the runtime EKS cluster. It is composed of the Agent and four internal controllers. Only the Agent holds Control Plane credentials. The controllers communicate with the Agent through in-cluster channels (gRPC over the pod network, not over the internet) and never have direct access to NATS or to the mTLS certificate.

### Agent Startup Sequence

When the Agent pod starts, it performs a fixed sequence before becoming operational:

1. Reads its mTLS certificate and private key from a Kubernetes Secret mounted at `/var/run/secrets/flowdeck/tls`.
2. Reads its bootstrap token from a Kubernetes Secret mounted at `/var/run/secrets/flowdeck/token`.
3. Opens a gRPC connection to `grpc.flowdeck.io:443` using the mTLS certificate as the client certificate.
4. Sends a `RegisterDataPlane` message: `{dp_id, type, region, version, cloud_account_id, bootstrap_token}`.
5. The Control Plane validates the bootstrap token (single-use, 24-hour TTL), creates or updates the registry row, and returns a `RegisterDataPlane` response with a short-lived JWT (24-hour expiry, scoped to gRPC only).
6. The Agent stores the JWT in BoltDB, discards the bootstrap token, and opens the bidirectional `Connect` stream using both the mTLS certificate and the JWT.
7. The Agent reads its local applied-version map from BoltDB. For each deployment that has a locally-persisted `in_progress` state, it sends a `StatusUpdate` to the Control Plane to resync, then resumes reconciliation from the last persisted phase checkpoint.

### Credential Rotation

Credentials rotate continuously. Seven days before the mTLS certificate expires, the Agent calls `RotateCertificate` on the Control Plane (using the existing certificate to authenticate), receives a new certificate, writes it to BoltDB, and on the next stream reconnect presents the new certificate. The old certificate is marked for revocation on the Control Plane side with a 1-hour grace window to allow in-flight reconnects. JWTs are rotated every 23 hours using a `RefreshToken` RPC; the overlap window is 1 hour so no in-flight request is invalidated.

## 5. Deployment Provisioning — The Three-Phase Flow

When the Deployment Controller receives a `CREATE` or `UPDATE` command from the Agent, it runs a three-phase reconciliation. Each phase has its own retry policy and its own failure handling.

```
  Agent receives command
        │
        ▼
  Check desired_state_version > applied_version?
        │ Yes                 │ No → silent ack, discard
        ▼
  Persist command to BoltDB (phase=INFRA, status=IN_PROGRESS)
  Ack to Control Plane (command accepted, not complete)
        │
  ┌─────▼─────────────────────────────────────────────────────┐
  │  PHASE 1 — INFRASTRUCTURE                                  │
  │                                                           │
  │  Terraform apply (state in S3, lock in DynamoDB)          │
  │  Resources created:                                       │
  │   - RDS PostgreSQL 15 cluster (pg_hba.conf via SSM)       │
  │   - ElastiCache Redis 7 (if CeleryExecutor)               │
  │   - S3 bucket (DAG storage, server-side KMS encryption)   │
  │   - IAM Role for IRSA (trust policy scoped to namespace   │
  │     service account, permission boundary to secret paths) │
  │   - Security groups wiring RDS/Redis/EKS                  │
  │                                                           │
  │  Retry: exponential backoff, max 30 min budget            │
  │  Failure types: AWS API throttle, quota exhaustion,       │
  │    transient errors → retryable                           │
  │  Terminal failure: IAM policy violation, VPC config error │
  │    → emit INFRA_FAILED with reason code, stop             │
  └─────┬─────────────────────────────────────────────────────┘
        │ success
        ▼
  Publish status: PROVISIONING → CONFIGURING
        │
  ┌─────▼─────────────────────────────────────────────────────┐
  │  PHASE 2 — RUNTIME                                         │
  │                                                           │
  │  Render Helm values from desired state JSON:              │
  │   - airflow.image: from OCI bundle digest                 │
  │   - executor: KubernetesExecutor or CeleryExecutor        │
  │   - scheduler.resources: from resource profile           │
  │   - workers.resources: from resource profile             │
  │   - env vars: injected from desired_state.env_refs        │
  │   - connections: references to Kubernetes Secrets         │
  │     (populated by Secret Distributor before Helm apply)   │
  │                                                           │
  │  helm upgrade --install airflow-{deployment_id}           │
  │    flowdeck/airflow --namespace {deployment_id}           │
  │    --values rendered-values.yaml --atomic --timeout 10m   │
  │                                                           │
  │  Retry: none (Helm failures are deterministic config      │
  │    errors — retry without changing config is pointless)   │
  │  Failure types: image not found, values schema error,     │
  │    Helm timeout → emit RUNTIME_FAILED, surface to user    │
  └─────┬─────────────────────────────────────────────────────┘
        │ success
        ▼
  Publish status: CONFIGURING → STARTING
        │
  ┌─────▼─────────────────────────────────────────────────────┐
  │  PHASE 3 — VALIDATION                                      │
  │                                                           │
  │  Poll until all checks pass (timeout: tier-dependent):    │
  │   1. Webserver pod: GET /health → 200                     │
  │   2. Scheduler heartbeat: query RDS                       │
  │      SELECT * FROM scheduler_job WHERE heartbeat_date     │
  │      > now() - interval '30 seconds'                      │
  │   3. Triggerer pod: liveness probe passes                 │
  │   4. Worker pods (if Celery): min 1 ready pod             │
  │   5. DAG parse: GET /api/v1/dags → no parse errors        │
  │                                                           │
  │  On success: register tenant hostname in Route 53,        │
  │    publish endpoint info to Control Plane,                │
  │    transition to RUNNING, write applied version to BoltDB │
  │                                                           │
  │  Failure: workload defects (DAG parse error, bad dep)     │
  │  Policy: leave on old state, emit VALIDATION_FAILED +     │
  │    parser output, do NOT auto-rollback (customer must fix)│
  └─────────────────────────────────────────────────────────────┘
```

For `UPDATE` commands, the controller computes a diff between the new desired state and the current Helm release values. If only the DAG image digest changed, only a `helm upgrade` is run (Phase 2 + Phase 3). If infrastructure parameters changed (RDS instance type, for example), all three phases run. Phases that produce no diff are skipped entirely; Terraform's `plan` output with no changes is the gate.

## 6. Health Monitoring and Automated Recovery

The Health Monitor is an independent goroutine pool inside the Data Plane. It does not wait for commands from the Agent; it continuously polls every deployment it manages. It publishes health events back to the Control Plane through the Agent's status channel.

```
  ┌───────────────────────────────────────────────────────┐
  │              HEALTH MONITOR (continuous loop)          │
  │                                                       │
  │  Every 30 seconds, per deployment:                    │
  │                                                       │
  │  ┌─────────────────────────────────────────────────┐  │
  │  │  CHECK 1: Webserver                             │  │
  │  │  GET https://{internal_svc}/health              │  │
  │  │  Expect: 200 within 5s                          │  │
  │  └─────────────────────────────────────────────────┘  │
  │  ┌─────────────────────────────────────────────────┐  │
  │  │  CHECK 2: Scheduler heartbeat                   │  │
  │  │  SQL: SELECT heartbeat_date FROM scheduler_job  │  │
  │  │       WHERE heartbeat_date > now() - '30s'      │  │
  │  │  Via RDS Proxy (no direct DB connection per pod)│  │
  │  └─────────────────────────────────────────────────┘  │
  │  ┌─────────────────────────────────────────────────┐  │
  │  │  CHECK 3: Worker pool (CeleryExecutor only)     │  │
  │  │  kubectl get pods -n {ns} -l role=worker        │  │
  │  │  Expect: ≥1 pod in Running + Ready state        │  │
  │  └─────────────────────────────────────────────────┘  │
  │  ┌─────────────────────────────────────────────────┐  │
  │  │  CHECK 4: Metadata DB reachability              │  │
  │  │  TCP connect to RDS endpoint on port 5432       │  │
  │  │  Expect: SYN-ACK within 3s                      │  │
  │  └─────────────────────────────────────────────────┘  │
  │  ┌─────────────────────────────────────────────────┐  │
  │  │  CHECK 5: Broker (CeleryExecutor only)          │  │
  │  │  PING to ElastiCache Redis endpoint             │  │
  │  │  Expect: PONG within 2s                         │  │
  │  └─────────────────────────────────────────────────┘  │
  │                                                       │
  │  Component results roll up: all HEALTHY → HEALTHY     │
  │  Any DEGRADED → deployment DEGRADED                   │
  │  Webserver or Scheduler UNHEALTHY → UNHEALTHY         │
  │                                                       │
  └──────────────────────┬────────────────────────────────┘
                         │
         health state changed?
                ┌────────┴──────────┐
                │ Yes               │ No (no-op)
                ▼
        Publish to Agent → status channel → Control Plane
        Control Plane Notification Service → customer channels
                │
                ▼
        Automated remediation decision tree:
        ┌─────────────────────────────────────────────────┐
        │  Scheduler DEGRADED (1 consecutive failure):    │
        │    → kubectl rollout restart deploy/scheduler   │
        │    → wait 60s, re-check                         │
        │                                                 │
        │  Scheduler DEGRADED (2 consecutive failures):   │
        │    → kubectl delete pod -l component=scheduler  │
        │    → wait 90s, re-check                         │
        │                                                 │
        │  Scheduler UNHEALTHY (3+ consecutive failures): │
        │    → emit UNHEALTHY event to Control Plane      │
        │    → page Flowdeck SRE (internal alert only)    │
        │    → notify customer via Notification Service   │
        │                                                 │
        │  Worker pods < min_replicas (2 consecutive):    │
        │    → kubectl scale deployment/worker            │
        │      --replicas={desired_min}                   │
        │                                                 │
        │  RDS unreachable:                               │
        │    → do NOT auto-remediate (DB restart is       │
        │      dangerous; alert only)                     │
        └─────────────────────────────────────────────────┘
```

The Health Monitor uses tier-aware thresholds. Enterprise deployments trigger `DEGRADED` after 1 consecutive failed check; Starter deployments wait for 2. This prevents alert noise on shared infrastructure where a single transient poll failure is common, while meeting tighter SLO response expectations for paying enterprise customers.

## 7. Runtime Plane — Structure and Isolation

Each deployment is a Kubernetes namespace named `dep-{deployment_id}`. Inside the namespace, the Deployment Controller maintains a set of standard Kubernetes resources:

```
  Namespace: dep-{deployment_id}
  │
  ├── ServiceAccount: airflow-{deployment_id}
  │     └── Annotation: eks.amazonaws.com/role-arn = arn:aws:iam::{acct}:role/{dp_id}-{dep_id}
  │         (IRSA: allows reading secrets at path /flowdeck/{dep_id}/*)
  │
  ├── ResourceQuota: airflow-quota
  │     requests.cpu: {from tier config}
  │     requests.memory: {from tier config}
  │     count/pods: {from tier config}
  │
  ├── LimitRange: airflow-limits
  │     default CPU request: 100m, limit: 2000m
  │     default memory request: 256Mi, limit: 4Gi
  │
  ├── NetworkPolicy: default-deny
  │     ingress: deny all except from ingress-controller namespace
  │     egress: allow to {rds_sg_cidr}:5432, {redis_sg_cidr}:6379,
  │             169.254.169.254:80 (IMDS for IRSA), s3.{region}.amazonaws.com
  │
  ├── Deployment: airflow-webserver
  │     image: apache/airflow:{version}
  │     env: AIRFLOW__DATABASE__SQL_ALCHEMY_CONN (from Secret)
  │     readinessProbe: httpGet /health :8080
  │
  ├── StatefulSet: airflow-scheduler
  │     image: apache/airflow:{version} (or OCI bundle digest)
  │     env: connection string, executor config
  │     livenessProbe: exec airflow jobs check --job-type SchedulerJob
  │
  ├── Deployment: airflow-triggerer
  │     image: apache/airflow:{version}
  │
  ├── Deployment: airflow-worker   (CeleryExecutor only)
  │     image: apache/airflow:{version} (or OCI bundle digest)
  │     ScaledObject: keda-worker-scaler (watches Redis queue depth)
  │
  └── Secrets: (populated by Secret Distributor, not by Helm)
        airflow-db-conn: AIRFLOW__DATABASE__SQL_ALCHEMY_CONN
        airflow-broker-url: AIRFLOW__CELERY__BROKER_URL
        airflow-connections: per-connection env vars
        airflow-variables: sensitive Airflow Variables
```

Tenant isolation is a runtime property tied to the deployment's `isolation_tier` field in desired state:

- **Starter (namespace tier):** The namespace above is the boundary. All Starter deployments share one EKS cluster and one shared node pool. `ResourceQuota` and `LimitRange` cap the resources. `NetworkPolicy` prevents cross-namespace pod traffic.
- **Pro (node-pool tier):** Same as Starter, but each Pro organisation gets a dedicated node pool with a Kubernetes taint `flowdeck.io/org={org_id}:NoSchedule`. All Pro deployment pods carry the matching toleration. Nodes in one organisation's pool cannot run another organisation's pods.
- **Enterprise (cluster tier):** A dedicated EKS cluster per organisation. The Deployment Controller uses a different `kubeconfig` (stored in Secrets Manager, fetched via IRSA by the DP Agent) when creating resources. The cluster has its own dedicated RDS, its own ElastiCache, and its own ALB.

## 8. DAG Delivery Pipeline

The preferred path for DAG delivery is an OCI bundle: the customer CI pipeline builds a container image containing DAGs and their Python dependencies, signs it with `cosign`, and submits the image digest to the Flowdeck API.

```
  Customer CI Pipeline
       │
       ▼
  docker build -t {ecr_repo}/dags:{sha} .
  cosign sign --key {signing_key} {ecr_repo}/dags:{sha}@{digest}
  POST /v1/deployments/{id}/dags
    body: { image: "{ecr_repo}/dags:{sha}@{digest}",
            signed_by: "{cosign_key_id}" }
       │
       ▼  API Gateway → Deployment Service
  1. Validate caller has DEPLOYER role on workspace
  2. Verify cosign signature against the org's registered public key
  3. Write new desired_state: dag_image = digest
  4. Increment desired_state_version
  5. Publish UPDATE command to NATS
       │
       ▼  Data Plane Agent → Deployment Controller
  Helm upgrade: set image.dag_bundle={digest}
       │
       ▼  Kubernetes rolling update
  New pods start → pull image from ECR
       │
       ▼  Each new pod on startup:
  airflow dags list --output json
  If exit code ≠ 0 → pod fails readinessProbe
  Rolling update halts, old pods stay RUNNING
       │
       ▼  If all pods pass readiness:
  Controller transitions to RUNNING
  Writes audit entry: { dag_image: digest, applied_at, operator_id }
       │
       ▼  If any pod fails (parse error):
  Emit DAG_PARSE_FAILED event with parser stderr
  Deployment stays at previous image (Helm --atomic rollback)
  Customer sees failure in UI with parser output
```

The audit entry that ties a DAG run to its code is: `dag_run_id → desired_state_version → dag_image_digest → git_sha (from image label)`. This chain is exact because every pod in the deployment runs the same image digest.

## 9. Identity, Authn, and Authz Internals

```
  Incoming request
       │
       ▼
  API Gateway: extract credential
  ┌─────────────────────────────────────────────┐
  │  JWT Bearer token?                          │
  │    1. Decode header, check alg=RS256        │
  │    2. Fetch public key from Identity Svc    │
  │       (cached 5 min in-memory)              │
  │    3. Verify signature                      │
  │    4. Check exp claim > now                 │
  │    5. GET redis BLOCKLIST:{jti}             │
  │       if exists → 401 Unauthorized          │
  │    6. Extract: sub, org_id, roles, scopes   │
  ├─────────────────────────────────────────────┤
  │  API Key?                                   │
  │    1. SHA-256 hash the presented key        │
  │    2. SELECT * FROM api_keys WHERE          │
  │       hash = $1 AND revoked_at IS NULL      │
  │    3. Extract: service_account_id, org_id,  │
  │       workspace_scope, deployment_scope     │
  └─────────────────────────────────────────────┘
       │
       ▼  Identity resolved → pass to RBAC check
  ┌─────────────────────────────────────────────┐
  │  RBAC decision                              │
  │  1. GET redis RBAC:{user_id}:{org_id}       │
  │     Cache hit → use cached role set         │
  │     Cache miss → query Aurora role table,   │
  │       store in Redis TTL 5 min              │
  │  2. Check: requested action ∈ role's        │
  │     allowed actions at the target scope     │
  │     (org / workspace / deployment)          │
  │  3. Inheritance: org role implicitly grants  │
  │     same role in all workspaces; workspace  │
  │     role does NOT grant org privileges      │
  └─────────────────────────────────────────────┘
       │ RBAC passed
       ▼
  ┌─────────────────────────────────────────────┐
  │  Policy Engine evaluation                  │
  │  Inputs: {actor, action, resource, context} │
  │  Policy rules: stored as CEL expressions   │
  │    in `policy_rules` Aurora table           │
  │  Examples:                                  │
  │   - "deploy only during business hours"     │
  │     → rule: timestamp in [09:00..18:00]     │
  │   - "require IdP group = prod-deployers"    │
  │     → rule: actor.idp_groups contains ...  │
  │  Result: {ALLOW, matched_rule_id}           │
  │        or {DENY, matched_rule_id, reason}   │
  │  Both outcomes are written to audit log     │
  └─────────────────────────────────────────────┘
       │ ALLOW
       ▼
  Service proceeds, audit entry written
  in same Aurora transaction as the mutation
```

## 10. Secrets Management

Customer secrets never enter the Control Plane database. The Control Plane stores only references. The Secret Distributor in the Data Plane is responsible for fetching the actual values and materialising them as Kubernetes Secrets.

```
  Customer creates connection in UI
       │
       ▼
  Deployment Service stores in Aurora:
    { connection_id, name, type: "postgres",
      secret_ref: "arn:aws:secretsmanager:us-east-1:{acct}:
                   secret:/flowdeck/{dep_id}/conn/my_postgres" }
  ← actual password is NOT stored in Aurora
       │
       ▼
  Customer writes actual secret to Secrets Manager at the reference path
  (or Flowdeck writes it on their behalf if using the managed backend)
       │
       ▼  Secret Distributor reconciliation loop (every 60s):
  ┌──────────────────────────────────────────────────────────┐
  │  For each deployment:                                    │
  │  1. Fetch list of secret_refs from desired state         │
  │  2. For each ref:                                        │
  │     aws secretsmanager get-secret-value                  │
  │       --secret-id {ref} --version-stage AWSCURRENT       │
  │     Using the deployment's IRSA role                     │
  │     (trust policy: only this namespace's SA can call it) │
  │  3. Compare fetched value's VersionId against            │
  │     the last-seen VersionId stored in BoltDB             │
  │  4. If changed (or first time):                          │
  │     kubectl create secret generic airflow-connections    │
  │       --from-literal=AIRFLOW_CONN_{NAME}={value}         │
  │       --namespace dep-{deployment_id}                    │
  │       --dry-run=client -o yaml | kubectl apply -f -      │
  │  5. If secret was rotated AND pods are running:          │
  │     kubectl rollout restart deployment/airflow-worker    │
  │       --namespace dep-{deployment_id}                    │
  │     (scheduler is NOT restarted unless it uses the       │
  │      rotated secret directly)                            │
  └──────────────────────────────────────────────────────────┘
       │
       ▼  If secret is deleted from backend:
  kubectl delete secret airflow-connections -n dep-{dep_id}
  kubectl rollout restart deployment/airflow-worker
  Emit SECRET_DELETED event → customer notification
```

IRSA scope is strictly per-deployment. Each deployment's IAM role has a permission boundary that allows `secretsmanager:GetSecretValue` only on ARNs matching `arn:aws:secretsmanager:{region}:{acct}:secret:/flowdeck/{deployment_id}/*`. The Data Plane agent's own IRSA role has `secretsmanager:GetSecretValue` on `arn:aws:secretsmanager:{region}:{acct}:secret:/flowdeck/dp-*` to fetch the deployment's service credentials, but it cannot read any deployment's customer secrets. The Secret Distributor calls Secrets Manager as the deployment's service account, not as the agent.

## 11. Autoscaling Architecture

Worker autoscaling (logical demand) and node autoscaling (physical capacity) are two separate control loops that do not coordinate with each other.

```
  WORKER AUTOSCALING (KEDA)
  ─────────────────────────
  KEDA ScaledObject (per deployment):
    scaleTargetRef: airflow-worker deployment
    triggers:
      - type: redis
        metadata:
          address: {elasticache_endpoint}:6379
          listName: celery  (or per-queue name)
          listLength: "5"   (target tasks-per-worker)
    minReplicaCount: {from desired_state.min_workers}
    maxReplicaCount: {from desired_state.max_workers}
    cooldownPeriod: 120     (seconds before scale-down)

  Scale-up:  KEDA reads Redis LLEN celery every 30s.
             If (queue_depth / current_workers) > listLength:
             kubectl scale deployment/worker --replicas=N
             N = ceil(queue_depth / listLength)

  Scale-down: When queue_depth < listLength * current_workers:
              KEDA patches worker Deployment replicas down.
              Airflow's worker graceful shutdown hook runs:
               - Send SIGTERM to worker
               - Worker stops accepting new tasks
               - Worker waits for in-flight tasks (timeout:
                 tier_config.worker_drain_timeout_seconds)
               - Pod exits cleanly
              preStop hook in pod spec:
                exec: ["airflow", "celery", "worker", "--drain"]


  NODE AUTOSCALING (KARPENTER)
  ────────────────────────────
  Karpenter NodePool (per isolation tier):
    requirements:
      - key: node.kubernetes.io/instance-type
        operator: In
        values: [m7i.large, m7i.xlarge, m7i.2xlarge, c7i.xlarge]
      - key: karpenter.k8s.aws/capacity-type
        operator: In
        values: [on-demand, spot]  # spot allowed for worker pools
    limits:
      cpu: {cluster_cpu_ceiling}
      memory: {cluster_memory_ceiling}
    disruption:
      consolidationPolicy: WhenUnderutilized
      consolidateAfter: 5m

  When KEDA scales workers up and new pods are Pending:
    Karpenter sees Pending pods → evaluates NodeClaim
    Selects cheapest fitting instance type
    Calls EC2 RunInstances API
    Node joins cluster, pod scheduled, starts in ~60s

  When workers scale down:
    After all worker pods on a node are gone:
    Karpenter sees node underutilized (< 10% CPU for 5m)
    Cordon → drain → EC2 TerminateInstances
```

The two loops are explicitly separated because they answer different questions. KEDA answers "how many workers do I need for this queue depth?" — a pure demand question. Karpenter answers "what nodes fit the pods I have?" — a pure bin-packing question. Coupling them (the old Cluster Autoscaler model) forced a single component to answer both questions, which made it slow to react to demand spikes and prone to thrashing.

## 12. Reconnection, Replay, and Fault Tolerance

Network losses, agent restarts, and Control Plane brief outages all resolve through the same mechanism: replay from durable state.

```
  SCENARIO: Data Plane loses connectivity to Control Plane
  ──────────────────────────────────────────────────────────

  t=0   gRPC stream breaks (NAT timeout / DNS failure / CP restart)
        Agent detects: stream.Recv() returns io.EOF or DeadlineExceeded
        Agent starts exponential backoff reconnect: 1s, 2s, 4s... max 60s

  t=0 to reconnect:
        Agent continues reconciling in-progress deployments
          using last-persisted desired state from BoltDB
        Health Monitor continues polling (no dependency on stream)
        Status events are buffered in BoltDB (ring buffer, max 1000 events)
        New commands from CP cannot arrive (stream is down)
        CP marks Data Plane DEGRADED after 60s with no heartbeat

  t=reconnect:
        Agent re-dials grpc.flowdeck.io with mTLS + JWT
        Sends RegisterDataPlane (re-registration is idempotent)
        Sends ResumeFromOffset{last_acked_nats_sequence}
          - CP resumes writing commands from that NATS offset
          - Any commands published during the outage are delivered in order
        Agent flushes buffered status events to Control Plane
        Agent processes replayed commands:
          - Checks desired_state_version vs applied_version in BoltDB
          - Duplicate commands (already applied) → silent ack
          - New commands (version > applied) → reconcile
        CP marks Data Plane HEALTHY after successful heartbeat

  t=reconnect+60s:
        Registry health state: HEALTHY
        Buffered statuses visible in UI
        All missed commands processed
        No split-brain: BoltDB applied-version map is the
          single source of truth for what has been executed
```

The key property here is that no data is lost and no operation is duplicated, because:
- The NATS message for a command is retained until the agent acks it.
- The agent acks only after writing the command to BoltDB.
- BoltDB is on a PersistentVolumeClaim (EBS), so it survives pod restarts.
- The applied-version map in BoltDB makes re-execution of already-completed commands a no-op.

## 13. Deployment Mode Matrix

The same Data Plane code runs in all modes. The only differences are where the Data Plane runs and what cloud provider interfaces it uses.

```
┌──────────────────┬────────────────────┬────────────────────┬─────────────────────┐
│ Mode             │ Control Plane      │ Data + Runtime     │ Network direction   │
├──────────────────┼────────────────────┼────────────────────┼─────────────────────┤
│ Public Cloud     │ Flowdeck AWS       │ Flowdeck VPC       │ Internal (same org) │
│ (this doc)       │                    │                    │                     │
├──────────────────┼────────────────────┼────────────────────┼─────────────────────┤
│ Hybrid Agent     │ Flowdeck AWS       │ Customer VPC       │ Outbound HTTPS from │
│                  │                    │                    │ customer to CP APIs │
├──────────────────┼────────────────────┼────────────────────┼─────────────────────┤
│ Remote           │ Flowdeck AWS       │ Flowdeck VPC       │ Outbound HTTPS from │
│ Execution        │ (scheduler + meta) │ (workers only in   │ remote worker pool  │
│                  │                    │ customer cluster)  │ to task queue API   │
├──────────────────┼────────────────────┼────────────────────┼─────────────────────┤
│ Private Cloud    │ Customer-hosted    │ Customer VPC       │ Fully internal to   │
│                  │ (same CP code)     │                    │ customer network    │
└──────────────────┴────────────────────┴────────────────────┴─────────────────────┘
```

In **Hybrid** mode, the customer installs the Data Plane Agent with a Helm chart provided by Flowdeck. The chart creates the `flowdeck-system` namespace, deploys the Agent, and mounts a bootstrap token from a Kubernetes Secret the customer pre-creates. From this point, the Agent's startup sequence is identical to the managed mode. The only configuration difference is that the cloud provider interfaces are configured to use the customer's AWS account (passed as environment variables to the Agent pod). The CP does not know, and does not need to know, that the Data Plane is in a different AWS account; it only sees the Agent's registered `dp_id` and `type=HYBRID`.

In **Remote Execution** mode, a customer-owned worker cluster registers as `type=REMOTE`. The Agent on that cluster does not run a full Deployment Controller; it runs only a Task Claim loop: poll the task queue NATS subject for this deployment, pull tasks, run them as Kubernetes Jobs in the local cluster, return results via the status channel. The task payload is envelope-encrypted: the CP encrypts the task body with a per-task ephemeral key, and only the authorised remote executor's IRSA role can call `kms:Decrypt` to unwrap it.

In **Private** mode, the entire Flowdeck Control Plane is packaged into a Helm umbrella chart that the customer installs on their own Kubernetes cluster. The Control Plane services run with the same images as the managed product. Aurora is replaced with a customer-managed PostgreSQL 15 (the JDBC URL is a configuration input), ElastiCache Redis with a customer-managed Redis 7, and NATS is included in the Helm chart as a bundled StatefulSet. The outbound-only stream between Data Plane and Control Plane becomes in-cluster networking; the protocol does not change.

## Closing Note

The architecture described in this document is built around a small number of concrete wiring decisions: one outbound gRPC stream per Data Plane, one monotonic version per deployment's desired state, one credentialed boundary in the Data Plane, one NATS stream per command/status/heartbeat concern. These decisions are not abstract principles — they are implementation constraints that the codebase must enforce in every merge. Every section above ties a component to its dependencies, its data, its failure mode, and its recovery path. Any change that introduces a second source of desired state, a second credentialed boundary, or an inbound connection into the Data Plane violates the model and must be explicitly reviewed as an architecture change, not a feature addition.
