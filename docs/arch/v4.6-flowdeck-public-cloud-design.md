# Flowdeck Platform — Public Cloud Design Document

> **Version:** 4.6
> **Last Updated:** April 2026
> **Scope:** AWS Public Cloud Deployment with Extensible Multi-Mode Architecture
> **Status:** Technical Design (Revision of v4.5)

---

## 1. Executive Summary

### 1.1 Purpose

This document defines the target architecture for Flowdeck's public cloud deployment on AWS and clarifies the stable contracts required for hybrid and private deployments. It refines v4.5 with clearer control-plane messaging boundaries, explicit reliability targets, and tighter security/compliance language.

### 1.2 Architecture Goals

- Keep tenant intent, orchestration, and runtime execution separated into explicit planes with auditable contracts.
- Ensure the Control Plane remains centralised, secure, and operationally independent from customer workload environments.
- Preserve deployment portability so the same APIs and orchestration contracts work in public, hybrid, and private modes.
- Isolate customer workloads by default and minimise blast radius across tenants.
- Provide measurable SLOs, incident response expectations, and operational observability across all layers.

### 1.3 Three-Plane Model

| Layer | Responsibility | Location |
|---|---|---|
| **Control Plane** | Identity, RBAC, billing, audit, desired state, public APIs | Flowdeck-managed environment |
| **Data Plane** | Reconciliation, provisioning, lifecycle operations, health and metrics collection | Flowdeck VPC (public mode) or customer VPC (hybrid/private) |
| **Runtime Plane** | Airflow services and tenant execution workloads | Same environment as Data Plane |

> Key principle: Control Plane owns intent and policy; Data Plane owns execution and state reconciliation; Runtime Plane runs workloads only.

### 1.4 Deployment Modes (In Scope)

| Mode | Control Plane | Data Plane | Runtime Plane | Infra Operator |
|---|---|---|---|---|
| **Public Cloud** | Flowdeck | Flowdeck VPC | Flowdeck VPC | Flowdeck |
| **Hybrid** | Flowdeck | Customer VPC | Customer VPC | Shared (Flowdeck + customer) |

Private-cloud packaged installs are an extension track and are not part of the implementation scope of this document.

### 1.5 Communication Contract

- Data Plane initiates all control communication outbound to Control Plane endpoints.
- Control Plane does not require inbound access into Data Plane networks.
- Command delivery and status reporting occur through one logical control channel contract with these guarantees:
  - **Delivery:** at-least-once command delivery.
  - **Ordering:** per-deployment command ordering by `desired_state_version`.
  - **Acknowledgement:** command is acknowledged only after local acceptance by Data Plane reconciler.
  - **Replay:** reconnecting agents can replay unacknowledged commands from last committed version.
  - **Backpressure:** Control Plane may throttle command emission per deployment/agent and returns explicit retry hints.
- Public and hybrid modes may differ in internal transport implementation, but must preserve the same external semantics above.

## 2. Architecture Overview

### 2.1 High-Level Design

The platform is designed around a strict desired-state loop:

1. User/API writes desired state in Control Plane.
2. Control Plane emits versioned command intent for a deployment.
3. Data Plane reconciles actual infrastructure/runtime state to desired state.
4. Data Plane publishes status, health, and observed endpoints back to Control Plane.
5. Control Plane exposes current/target state and audit trail to users.

### 2.2 Key Decisions

#### Control vs Data Plane Separation

Business and policy logic remains in Control Plane; infrastructure actuation remains in Data Plane. This preserves tenant governance consistency regardless of Data Plane location.

#### Runtime vs Data Plane Separation

Data Plane orchestrates. Runtime Plane executes. This enables replacing executor/runtime patterns without redesigning orchestration services.

#### Outbound-Only Data Plane Connectivity

A hard requirement for enterprise firewalls and regulated environments. All registration, heartbeat, command polling/streaming, and status updates originate from Data Plane.

#### Tiered Isolation Strategy

- Starter/Pro: shared cluster + namespace isolation controls.
- Enterprise: dedicated cluster isolation.
- Hybrid/private: customer-owned cluster boundary.

## 3. Control Plane Design

### 3.1 Core Services

- API Gateway (REST/gRPC/WebSocket)
- Identity and RBAC
- Organisation/Workspace services
- Deployment service (desired state + state machine)
- Data Plane registry
- Billing and notification
- Immutable audit logger

### 3.2 API and Versioning

- REST base path is versioned (`/v1`, `/v2`).
- gRPC contracts are semantic-versioned with backward compatibility at major version boundaries.
- Version retirement policy: minimum 12 months support after successor GA.

### 3.3 Deployment State Semantics

```
PENDING -> PROVISIONING -> CONFIGURING -> STARTING -> RUNNING
RUNNING -> UPDATING | SCALING | STOPPING
STOPPING -> STOPPED
ANY -> ERROR
STOPPED -> TERMINATING -> TERMINATED
```

State transitions include:
- `actor` (user/system/service account)
- `reason` and `correlation_id`
- `desired_state_version` and `observed_state_version`

### 3.4 Data Layer

- Aurora PostgreSQL for platform system-of-record data.
- Redis for revocation, ephemeral session/rate-limit state, and short-lived distributed locks.
- S3 for archive artifacts, snapshots, and immutable export datasets.

### 3.5 Reliability and Consistency Controls

- Idempotent command processing keyed by `command_id` and `desired_state_version`.
- At-least-once command semantics with deterministic reconciliation.
- Effectively-once external outcomes achieved through provider idempotency keys, state checks, and reconciliation (without requiring strict exactly-once transport guarantees).
- Command timeout and dead-letter handling with operator-visible failure reasons.

## 4. Data Plane Design

### 4.1 Responsibilities

- Receive and validate control commands.
- Execute provisioning and upgrade operations.
- Continuously reconcile drift.
- Report health and observed status.
- Keep secrets synchronized from configured backend.

### 4.2 Agent and Transport

The Data Plane Agent is the only externally connected component in Data Plane.

- Maintains authenticated outbound control channel.
- Performs bootstrap registration and credential rotation.
- Implements backoff and jitter on reconnect.
- Buffers non-terminal status updates during transient disconnects.

### 4.3 Reconciliation Contract

For each deployment:

1. Compare observed runtime state with desired state.
2. Apply minimal converging change set.
3. Emit transition status with correlation metadata.
4. Continue until convergence or terminal failure.

### 4.4 Provisioning Phases

- **Phase 1:** Infrastructure provisioning (RDS/Redis/storage/network dependencies)
- **Phase 2:** Kubernetes release apply/upgrade
- **Phase 3:** Readiness and functional validation

If phase N fails, recovery policy is phase-specific and captured in status reason codes.

### 4.5 Health Model

Component health (webserver/scheduler/workers/db/broker) rolls up into deployment health:

- `HEALTHY`
- `DEGRADED`
- `UNHEALTHY`

Escalation thresholds and recovery actions are policy-driven and configurable by tier.

## 5. Runtime Plane Design

### 5.1 Runtime Composition

Per deployment namespace:

- Webserver/API
- Scheduler
- Triggerer
- Worker pool (executor-dependent)
- Metadata DB and broker integrations
- DAG and dependency delivery mechanism

### 5.2 Executor Support

- CeleryExecutor (default for most managed workloads)
- KubernetesExecutor (pod-per-task model for heterogeneous workloads)
- Executor choice encoded in desired state and validated at admission time.

### 5.3 Autoscaling

- Workload scaling driven by queue depth and scheduler pressure signals.
- Node scaling managed independently from worker scaling.
- Scale-down policies include drain behavior to minimise task interruption.

### 5.4 Isolation Controls

- Namespace-level quotas and limits
- Default-deny network policy
- Per-deployment IAM role binding
- Pod security standard `restricted`

## 6. Network and Security Architecture

### 6.1 Network Topology

- Separate Control Plane and Data/Runtime network domains.
- Private subnets for application and data components.
- VPC endpoints for AWS APIs to reduce NAT dependence.

### 6.2 Access Patterns

- Public endpoints: API, UI, gRPC, WebSocket.
- Tenant Airflow UI routed by deployment-specific hostnames.
- Internal service-to-service traffic constrained by SG and policy controls.

### 6.3 Security Controls

- WAF + DDoS protections at perimeter.
- JWT/API key/mTLS layered authentication depending on endpoint type.
- Encryption at rest with KMS-managed keys.
- TLS in transit with enforced minimum protocol policy.
- Secret backends: Vault and/or cloud-native manager by deployment mode.

### 6.4 Compliance Clarification

- SOC 2 controls mapped to audit, change management, and access controls.
- HIPAA/GDPR/PCI support depends on deployment mode and regional configuration.
- Control Plane stores operational and account metadata; customer workload data residency is mode- and configuration-dependent.

## 7. Reliability, DR, and SLOs

### 7.1 Backup and Recovery

- Platform data: automated backups + PITR.
- Deployment data stores: tier-based backup frequency and retention.
- Optional cross-region replication for enterprise tiers.

### 7.2 Recovery Targets

| Scenario | Target RPO | Target RTO |
|---|---:|---:|
| Single pod/node failure | near-zero | <5 min |
| Deployment namespace failure | <= 15 min | <15 min |
| Runtime cluster loss | <= 1 hour | 30-60 min |
| Enterprise warm standby failover | <= 5 min | <10 min |

### 7.3 Service-Level Objectives

| SLO | Target |
|---|---|
| Control Plane API availability | 99.9% monthly |
| Deployment create/update success (non-user-error) | 99.5% monthly |
| Deployment reconciliation latency (P95) | <5 min |
| Data Plane reconnect time (P95) | <60s |

SLOs are measured on a rolling 28-day window from canonical telemetry queries owned by SRE. Error-budget consumption is reviewed each release cycle with multi-window burn-rate alerts for paging and ticketing thresholds.

---

## 8. Observability and Operations

### 8.1 Telemetry

- Metrics: Prometheus-compatible collection + long-term object storage.
- Logs: structured JSON with deployment/org correlation fields.
- Traces: end-to-end correlation across API, orchestration, and Data Plane events.

### 8.2 Alerting

- Platform alerts route to Flowdeck SRE.
- Deployment alerts route to customer-defined channels.
- Alert payloads include recommended remediation where deterministic.

### 8.3 Auditability

All mutating actions produce immutable audit records containing actor, request context, before/after snapshots, and correlation identifiers.

---

## 9. Extensibility

### 9.1 Stable Contract

The durable product contract is: **Control Plane API + Data Plane control semantics + desired-state model**.

Future deployment modes must preserve this contract while allowing internal implementation variation.

### 9.2 Planned Extensions

- Hybrid onboarding hardening and private connectivity options
- Remote execution mode aligned with Airflow task execution APIs
- Private cloud packaged installs
- Multi-cloud Data Plane support via provider abstraction

## 10. Notable Changes from v4.5

1. Corrected document versioning and revision metadata.
2. Clarified deployment mode ownership model (including private mode Control Plane ownership).
3. Reframed Control Plane ↔ Data Plane messaging to avoid transport-level ambiguity while preserving outbound-only guarantees.
4. Added explicit reliability semantics (idempotency, retries, dead-letter, timeout behavior).
5. Tightened compliance language to avoid implying "no PII in Control Plane".
6. Added concrete SLO table with measurable monthly targets and error-budget governance.
7. Restricted implementation scope to public + hybrid modes; kept private cloud as extension only.
8. Added explicit control-channel semantics (delivery, ordering, ack, replay, backpressure).
9. Replaced strict exactly-once claim with effectively-once outcomes via idempotency and reconciliation.
