## 1. What the Control Plane is

The Control Plane is the system of record for everything Flowdeck decides. It owns who you are, what organisations you belong to, what workspaces and projects those organisations contain, what deployments live in those projects, and what each deployment is supposed to look like. It also owns every audit event, every billing event, and every policy decision the platform has ever made.

The Control Plane does **not** own what is currently running. That is observed state, which the Data Plane reports. The Control Plane owns *desired state* — what should be running — and the Data Plane is responsible for closing the gap. This split is what makes the system resilient: if a Data Plane disconnects, customers can still update desired state through the API, and the Data Plane catches up when it reconnects.

**What the Control Plane runs:**
- All Flowdeck API endpoints (REST, gRPC, WebSocket).
- The identity, RBAC, and SSO stack.
- The resource hierarchy: Organisation → Workspace → Project → Deployment.
- The desired-state store for every deployment across all customers.
- The command bus that delivers desired state to Data Planes.
- Audit, billing, and notification pipelines.

**What the Control Plane explicitly does not do:**
- Run Airflow.
- Execute Terraform.
- Make Kubernetes API calls to runtime clusters.
- Open inbound connections into a Data Plane network.

Every one of those things is the Data Plane's responsibility.

The Control Plane runs on an EKS cluster in the `flowdeck-prod-control` AWS account, separate from the runtime account where customer workloads live. Every service is stateless at the process level — all durable state lives in Aurora PostgreSQL, ElastiCache Redis, or NATS JetStream. Horizontal scaling is safe for all services because no local state survives a pod restart.

## 2. Ingress layer

External traffic enters through a layered defence designed to absorb attacks before they reach application code.

**AWS WAF v2** sits at the perimeter. It applies OWASP Core Rule Set v3, enforces per-source-IP rate limits, checks IP reputation lists, and blocks known bad user agents. WAF is configured in blocking mode for production.

**CloudFront** handles static UI assets. It terminates TLS at AWS edge locations, caches static content, and provides AWS Shield Standard for volumetric DDoS mitigation at no extra cost. Dynamic API requests pass through CloudFront uncached with a `Cache-Control: no-store` header.

**Application Load Balancer (ALB)** routes dynamic traffic to three target groups:

| Target group | Hostname | Protocol | Purpose |
|---|---|---|---|
| `api` | `api.flowdeck.io` | HTTPS / HTTP2 | REST API from UI, CLI, and CI pipelines |
| `grpc` | `grpc.flowdeck.io` | HTTPS / HTTP2 | gRPC stream from Data Plane Agents |
| `ws` | `wss.flowdeck.io` | WSS | WebSocket — real-time deployment status to browser |

The REST and gRPC target groups are separated because they have different authentication models and listener policies. REST accepts bearer JWTs and API keys. gRPC requires both an mTLS client certificate and a JWT. Separate hostnames allow separate ACM certificates and separate ALB listener rules.

ACM certificates are issued for `*.flowdeck.io` and renewed automatically. The gRPC ALB listener additionally requires client certificate validation against Flowdeck's internal CA — it rejects connections that do not present a valid mTLS certificate before any application code runs.

## 3. API Gateway

The API Gateway is the single entry point to all Control Plane application logic. It is a stateless Go service with a minimum of three replicas (Horizontal Pod Autoscaler scales on CPU and request latency) and is the first component that runs application-level code.

It does exactly four things: terminate the protocol, authenticate the request, enforce rate limits, and forward to the correct downstream service.

### 3.1 Protocol termination

**REST** requests arrive as HTTP/2 over TLS from the ALB. The gateway parses the path, verb, and headers. Requests destined for `/v1/...` are the public API; requests to `/internal/...` are rejected at this layer and are not reachable from outside the cluster.

**gRPC** requests arrive on the gRPC target group. The gateway runs a gRPC server on the same process, routing by the gRPC service and method name. Data Plane Agents always connect here; no human user uses this endpoint.

**WebSocket** connections upgrade from HTTP at the `/ws` path and are proxied to the WebSocket Hub — a separate service that maintains long-lived connections and pushes events from `deployments.status` to the browser in near-real-time.

### 3.2 Authentication

Every request is authenticated before it reaches any business logic. The authentication path depends on the credential type:

**Bearer JWT (human user, CLI):**

1. Decode the JWT header; reject anything not signed with RS256.
2. Fetch the public key from the Identity Service's JWKS endpoint (cached 5 minutes in-process).
3. Verify the signature, `exp` claim, and `iss` claim.
4. Hash the `jti` claim with SHA-256 and check the Redis revocation blocklist. A match means the token was explicitly revoked — reject.
5. Extract `org_id`, `user_id`, `roles`, and `scopes` from the claims and attach to the request context.

**API key (CI/CD, service accounts):**

1. Extract the key from the `Authorization: Bearer` or `X-API-Key` header.
2. SHA-256 hash the presented key and look it up in the `api_keys` table in Aurora.
3. Resolve the associated service account, organisation, and permission set.
4. Attach identity context to the request.

**gRPC mTLS + JWT (Data Plane Agent):**

1. The ALB validates the mTLS client certificate against Flowdeck's internal CA before the request reaches the gateway. A missing or invalid certificate is rejected at the load balancer.
2. The gateway extracts the `dp_id` from the certificate's CN and SAN fields.
3. The gateway reads the `:authorization` gRPC metadata header and validates the JWT using the same RS256 path as bearer tokens, but the JWT's `scope` claim must include `dataplane_grpc` — ordinary user JWTs are rejected here even if their signature is valid.
4. Both checks must pass. The certificate proves network identity; the JWT proves logical identity.

**WebSocket token:**

A short-lived 5-minute token issued by the Identity Service specifically for a WebSocket session. It carries a narrow scope (`ws_read_only`) that cannot be replayed against the REST API. The gateway validates it on connection upgrade and does not re-validate on each message — the connection itself is the authenticated session.

### 3.3 Rate limiting

Rate limiting is layered:

1. **WAF** — per-source-IP limit at the edge; prevents volumetric traffic from reaching application code.
2. **API Gateway** — per-user and per-organisation limit using atomic `INCR` operations in Redis with a 60-second sliding window. Burst headroom: 2× the base limit for up to 10 seconds, to absorb legitimate spikes without rejecting them.
3. **Circuit breaker** — on every downstream call. Opens at 50% error rate over a 10-second window. While open, returns `503 Service Unavailable` with a `Retry-After: 30` header. Closes after a half-open probe succeeds. This prevents one degraded downstream from queuing up the entire API layer.

Rate limit responses carry `X-RateLimit-Limit`, `X-RateLimit-Remaining`, and `X-RateLimit-Reset` headers so CLI clients can back off gracefully.

## 4. Identity Service

The Identity Service is the only component in the platform permitted to mint or validate credentials. Every other service's authentication path goes through it. It runs as a Kubernetes Deployment with a minimum of two replicas.

### 4.1 Human authentication

**Email and password:** Passwords are stored as bcrypt hashes with cost 12. Login returns a JWT plus an opaque refresh token. TOTP is the supported second factor; it is required for all users with `org:owner` or `org:admin` roles, optional for others. Failed login attempts are rate-limited with an exponential lockout: 5 failures in 60 seconds trigger a 15-minute lockout of that user account, not the IP (to avoid locking out NAT-shared IPs in enterprise networks).

**Enterprise SSO (SAML 2.0 / OIDC):** An organisation can configure one or more IdP connections. On SSO login, the Identity Service acts as the service provider (SAML) or relying party (OIDC), receives the IdP's assertion, and maps the user's IdP group memberships to Flowdeck workspace roles via a configurable attribute mapping table in Aurora. This means adding a user to the correct Okta group automatically grants the right Flowdeck access without a separate provisioning step. The first SSO login for a new user auto-provisions the user record. Subsequent logins update group membership from the IdP assertion.

**SCIM v2:** Enterprises can push user provisioning and deprovisioning from their IdP directly via SCIM v2. SCIM deprovisioning sets the user's status to `DISABLED` and immediately adds all their active JWTs to the Redis revocation blocklist.

### 4.2 Token lifecycle

The Identity Service issues two tokens on successful authentication:

| Token | Type | Signed with | Expiry | Purpose |
|---|---|---|---|---|
| Access token | JWT (RS256) | Platform private key (rotated quarterly) | 1 hour | Authenticates API requests |
| Refresh token | Opaque 256-bit random | Stored as SHA-256 hash in Aurora | 30 days, rotated on each use | Issues new access tokens |

JWT claims: `sub` (user ID), `org_id`, `roles` (list of `{scope:role}` pairs), `scopes` (API scope list), `jti` (unique token ID for revocation), `iat`, `exp`.

The refresh token is stored as a SHA-256 hash in the `refresh_tokens` table alongside its expiry and a `used_at` timestamp. When a refresh is requested, the service fetches the hash, checks it has not expired or been used, issues a new access token and a new refresh token, and immediately marks the old refresh token as used. This rotation-on-use model means a stolen refresh token can only be used once before it becomes invalid when the legitimate client next refreshes.

### 4.3 Token revocation

Revocation is implemented as a Redis blocklist keyed by `jti`. Entries are written on:

- Explicit logout.
- Password change.
- SCIM deprovisioning.
- Security incident (admin action via internal tooling).
- Refresh token rotation (old refresh token blocks the old access token's `jti`).

Every request validates the JWT signature and then checks the blocklist (Redis GET with ~0.3ms latency at production volumes). Blocklist entries expire automatically at the token's natural `exp` time. This is how revocation works without shortening the 1-hour token lifetime: any access token can be invalidated within the 0.3ms of the next request.

### 4.4 RBAC

Authorisation is a hierarchical, additive role model. Roles are assigned at four levels of the resource hierarchy: Organisation, Workspace, Project, and Deployment. A role assigned at a higher level implies the equivalent access at all levels below it.

| Scope | Role | What it grants |
|---|---|---|
| Organisation | `org:owner` | Full control including billing and member management. Implicit admin of every workspace/project/deployment within the org. |
| Organisation | `org:admin` | Manage workspaces, projects, and members. No billing access. |
| Organisation | `org:member` | Access only to workspaces explicitly assigned. No management capabilities. |
| Workspace | `workspace:admin` | Manage everything in the workspace: deployments, connections, variables, member access. |
| Workspace | `workspace:developer` | Create and update deployments and DAGs. Cannot manage workspace members. |
| Workspace | `workspace:viewer` | Read-only across the workspace. |
| Project | `project:lead` | Manage deployments within the project. |
| Project | `project:developer` | Deploy and update within the project. |
| Deployment | `deployment:deployer` | Push DAG bundles. Used for service accounts in CI/CD pipelines. |
| Deployment | `deployment:viewer` | Read-only on a single deployment. |

Roles are additive: an `org:owner` is implicitly a `workspace:admin` for every workspace in that org. There are no deny rules — if a role grants access, access is granted. There is no mechanism for a lower-level role to override a higher-level grant.

RBAC decisions are cached in Redis per `(user_id, resource_id)` with a 5-minute TTL. Role changes explicitly invalidate the cache entry by publishing to a Redis pub/sub channel that all API Gateway instances subscribe to. This means the effective propagation time for a role removal is at most 5 minutes in steady state, and faster if the gateway receives the invalidation before the TTL expires.

### 4.5 Service accounts and API keys

Service accounts are machine identities used by CI/CD pipelines and automated scripts. A service account is created under an organisation and assigned roles using the same model as human users.

Each service account can have multiple API keys. An API key is a 256-bit random secret generated once, shown to the user in plaintext exactly once (at creation), and stored in Aurora as a SHA-256 hash. On each API request, the gateway hashes the presented key and looks it up in the `api_keys` table. There is no way to retrieve the plaintext after creation; a lost key must be rotated.

API keys do not expire by default but can be set to expire at a configurable date. They are scoped to an explicit list of API operations at creation time — a key for a CI/CD pipeline can be given only the `deployment:dag.deploy` scope, which prevents it from deleting deployments even if it is stolen.

## 5. Organisation, Workspace, and Project Services

These three services share an Aurora schema and treat the `org → workspace → project → deployment` hierarchy as a single aggregate. They use serialisable transactions plus short-lived Redis distributed locks around all resource-creation operations to prevent quota races.

### 5.1 Organisation Service

An Organisation is the billing and compliance boundary — usually a company. Creating an organisation:

1. Validates the organisation slug (globally unique, URL-safe, immutable after creation).
2. Creates the organisation record in Aurora with plan, region, quota, and billing metadata.
3. Provisions a Stripe customer ID and links it to the organisation.
4. Creates a default workspace named `default`.
5. Sets the creator as the first `org:owner`.

**Quotas** are enforced at admission time. Each organisation has a quota record with limits on: maximum active deployments, total CPU requests, total memory requests, and total storage. When the Deployment Service receives a create request, it reads the organisation's current consumption from Aurora (read replica) and the quota record, acquires a short-lived Redis lock (`quota_lock:{org_id}`, TTL 5 seconds, SET NX), and validates that the new request does not exceed the quota. The lock prevents two concurrent creates from both passing the check against the same consumed value.

When an organisation crosses a soft quota threshold (80% of any limit), the owner is notified. Hard quota is enforced at 100% — new deployments are blocked but existing ones keep running.

### 5.2 Workspace Service

A Workspace is a logical grouping of projects inside an organisation, typically aligned to a team or environment (e.g. `data-engineering`, `production`). Workspaces have their own member and role assignments, independent of organisation-level membership.

**Workspace-level configuration carried on each workspace:**

| Config field | Type | Effect |
|---|---|---|
| `default_airflow_version` | Enum | Pre-fills the Airflow version when creating a new deployment. |
| `allowed_executor_types` | Set | Restricts which executors are permitted for new deployments in this workspace. |
| `require_oci_bundles` | Boolean | When `true`, DAG deployment via git-sync or S3 sync is rejected. Only OCI bundle path is accepted. |
| `ci_cd_only` | Boolean | When `true`, only service-account API tokens with the `DEPLOYER` role can create or update deployments. Human users can read but not write. This is the "all production changes go through the pipeline" policy. |
| `notification_channels` | List | Where alerts for this workspace are routed (email, Slack, PagerDuty). |

### 5.3 Project Service

A Project is a **DAG codebase unit** — a repository of DAG files that gets deployed to a deployment as its deploy target. A project is not a container for deployments; it is a peer of deployments at the workspace level, with a formal `target_deployment_id` relationship.

**Project record schema:**

| Field | Type | Purpose |
|---|---|---|
| `project_id` | UUID | Primary key. |
| `workspace_id` | UUID | The workspace this project belongs to. |
| `name` | String | Human-readable name (e.g. `payments-platform`). |
| `slug` | String | URL-safe identifier, unique within the workspace. |
| `target_deployment_id` | UUID | The deployment this project deploys DAG bundles to. Must be in the same workspace. |
| `manifest` | JSONB | Optional contents of the project's `project.yaml` — declares required Airflow version, Python packages, providers, connections. Informational in V1; not enforced. |
| `created_by` | UUID | User who created the project. |
| `created_at` | Timestamp | Creation time. |

**Deployment relationship:**

- One project → one `target_deployment_id` (V1 constraint).
- One deployment can be referenced as `target_deployment_id` by **one or many projects**. When more than one project targets the same deployment, that deployment is considered **shared**.
- The Project Service validates at creation time that `target_deployment_id` exists in the same workspace as the project.

**V1 deploy mechanism — DAG-only:**

In V1, a project can only push DAG files to its target deployment. It cannot change the deployment's environment (Airflow version, Python packages, providers, connections, variables, resource sizing). Environment changes require a separate action by a user with `workspace:admin` or `deployment:owner` role.

The Project Service records each DAG deploy as a `project_deploy_event`:

```json
{
  "deploy_id": "uuid",
  "project_id": "uuid",
  "target_deployment_id": "uuid",
  "dag_bundle_digest": "sha256:...",
  "deployed_by": "service_account_id",
  "deployed_at": "2026-05-03T...",
  "result": "SUCCESS | FAILED",
  "reason": "DAG_PARSE_FAILED | null"
}
```

This record is written to the audit log and is what ties a running DAG back to the specific project version that deployed it.

**project.yaml manifest (V1: stored, not enforced):**

When a project repo contains a `project.yaml`, the CI pipeline can submit it alongside the DAG deploy. The Project Service stores the parsed manifest in the `manifest` JSONB column. In V1 it is displayed to the deployment owner as a checklist but does not gate the deploy. The Control Plane returns the manifest on `GET /v1/projects/{id}` so deployment owners can see what each project claims to need.

**DAG ID registry — shared deployment conflict prevention:**

Airflow requires all DAG IDs to be unique within a deployment. On a shared deployment (multiple projects pointing at the same `target_deployment_id`), two projects could independently define a DAG with the same ID. Airflow would silently discard one — whichever file was parsed last — with no error surfaced to either team.

To prevent this, the Project Service maintains a `dag_id_registry` table per deployment:

```
dag_id_registry
  deployment_id   UUID       FK → deployments
  dag_id          TEXT       The DAG ID string as written in the Python file
  project_id      UUID       FK → projects — the project that owns this DAG ID
  registered_at   TIMESTAMP
  PRIMARY KEY (deployment_id, dag_id)
```

On every DAG bundle deploy to a shared deployment, the Project Service:

1. Parses all `dag_id` values from the incoming bundle (using the same parse gate as the readiness probe — `airflow dags list --output json`).
2. For each `dag_id`:
   - If not in `dag_id_registry` for this deployment → insert row, owned by the deploying project.
   - If in `dag_id_registry` and `project_id` matches the deploying project → allow (update).
   - If in `dag_id_registry` and `project_id` is a **different** project → **reject the deploy** immediately with HTTP 409 and error body: `{"error": "DAG_ID_CONFLICT", "dag_id": "etl_pipeline", "owned_by_project": "fraud-detection"}`.
3. If all DAG IDs pass, the deploy proceeds. Registry rows for DAG IDs that are in the existing bundle for this project but absent from the new bundle (deleted DAGs) are removed.

On a **dedicated** deployment (only one project), the registry still exists but conflict checks always pass — no other project can claim IDs on that deployment.

The registry is also returned on `GET /v1/deployments/{id}/dag-id-registry` so the deployment owner can see which project owns each DAG ID running on their deployment.

**Member and role assignments:**

Each project has its own role assignments, independent of workspace membership. Roles are:

| Role | What it allows |
|---|---|
| `project:lead` | Manage the project: rename it, change `target_deployment_id`, manage project members, view deploy history. |
| `project:developer` | Deploy DAG bundles to the target deployment. Cannot change project config or workspace settings. |
| `project:viewer` | Read-only: view project config and deploy history. |

Workspace-level roles inherit downward — a `workspace:admin` implicitly has `project:lead` on every project in the workspace.

## 6. Deployment Service and state machine

The Deployment Service is the orchestrator of deployment lifecycle. It is the service that owns the desired state of every deployment and drives the transition between deployment lifecycle states.

### 6.1 Desired state

Each deployment's desired state is stored as a JSONB document in the `deployments` table in Aurora, alongside normalised columns for fields that carry foreign-key constraints (workspace ID, project ID, Airflow version enum, executor type, isolation tier).

The JSONB document contains:

```json
{
  "airflow_version": "2.9.2",
  "executor": "CeleryExecutor",
  "isolation_tier": "PRO",
  "resource_profile": {
    "scheduler": { "cpu_request": "500m", "cpu_limit": "2000m", "memory_request": "1Gi", "memory_limit": "3Gi" },
    "webserver": { "cpu_request": "250m", "cpu_limit": "1000m", "memory_request": "512Mi", "memory_limit": "2Gi" },
    "triggerer": { "cpu_request": "100m", "cpu_limit": "500m", "memory_request": "512Mi", "memory_limit": "2Gi" },
    "worker":    { "cpu_request": "500m", "cpu_limit": "2000m", "memory_request": "1Gi",  "memory_limit": "4Gi",
                   "concurrency": 16, "min_workers": 1, "max_workers": 10 }
  },
  "dag_source": {
    "type": "OCI",
    "image": "123456789.dkr.ecr.us-east-1.amazonaws.com/dags:abc123@sha256:...",
    "signed_by": "cosign-key-id"
  },
  "env_refs": [
    { "name": "MY_ENV_VAR", "secret_ref": "arn:aws:secretsmanager:...:secret:/flowdeck/{dep_id}/env/MY_ENV_VAR" }
  ],
  "connection_refs": [
    { "name": "my_postgres", "secret_ref": "arn:aws:secretsmanager:...:secret:/flowdeck/{dep_id}/conn/my_postgres" }
  ],
  "variable_refs": [
    { "name": "my_secret_var", "secret_ref": "arn:aws:secretsmanager:...:secret:/flowdeck/{dep_id}/var/my_secret_var", "sensitive": true }
  ],
  "log_retention_days": 90
}
```

Using JSONB lets the schema evolve without migrations every time a new configuration field is added. Normalised columns hold only the fields used in queries and indexes.

### 6.2 Desired state versioning

Every change to desired state is atomic. The Deployment Service:

1. Acquires a row-level lock on the deployment row.
2. Validates the change against workspace policy and organisation quota.
3. Writes the new JSONB document to the `deployments` table.
4. Increments `desired_state_version` (monotonically increasing integer, never reused).
5. Writes an audit entry (actor, before/after snapshots, correlation ID).
6. Publishes a command to NATS JetStream inside the same Aurora transaction using the transactional outbox pattern.

The last point is critical: the NATS publish is inside the transaction boundary. There is no scenario in which desired state is updated but no command is published, because they commit together. If the transaction rolls back, neither the Aurora write nor the NATS message is committed.

### 6.3 Deployment state machine

A deployment moves through the following states. Every transition is written to the `deployment_events` table with: `deployment_id`, `from_state`, `to_state`, `actor_id` (user or system), `reason_code`, `correlation_id`, and `timestamp`.

**Creation path:**

| State | Meaning | Who sets it |
|---|---|---|
| `PENDING` | Deployment record created, command published to NATS. | Deployment Service |
| `PROVISIONING` | Data Plane Agent has accepted the CREATE command. Phase 1 (infra) running. | Data Plane Agent |
| `CONFIGURING` | Phase 1 complete. Phase 2 (Helm/runtime) running. | Deployment Controller |
| `STARTING` | Phase 2 complete. Phase 3 (validation) running. | Deployment Controller |
| `RUNNING` | All validation checks passed. Deployment is live. | Deployment Controller |

**Steady-state branches from RUNNING:**

| State | Triggered by | Resolution |
|---|---|---|
| `UPDATING` | New desired state received while RUNNING. | Returns to RUNNING after successful reconciliation. |
| `SCALING` | Resource profile or worker count changed only. | Returns to RUNNING; faster path than full UPDATE. |
| `STOPPING` | Customer or system stop command. | Moves to STOPPED. |
| `ERROR` | Unrecoverable failure in any phase. | Stays in ERROR until manual intervention. |

**From STOPPED:**

| State | Triggered by | Resolution |
|---|---|---|
| `STARTING` | Restart command. | Returns to RUNNING. |
| `TERMINATING` | Delete command. | Moves to TERMINATED. |

`TERMINATED` is a terminal state. The deployment record is retained for audit purposes but no cloud resources remain.

### 6.4 Observed state

Observed state is stored separately from desired state. The `deployment_observed_states` table holds the last-reported state from the Data Plane — component health, active deployment phase, last heartbeat, endpoint URL — and is updated exclusively by the Data Plane via the status stream. The API returns both desired and observed state on a GET request so that callers can see the gap when the two diverge.

## 7. Data Plane Registry

The Data Plane Registry stores the identity and routing information for every Data Plane that the Control Plane manages.

### 7.1 Registry record schema

Each registry row holds:

| Column | Type | Purpose |
|---|---|---|
| `dp_id` | UUID | Primary key, generated at provisioning time. |
| `type` | Enum | `MANAGED`, `HYBRID`, `PRIVATE`, `REMOTE`. |
| `region` | String | AWS region code (e.g. `us-east-1`). |
| `grpc_endpoint` | String | The address the Control Plane uses to stream commands back when needed. (Informational only — the stream is always initiated outbound from the Data Plane.) |
| `health` | Enum | `HEALTHY`, `DEGRADED`, `OFFLINE`. |
| `capacity_score` | Float | Aggregate load metric reported by the Agent (0.0 = empty, 1.0 = full). Used for routing new deployments. |
| `last_heartbeat_at` | Timestamp | Last time a heartbeat was received from this Data Plane's Agent. |
| `cloud_account_id` | String | AWS account ID where this Data Plane's runtime cluster lives. |
| `agent_version` | String | Software version of the Data Plane Agent. Used to gate feature flags. |

### 7.2 Registration flow

When a new Data Plane Agent starts (or restarts), it registers itself by sending a `RegisterDataPlane` message on the gRPC stream. The message carries: `dp_id`, `type`, `region`, `agent_version`, `cloud_account_id`, and a one-time bootstrap token.

The registry validates the bootstrap token (single-use, 24-hour TTL, stored as a SHA-256 hash in Aurora), creates or updates the registry row, and returns a `RegisterDataPlane` response containing a short-lived JWT scoped to gRPC operations. The Agent stores this JWT in BoltDB, discards the bootstrap token, and uses the JWT for all subsequent gRPC calls.

Bootstrap tokens are provisioned out-of-band when a new Data Plane is set up (via Flowdeck's internal provisioning tooling or, for hybrid customers, via a guided onboarding flow). They are single-use to prevent replay attacks.

### 7.3 Heartbeat and health

Agents send a heartbeat every 15 seconds on the `agents.heartbeat` NATS subject (last-value-per-agent semantics — only the most recent heartbeat is retained). Each heartbeat carries the Agent's `dp_id`, a `capacity_score`, and a summary of active deployment states.

A background registry process reads the NATS stream and updates `last_heartbeat_at` in Aurora. Health transitions:

- `now() - last_heartbeat_at > 60s` → Data Plane marked `DEGRADED`. Internal alert fired. Commands continue to be queued.
- `now() - last_heartbeat_at > 300s` → Data Plane marked `OFFLINE`. Customers with deployments on this Data Plane are notified. New deployment commands are not routed to this Data Plane.

When a `DEGRADED` or `OFFLINE` Data Plane reconnects, it returns to `HEALTHY` after the first successful heartbeat round-trip, and the registry updates the health field accordingly.

### 7.4 Routing new deployments

When the Deployment Service emits a `CREATE` command for a new deployment, it calls the registry's routing function: "return the healthiest `MANAGED` Data Plane in region `{r}` with capacity for one more deployment". The registry returns the `dp_id` of the least-loaded healthy member of the shared Data Plane pool for that region (see `v5.3-flowdeck-shared-data-plane.md` for pool architecture).

The selected `dp_id` is stored on the deployment record and does not change after assignment. A deployment cannot be transparently migrated to a different Data Plane without a coordinated handoff operation — there is no automatic rebalancing.

## 8. Command and status channel to Data Planes

The channel between the Control Plane and Data Plane Agents is the most architecturally important part of the system. It is a single bidirectional gRPC stream, always initiated by the Data Plane Agent, always outbound from the Data Plane's network. The Control Plane never opens an inbound connection into any Data Plane network.

### 8.1 Stream lifecycle

**Connection setup (ordered steps):**

1. Agent opens a TCP connection to `grpc.flowdeck.io:443`.
2. TLS handshake. ALB validates the Agent's mTLS client certificate against Flowdeck's internal CA before the packet reaches application code. A missing or invalid certificate terminates the connection at the load balancer.
3. Agent sends a `RegisterDataPlane` message. Control Plane validates the bootstrap token (first startup) or verifies the mTLS certificate matches an existing registry row (reconnect).
4. Control Plane returns `RegisterDataPlane` response with a short-lived JWT (first startup) or confirms registration (reconnect).
5. Agent opens the bidirectional `Connect` streaming RPC (`/flowdeck.v1.DataPlaneService/Connect`) carrying both the mTLS certificate and the JWT in gRPC metadata.
6. Stream is established. Both sides maintain independent goroutines for reading and writing.

**On the established stream, three message types flow:**

| Message | Direction | Frequency | Purpose |
|---|---|---|---|
| `Command` | CP → Agent | On desired state change | Carry a new `desired_state_version` + full desired state JSON + command type (CREATE / UPDATE / DELETE / STOP). |
| `CommandAck` | Agent → CP | Immediately after local persistence | Agent confirms it has durably written the command to BoltDB. Control Plane uses this to ack the NATS message. |
| `StatusUpdate` | Agent → CP | After phase transitions, on health change | Reports deployment state, phase, reason code, observed endpoint, health summary. |
| `StatusAck` | CP → Agent | Immediately after writing status to Aurora | Confirms the status was persisted. Agent can discard its locally buffered copy. |
| `Heartbeat` | Agent → CP | Every 15 seconds | Carries `dp_id` and `capacity_score`. Updates registry health. |
| `HeartbeatAck` | CP → Agent | Immediately | Confirms the heartbeat was received. |

### 8.2 At-least-once delivery guarantee

The two-hop acknowledgement model:

1. Deployment Service writes desired state to Aurora and publishes a command to NATS (`deployments.commands.{dp_id}.{dep_id}`). NATS retains the message until acknowledged.
2. Control Plane's writer goroutine reads the NATS message and sends it on the gRPC stream to the Agent.
3. Agent writes the command to BoltDB, sends `CommandAck` back on the stream.
4. **Only then** does the Control Plane's NATS consumer call `+ACK` on the NATS message, removing it from the stream.

This two-hop ack is what makes the system at-least-once safe. The NATS message is retained until the Agent has physically persisted the work. If the stream drops between steps 2 and 3, NATS retains the message and redelivers it on the Agent's next connection. The Agent deduplicates by comparing the incoming `desired_state_version` against its locally stored applied version for that deployment. A version it has already processed is silently acked and discarded.

### 8.3 Reconnection and replay

If the gRPC stream breaks (network loss, Agent restart, idle TCP timeout), the Control Plane marks the stream as disconnected. Commands published to NATS during the outage accumulate in the stream — NATS retains them because they have not been acked.

When the Agent reconnects:

1. Agent re-registers (if a restart) or re-opens `Connect` (if a reconnect).
2. Control Plane's writer goroutine resumes reading from the NATS stream for this `dp_id` and sends accumulated commands.
3. Agent deduplicates against its local applied-version map.
4. For any deployment that was `in_progress` at reconnect time, the Agent sends a `StatusUpdate` to resync the Control Plane before continuing reconciliation.

Reconnection is transparent to customers. The deployment's observed state in the API may lag during the outage (the last known state is returned), but desired state is always consistent and commands are never lost.

### 8.4 Credential rotation

**mTLS certificate:** Seven days before the certificate expires, the Agent calls `RotateCertificate` on the Control Plane (using the existing certificate to authenticate). The Control Plane issues a new certificate signed by Flowdeck's internal CA, returns it on the stream, and marks the old certificate as `PENDING_REVOCATION`. The Agent writes the new certificate to BoltDB and uses it on the next gRPC connection. The old certificate is fully revoked after a 1-hour grace window that covers in-flight connections during rotation.

**gRPC JWT:** Rotated every 23 hours using a `RefreshToken` RPC. The overlap window is 1 hour (old token remains valid) to avoid invalidating in-flight requests. The Agent stores the new JWT in BoltDB before the old one expires.

Both rotations are fully automated and require no operator action under normal circumstances.

## 9. Event bus — NATS JetStream

NATS JetStream is the internal event bus for the Control Plane and the command delivery channel to Data Plane Agents.

**Infrastructure:** 3-node Raft StatefulSet on the Control Plane EKS cluster. Each node uses an EBS gp3 volume for durable message storage. The Raft quorum requires 2 of 3 nodes to commit a message — one node failure is tolerated without message loss or availability impact.

### 9.1 Streams

**`deployments.commands`**

- **Type:** Work-queue (each message delivered to exactly one consumer).
- **Subject:** `deployments.commands.{dp_id}.{deployment_id}`.
- **Retention:** Until explicitly acknowledged by the Data Plane Agent. A message is never discarded until the Agent has durably accepted the corresponding command.
- **Redelivery:** If not acked within `ack_wait` (default 30 seconds), the message is redelivered with exponential backoff, up to 30 minutes between attempts. After `max_deliver` (configurable, default 50) attempts without ack, the message is moved to a dead-letter subject for human inspection.
- **Payload:** `{command_type, deployment_id, dp_id, desired_state_version, desired_state_json, idempotency_key}`.

**`deployments.status`**

- **Type:** Limits-based stream (bounded per-subject retention).
- **Subject:** `deployments.status.{deployment_id}`.
- **Retention:** Last 100 messages per deployment subject. 100 is enough for the UI's history view and for debugging without growing unbounded.
- **Consumers:** Control Plane writes status events to Aurora on receive. API Gateway WebSocket hub subscribes and pushes events to the browser in real time. Notification Service subscribes and routes alerts to customer channels.
- **Payload:** `{deployment_id, state, phase, reason_code, observed_state, timestamp, correlation_id}`.

**`agents.heartbeat`**

- **Type:** Last-value-per-key (exactly one message per agent retained at any time).
- **Subject:** `agents.heartbeat.{dp_id}`.
- **Consumer:** Registry background process updates `last_heartbeat_at` and `capacity_score` in Aurora on each receive.
- **Payload:** `{dp_id, capacity_score, active_deployment_count, agent_version, timestamp}`.

### 9.2 Message ordering and idempotency

NATS JetStream guarantees per-subject ordering. Because deployment commands are keyed by `{dp_id}.{deployment_id}`, all commands for a given deployment arrive at the agent in the order they were published. Combined with the Agent's version-check deduplication, this ensures the deployment converges to exactly the latest desired state even if multiple commands were queued during an outage.

## 10. Audit Logger

Every mutating action on the Control Plane is recorded before the action is committed. The Audit Logger is middleware on the API Gateway — it does not wait for responses from downstream services.

### 10.1 Audit entry schema

Each audit entry records:

| Field | Type | Example |
|---|---|---|
| `audit_id` | UUID | `a1b2c3d4-...` |
| `org_id` | UUID | The organisation the action occurred in. |
| `workspace_id` | UUID | The workspace, if applicable. |
| `project_id` | UUID | The project, if applicable. |
| `deployment_id` | UUID | The deployment, if applicable. |
| `actor_id` | UUID | User or service account that performed the action. |
| `actor_type` | Enum | `HUMAN`, `SERVICE_ACCOUNT`, `SYSTEM`. |
| `source_ip` | String | The client IP from the `X-Forwarded-For` header (after WAF/CloudFront stripping). |
| `action` | String | `deployment.create`, `deployment.dag.deploy`, `connection.update`, `member.role.set`, etc. |
| `resource_type` | String | The type of resource being acted on. |
| `resource_id` | UUID | The specific resource ID. |
| `before_state` | JSONB | Snapshot of the resource state before the action (null for creates). |
| `after_state` | JSONB | Snapshot of the resource state after the action (null for deletes). |
| `result` | Enum | `SUCCESS`, `FAILURE`, `DENIED`. |
| `correlation_id` | UUID | Request correlation ID, matching the `X-Correlation-ID` header and log traces. |
| `timestamp` | Timestamp | UTC, microsecond precision. |

### 10.2 Immutability and retention

Audit entries are written in append-only mode. There is no UPDATE or DELETE path for audit records — the Aurora row is inserted with a write-only service account. Application code cannot overwrite audit history.

Entries are stored in Aurora for 90 days. At the end of each month, the previous month's entries are exported to S3 in Parquet format partitioned by `org_id` and `month`. The Aurora rows are dropped after a confirmed successful S3 export. The S3 archive is retained for seven years to support SOC 2, ISO 27001, and customer compliance requirements.

Customers can query their own organisation's audit log via the API (`GET /v1/orgs/{id}/audit-log`). Flowdeck staff access the raw S3 archive via Athena for cross-customer analysis and incident investigation.

## 11. Billing Service

The Billing Service converts raw usage data into invoices. It runs on a dedicated schedule — billing aggregation is not on the hot path of any customer-facing API.

### 11.1 Usage ingestion

The Data Plane's Metrics Collector aggregates billable metrics hourly and pushes them to the Control Plane via the same gRPC status stream used for health updates. The Billing Service receives these events from `deployments.status` (a separate consumer group) and writes them to the `usage_events` table in Aurora.

**Billable dimensions:**

| Metric | Unit | How measured |
|---|---|---|
| Worker uptime | Core-hours | Worker pod CPU requests × pod running time. |
| Scheduler uptime | Instance-hours | Per-deployment scheduler pod running time. |
| DAG run count | Count | Total DagRun records created in a billing period. |
| Task execution count | Count | Total TaskInstance records completed (SUCCESS + FAILED). |
| Storage | GB-hours | RDS storage GB × hours + S3 objects GB × hours. |

### 11.2 Invoice generation

Monthly invoices are generated on the first day of the month for the previous month's usage. The Billing Service:

1. Aggregates hourly `usage_events` into per-deployment monthly totals.
2. Applies the organisation's pricing plan (per unit rates, included quantities, volume discounts).
3. Creates a Stripe invoice with line items for each billable dimension, tagged by workspace and deployment for customer cost allocation.
4. Marks the invoice as `FINALIZED` and stores it in Aurora.
5. Emails the invoice to the organisation's billing contacts.

### 11.3 Spend caps

Each organisation can set a monthly spend cap. When cumulative spend (calculated from ingested usage in real time) reaches 80% of the cap, the billing owner is notified. At 100%, new deployment creates are blocked and the owner is notified again. Existing deployments continue running — blocking them would break production for a billing misconfiguration.

## 12. Notification Service

The Notification Service routes platform events to customer-configured channels. It subscribes to `deployments.status` (for health events) and to internal Control Plane events (for billing and access events).

### 12.1 Routing configuration

Routing rules are configured at the workspace level. Each workspace can define one or more notification channels:

| Channel type | Configuration | What it supports |
|---|---|---|
| Email | SES from-address, list of recipient addresses | All event classes |
| Slack | Incoming webhook URL | All event classes |
| PagerDuty | Events v2 API routing key | `UNHEALTHY`, `OFFLINE`, `ERROR` states only |
| HTTP webhook | Target URL, optional shared secret for HMAC-SHA256 signature | All event classes |

### 12.2 Event classes

| Class | Events | Default routing |
|---|---|---|
| `DEPLOYMENT_HEALTH` | DEGRADED, UNHEALTHY, HEALTHY (recovery), OFFLINE | Workspace notification channels |
| `DEPLOYMENT_LIFECYCLE` | RUNNING, STOPPED, TERMINATED, ERROR, DAG_PARSE_FAILED | Workspace notification channels |
| `BILLING` | Spend cap threshold reached, invoice ready | Organisation billing contacts |
| `SECURITY` | New API key created, role granted, SSO login from new device | Workspace admin and org owner |
| `PLATFORM` | Data Plane degraded or offline (affects multiple customers) | All affected workspace admins |

### 12.3 Delivery and retry

Notifications are sent with at-least-once delivery. HTTP webhook and PagerDuty deliveries are retried up to 5 times with exponential backoff (2, 4, 8, 16, 32 minutes). After 5 failures, the notification is marked `FAILED` and logged. Email delivery uses SES with bounce and complaint handling; hard-bouncing addresses are suppressed automatically.

## 13. Data layer — Aurora, Redis, S3

### 13.1 Aurora PostgreSQL

Aurora PostgreSQL 15 in Multi-AZ configuration. One writer instance and two reader instances across three availability zones.

**Connection management:** PgBouncer in transaction mode runs as a sidecar to API Gateway pods. This allows many short-lived application connections (each API request) to share a small pool of Aurora connections. Without this, connection storms during high-traffic periods would exhaust Aurora's `max_connections`. PgBouncer is configured with `pool_size = 25` per application instance and `max_client_conn = 1000`.

**Read/write split:** Read-heavy workloads (deployment listing, audit log queries, metrics aggregation) are explicitly routed to Aurora read replicas. Write operations (desired state changes, audit inserts, billing events) go to the writer. This is enforced in the application code by using separate connection pools pointing to the writer and reader endpoints.

**Key tables:**

| Table | Schema | Primary workload |
|---|---|---|
| `organisations` | org_id, slug, plan, region, quota_json, billing_id | Read-heavy on listing, write on membership changes |
| `workspaces` | workspace_id, org_id, config_json | Read-heavy |
| `projects` | project_id, workspace_id, target_deployment_id, manifest_jsonb | Read-heavy; write on project creation and target change |
| `dag_id_registry` | deployment_id, dag_id, project_id, registered_at — PK (deployment_id, dag_id) | Write on every DAG deploy; read on conflict check |
| `deployments` | deployment_id, project_id, desired_state_jsonb, desired_state_version, state | Write on every state change |
| `deployment_observed_states` | deployment_id, observed_state_jsonb, last_updated_at | Write on every Data Plane status update |
| `deployment_events` | deployment_id, from_state, to_state, actor_id, reason_code, correlation_id, timestamp | Append-only |
| `data_planes` | dp_id, type, region, health, capacity_score, last_heartbeat_at | Write on every heartbeat |
| `users` | user_id, org_id, email, password_hash, totp_secret, status | Read-heavy |
| `api_keys` | key_id, service_account_id, key_hash, scopes, expires_at | Read on every API request |
| `audit_log` | audit_id, org_id, actor_id, action, resource_id, before_state, after_state, timestamp | Append-only, partitioned monthly |
| `usage_events` | event_id, deployment_id, metric, value, period_start | Write on each hourly rollup from Data Plane |
| `refresh_tokens` | token_id, user_id, token_hash, used_at, expires_at | Write on each token rotation |

**Encryption:** All Aurora data is encrypted at rest with KMS customer-managed keys (one key per AWS account). In-transit encryption is enforced via SSL for all Aurora connections.

**Backup:** Automated daily snapshots with 7-day PITR. Monthly snapshots exported to S3 for long-term retention. All backups encrypted with the same KMS key.

### 13.2 ElastiCache Redis

Redis 7 in Cluster mode, 3 shards, 1 replica per shard. Used for short-lived, high-throughput data that would be expensive to put in Aurora:

| Use case | Key pattern | TTL | Notes |
|---|---|---|---|
| JWT revocation blocklist | `jti:{jti_hash}` | Token `exp` time | Checked on every authenticated request |
| Rate limit counters | `ratelimit:{user_id}:{window}` | 60 seconds | Atomic INCR for sliding window |
| RBAC decision cache | `rbac:{user_id}:{resource_id}` | 5 minutes | Invalidated by role change pub/sub |
| Distributed locks | `lock:{resource_type}:{resource_id}` | 5 seconds | SET NX for quota check-and-create |
| WebSocket session state | `ws_session:{session_id}` | 5 minutes | Short-lived WebSocket token validation |
| NATS stream cursor cache | `nats_cursor:{dp_id}` | None | Cached offsets for fast replay |

All Redis traffic is TLS-encrypted in transit. Redis data is encrypted at rest using ElastiCache at-rest encryption. Redis is not used for any data that cannot be reconstructed from Aurora — losing Redis state means a brief authentication and rate-limit cache miss, not data loss.

### 13.3 S3

S3 holds anything that should persist beyond the lifetime of a database row or that is too large to store in Aurora efficiently:

| Bucket | Contents | Retention |
|---|---|---|
| `flowdeck-audit-archive` | Monthly Parquet exports of the `audit_log` table | 7 years |
| `flowdeck-terraform-state` | Terraform remote state files for Data Plane infrastructure | Versioned; retained indefinitely |
| `flowdeck-backups` | Aurora and Redis backup exports | 90 days |
| `flowdeck-cli-releases` | CLI binary releases | Indefinite |
| `flowdeck-platform-config` | Periodic snapshots of org/workspace/deployment configs for DR | 90 days |

All buckets are encrypted with KMS server-side encryption. Public access is blocked at the account level. All bucket policies enforce `aws:SecureTransport` (HTTPS only).

## 14. AWS account and network topology

### 14.1 Account strategy

Flowdeck operates inside an AWS Organisation with separate accounts:

| Account | Purpose |
|---|---|
| `flowdeck-prod-control` | Control Plane services, Aurora, Redis, NATS, audit data |
| `flowdeck-prod-runtime` | Data Plane and Runtime Plane EKS clusters, customer deployments |
| `flowdeck-security` | GuardDuty, Security Hub, CloudTrail archive, AWS Config aggregator |
| `flowdeck-observability` | Long-term metrics (AMP), log archive (Loki / S3), traces (XRAY / Grafana Tempo) |
| `flowdeck-staging` | Mirror of production topology for integration testing |
| `flowdeck-dev` | Per-engineer sandbox environments |

Service Control Policies (SCPs) at the Organisation level enforce guardrails that no individual account administrator can override:
- No creation of long-lived static IAM access keys.
- No public S3 buckets.
- No disabling of CloudTrail or GuardDuty.
- KMS encryption required for Aurora, S3, and ElastiCache.
- Deployment only to approved regions.

### 14.2 Network topology

The Control Plane VPC (`10.0.0.0/16`) and the Runtime VPC (`10.10.0.0/16`) are **not peered**. The only path between them is the public Control Plane API endpoint, which the Data Plane reaches via outbound HTTPS/gRPC. This is intentional and enforces the outbound-only model at the network layer, not just at the application layer.

Within the Control Plane VPC:
- **Public subnets** (one per AZ): ALB, NAT Gateway. Nothing else.
- **Private subnets** (one per AZ): EKS control plane nodes, Aurora, ElastiCache Redis, NATS StatefulSet.
- **VPC interface endpoints**: S3 (gateway), KMS, Secrets Manager, STS, ECR, CloudWatch Logs. AWS service traffic never leaves the VPC over the internet.

Security group rules on the Control Plane EKS cluster:
- Inbound: only from the ALB security group, on the specific ports used.
- No inbound rules permitting direct internet access to any pod.
- Outbound: allowed to the Aurora and Redis security groups on their respective ports. AWS service endpoints via the VPC endpoints. No open outbound internet.

## 15. Reliability and disaster recovery

### 15.1 Redundancy

Every Control Plane component runs with at least two replicas across at least two availability zones. Pod Disruption Budgets ensure that rolling updates and node drains always leave at least one replica running. The EKS control plane itself is managed by AWS with multi-AZ masters.

Aurora Multi-AZ provides synchronous replication to a standby; automatic failover typically completes in under 60 seconds. ElastiCache Redis in Cluster mode has one replica per shard; failover is automatic and typically completes within 30–60 seconds with cluster mode enabled.

NATS JetStream's 3-node Raft cluster tolerates one node failure without availability or data loss impact. NATS replication factor is set to 3 for all critical streams — a message is only committed (and acked to the publisher) after all 3 nodes confirm persistence.

### 15.2 Regional strategy

**Phase 1** (current): One primary region (`us-east-1`) with a warm-standby secondary (`us-west-2`) for disaster recovery. The warm standby holds an Aurora read replica (promoted to writer on failover), a Redis cluster (repopulated from Aurora on failover), and a minimal EKS footprint ready to receive traffic. Route 53 health checks detect primary region failure and redirect DNS to the secondary endpoint within 60 seconds.

**Phase 2** (roadmap): Multiple primary regions for data residency. Each customer organisation is bound to a home region at creation. All customer data — desired state, audit events, observed state — stays within that region's boundary. The Control Plane runs in every primary region but with per-customer region affinity on writes.

### 15.3 Recovery time and point objectives

| Scenario | RTO | RPO |
|---|---|---|
| Single pod failure | < 30 seconds (Kubernetes reschedule) | 0 (stateless pods) |
| Single AZ failure | < 60 seconds (Aurora and Redis failover) | < 1 second (Aurora synchronous standby) |
| Full primary region failure | < 15 minutes (DNS failover + secondary region warm-up) | < 5 minutes (Aurora read replica lag bound) |

## 16. Security posture

### 16.1 Principle of least privilege

Every service in the Control Plane runs with a Kubernetes ServiceAccount bound to a minimal IAM role via IRSA. The IAM roles grant access only to the specific AWS resources that service needs:
- API Gateway: no AWS service access (all data via in-cluster connections).
- Audit Logger: `s3:PutObject` on the audit archive bucket only.
- Billing Service: `ses:SendEmail` for invoice delivery, read access to the usage S3 bucket.
- Terraform operator (Data Plane provisioning): scoped to `flowdeck-prod-runtime` account resources only, via cross-account role assumption.

No service has `AdministratorAccess`. Long-lived IAM access keys do not exist in any Flowdeck account (enforced by SCP).

### 16.2 Secret management

All secrets used by Control Plane services (database passwords, NATS credentials, the JWT signing private key) are stored in AWS Secrets Manager and injected into pods at startup via the AWS Secrets and Config Provider (ASCP) CSI driver. Secrets never appear in Kubernetes ConfigMaps, Helm values, or environment variables visible in the EKS console.

The JWT signing key pair is rotated quarterly. The rotation window is managed by the Identity Service: a new key pair is generated, the new public key is added to the JWKS endpoint (alongside the old one for the 1-hour overlap window), and the new private key is used for all new token issuances. After 1 hour, the old public key is removed from the JWKS endpoint — any tokens signed with the old key are now unvalidatable and will be rejected, which is safe because no valid token issued with the old key can have an `exp` in the future.

### 16.3 Encryption in transit

- All external traffic: TLS 1.2 minimum, TLS 1.3 preferred. TLS 1.0 and 1.1 are disabled on the ALB listener.
- gRPC to Data Plane Agents: mTLS (both sides present certificates). Minimum TLS 1.3.
- Internal pod-to-pod traffic: TLS enforced for Aurora, Redis, NATS, and any inter-service gRPC calls. Kubernetes pod-to-pod traffic within the cluster uses Cilium's transparent encryption (WireGuard) to encrypt traffic even within the VPC.

### 16.4 Supply chain security

All Control Plane container images are built by Flowdeck's CI pipeline, signed with cosign, and verified by a Kyverno admission policy before any pod is scheduled on the Control Plane EKS cluster. Images from registries other than Flowdeck's ECR are blocked. Base images are updated weekly and scanned with Trivy on every build; critical CVEs block the build.

## 17. Operational failure modes

### 17.1 Aurora writer failure

**What happens:** Aurora automatic failover promotes the standby to writer in under 60 seconds. During failover, write operations fail with `connection refused` or `read-only transaction`. PgBouncer retries connections during the failover window. API requests that hit write operations during the ~10–30 second failover window receive `503 Service Unavailable` with a `Retry-After` header.

**Customer impact:** API write operations (deployment creates, DAG deployments, connection updates) fail for the duration of the failover window. Read operations (listing deployments, viewing status) continue against the read replica. Existing running deployments are not affected — they are managed by the Data Plane and do not require the Control Plane to be writable to keep running.

**Recovery:** Automatic. PgBouncer reconnects to the new writer endpoint after Aurora updates the cluster endpoint in Route 53.

### 17.2 Redis failure

**What happens:** If the Redis leader for a shard fails, the shard's replica is promoted in under 60 seconds. During failover:
- The JWT revocation blocklist is briefly unavailable — revoked tokens may be accepted for the failover window. This is a known and accepted trade-off; the rate of attacker exploitation of a ~30-second window is lower than the cost of making revocation synchronous with Aurora.
- Rate limiting counters are lost for the failed shard. The effective rate limit is reset for the affected users. This is safe — it means slightly more requests are allowed temporarily, not fewer.
- RBAC cache entries on the failed shard are lost. Requests fall through to Aurora for RBAC decisions for the affected users until the cache repopulates (~5 minutes).

**Recovery:** Automatic. ElastiCache handles shard failover transparently.

### 17.3 NATS cluster degraded

**What happens:** If one NATS node fails, the 2-node Raft quorum can still commit and deliver messages (with replication factor 3, a message needs all 3 nodes to confirm durability; if one is down, messages queue until it recovers or is replaced). Message publishing from the Deployment Service blocks until the NATS cluster is healthy or times out.

**Customer impact:** Deployment creates and updates that require a new NATS publish are delayed or time out (the API returns `503` after a 10-second wait). In-flight commands already in the stream continue to be delivered to Agents. Health events from the Data Plane continue to flow (separate stream).

**Recovery:** Replace the failed NATS pod. The StatefulSet controller respawns it on a healthy node. NATS Raft re-syncs the rejoining node from the leader's snapshot.

### 17.4 API Gateway pod failure

**What happens:** Kubernetes detects the failed pod within 5 seconds and removes it from the ALB target group. Traffic is routed to the remaining pods. The HPA may spin up a replacement if the remaining pods are under CPU pressure.

**Customer impact:** Zero to minimal. The ALB load-balances across healthy pods. Requests in-flight on the failed pod return connection errors to the client; well-behaved clients (UI and CLI) retry.

### 17.5 Data Plane disconnection

**What happens:** The gRPC stream from an Agent disconnects. The Control Plane marks the stream `DISCONNECTED`. Commands accumulate in NATS for the affected Data Plane. Health updates from that Data Plane stop arriving; after 60 seconds, the registry marks it `DEGRADED`.

**Customer impact:** Deployments on the disconnected Data Plane continue running — they are managed by the Data Plane's local state machine, which keeps running based on the last known desired state. New deployment creates and updates targeting this Data Plane are queued and delivered when the Agent reconnects. If the disconnection lasts more than 5 minutes (`OFFLINE` threshold), customers are notified.

**Recovery:** The Agent reconnects (either automatically after a network blip or after an operator restarts it). The stream is re-established. Accumulated commands are delivered and applied. Status updates are flushed. The Data Plane returns to `HEALTHY` after the first successful heartbeat.
