## 1. Executive Summary

### 1.1 Purpose

This document defines the target architecture for AirflowCloud's public cloud deployment on AWS and establishes the control model that must remain stable across future hybrid and private deployment modes. It describes how the platform is structured, how its components interact, and how it is extended to new deployment contexts without breaking the core contract between planes.

### 1.2 Architecture Goals

- Separate tenant intent, orchestration, and runtime execution into distinct planes with well-defined contracts between them.
- Keep the platform control surface centralised, auditable, and secure — always owned and operated by AirflowCloud regardless of where customer workloads run.
- Support deployment portability so that the same Control Plane API governs public cloud, hybrid, and private deployments without modification.
- Isolate each customer deployment so that failures, noisy neighbours, and security incidents do not spill across tenants.
- Provide consistent health, audit, metrics, and operational visibility across all layers and all deployment modes.

### 1.3 Three-Plane Layer Model

The platform is structured around three distinct planes, each with a clear and non-overlapping responsibility:

| Layer | Responsibility | Location |
|---|---|---|
| **Control Plane** | User identity, organisation and workspace state, billing, policies, audit logging, public APIs, and deployment intent (desired state) | AirflowCloud AWS account — always |
| **Data Plane** | Deployment orchestration and reconciliation, Helm lifecycle management, health monitoring, secret distribution, and secure outbound communication with the Control Plane | AirflowCloud VPC (Public Cloud mode) or customer VPC (Hybrid/Private mode) |
| **Runtime Plane** | Actual Airflow workloads — webserver, scheduler, workers, triggerer, metadata database, Celery broker, DAG storage, and task execution | Same environment as the Data Plane for the chosen mode |

The critical design principle is that the Control Plane never runs Airflow workloads, and the Data and Runtime planes never make decisions — they only execute what the Control Plane instructs and report back what is actually running.

### 1.4 Deployment Modes

| Mode | Data Plane Location | Runtime Plane Location | Who Operates Infrastructure |
|---|---|---|---|
| **Public Cloud** | AirflowCloud VPC | AirflowCloud VPC | AirflowCloud operates everything |
| **Hybrid** | Customer VPC | Customer VPC | AirflowCloud manages Control Plane; customer hosts execution |
| **Private** | Customer VPC | Customer VPC | Customer operates everything under AirflowCloud's platform contract |

### 1.5 Communication Model

The Control Plane holds the source of truth for desired state. It sends orchestration commands to the appropriate Data Plane, which applies them to the Runtime Plane and continuously reports actual state back. All Data Plane communication is initiated outbound from the Data Plane toward the Control Plane — the Control Plane never opens inbound connections into a Data Plane. This outbound-only model ensures the architecture works identically for Data Planes that live in customer VPCs behind firewalls.

### 2.2 Key Design Decisions

**Why separate the Data Plane from the Control Plane?**
The Control Plane is AirflowCloud's business logic — RBAC, billing, audit, user management. The Data Plane is pure infrastructure orchestration. Keeping them separate means the Data Plane can be relocated to a customer's VPC in Hybrid mode without changing the Control Plane at all. The contract between them (a gRPC API over mTLS) remains identical regardless of where the Data Plane runs.

**Why separate the Runtime Plane from the Data Plane?**
The Data Plane is the orchestrator (runs Helm, watches Kubernetes, manages lifecycle). The Runtime Plane is the workload (runs Airflow). Keeping them in separate concerns means the Data Plane can manage multiple Runtime Plane namespaces simultaneously and health-check them independently. It also means future Runtime Plane variation (different executors, remote execution agents) does not require changing the orchestration logic.

**Why outbound-only communication?**
Requiring the Data Plane to initiate all connections outward to the Control Plane means that in Hybrid and Private modes, customers never need to open inbound firewall rules to their VPC. This is a hard requirement for regulated enterprise customers and simplifies network security posture significantly.

**Why one shared EKS cluster per tier rather than one cluster per customer?**
For Starter and Pro tier customers, a shared Runtime Plane EKS cluster with namespace-level isolation (ResourceQuota, NetworkPolicy, IRSA, Secrets Manager path scoping) provides sufficient isolation at dramatically lower operational cost. For Enterprise tier customers, a dedicated EKS cluster per organisation provides hard blast-radius isolation. Hybrid mode customers use their own clusters and are therefore always cluster-isolated by definition.

### 2.3 Tenant Isolation Model by Tier

| Tier | Isolation Level | Shared Resources | Dedicated Resources |
|---|---|---|---|
| Starter | Namespace | EKS cluster, node pools, ALB | Kubernetes namespace, ResourceQuota, IRSA role |
| Pro | Namespace + dedicated node pool | EKS cluster, ALB | Namespace, node pool (tainted), IRSA role |
| Enterprise | Dedicated cluster | Nothing | Full EKS cluster, ALB, RDS |
| Hybrid | Customer's own cluster | Nothing (customer owns it) | Full customer VPC and cluster |

### 2.4 AWS Account Structure

AirflowCloud operates separate AWS accounts within an AWS Organisation to enforce billing isolation, IAM blast-radius containment, and compliance boundaries:

- **airflowcloud-prod** — Production Control Plane and Public Cloud Data/Runtime Planes
- **airflowcloud-staging** — Staging environment, mirrors production topology
- **airflowcloud-dev** — Development environment for engineering
- **airflowcloud-security** — Centralised security tooling: GuardDuty aggregator, Security Hub, CloudTrail archive, AWS Config

Service Control Policies (SCPs) at the Organisation level enforce guardrails that cannot be overridden even by account administrators — for example, preventing creation of static IAM access keys.

### 3.1 Overview

### 3.3 Ingress Layer

All external traffic enters the Control Plane through a layered ingress stack designed to protect against abuse before requests reach application services.

**AWS WAF v2** sits at the outermost perimeter. It enforces OWASP Top 10 rules (SQL injection, XSS, path traversal), applies rate limiting at the IP and organisation level, and blocks traffic from known malicious IP ranges. The WAF processes every request before it reaches either CloudFront or the ALB.

**Amazon CloudFront** serves the AirflowCloud web UI as a global CDN. It caches static assets (JavaScript bundles, CSS, images) at edge locations worldwide, reducing latency for international users. CloudFront terminates TLS using a wildcard certificate managed by AWS Certificate Manager and forwards dynamic API requests to the ALB. CloudFront also provides AWS Shield Standard for DDoS mitigation at no additional cost.

**AWS Application Load Balancer (ALB)** handles REST API and gRPC traffic. Two separate target groups handle HTTP/2 REST traffic (port 8080 on API Server pods) and gRPC traffic (port 9090 on API Server pods). The ALB terminates TLS using an ACM wildcard certificate (`*.airflowcloud.io`) and routes based on host header — `api.airflowcloud.io` for REST, `grpc.airflowcloud.io` for gRPC. The ALB also exposes the WebSocket endpoint (`wss://ws.airflowcloud.io`) for real-time deployment status and log streaming.

### 3.4 API Gateway Service

The API Gateway Service is the single entry point for all inbound communication. It is a stateless service deployed as a Kubernetes Deployment with three replicas on the Control Plane EKS cluster, scaled horizontally by HPA based on CPU utilisation and request rate.

**REST API** is available at `https://api.airflowcloud.io/v1`. It uses JSON over HTTPS, requires Bearer token or API key authentication on every request, and conforms to an OpenAPI 3.0 schema. All requests are validated against this schema before reaching downstream services — malformed requests are rejected at the gateway with a 400 response and never touch core services.

**gRPC API** is available at `grpcs://grpc.airflowcloud.io:443`. It is used exclusively by Data Plane Agents for lifecycle communication. It uses mTLS in addition to JWT tokens — the Data Plane Agent must present a valid client certificate signed by AirflowCloud's internal CA, and must also supply a valid JWT in the `authorization` metadata field. This dual-layer authentication ensures that neither a stolen JWT nor a stolen certificate alone is sufficient to impersonate a Data Plane.

**WebSocket** is available at `wss://ws.airflowcloud.io`. It provides real-time updates to the UI: deployment status transitions, live log streaming, and alert notifications. WebSocket connections are authenticated on upgrade using a short-lived WebSocket token (separate from the API JWT, valid for 5 minutes) issued by the Identity Service.

**Rate limiting** is applied at three levels: per IP address (WAF layer, 1,000 requests per minute), per user (API Gateway layer, 300 requests per minute), and per organisation (API Gateway layer, 5,000 requests per minute). Rate limit counters are stored in ElastiCache Redis and are shared across all API Gateway replicas. Burst allowances are applied so that short legitimate spikes are not rejected.

**Circuit breaker** logic is applied on calls from the API Gateway to downstream services (Identity Service, Deployment Service, etc.). If a downstream service returns errors on more than 50% of requests over a 10-second window, the circuit opens and the API Gateway returns a 503 with a `Retry-After` header. This prevents cascade failures where one degraded service causes the entire API to queue up requests.

**API versioning** is managed via URL path prefixes (`/v1`, `/v2`). The API Gateway routes each version to the appropriate service version. Breaking changes are never made within a version — a new version is released and the previous version is maintained for a minimum of 12 months before deprecation.

### 3.5 Core Services

#### 3.5.1 Identity Service

The Identity Service is responsible for all authentication and authorisation decisions across the platform. Every inbound request to the API Gateway is validated by the Identity Service before reaching any other service.

**Authentication methods** supported:

- **Email and Password with MFA:** Passwords are stored as bcrypt hashes (cost factor 12). Multi-factor authentication uses TOTP (RFC 6238 compatible with Google Authenticator, Authy, etc.). On login, a short-lived access token (JWT, RS256, 1-hour expiry) and a long-lived refresh token (opaque, stored hashed in the platform database, 30-day expiry) are issued. The refresh token is rotated on each use.
- **SSO via SAML 2.0 and OIDC:** AirflowCloud acts as a Service Provider. Any SAML 2.0 or OIDC-compliant Identity Provider (Okta, Microsoft Entra ID, Google Workspace, Auth0) is supported. User attributes (email, name, groups) are mapped from the IdP assertion into AirflowCloud's user model. Group mappings can be configured to automatically assign workspace roles based on IdP group membership.
- **API Keys:** Long-lived opaque keys generated for CLI and CI/CD automation. Stored as SHA-256 hashes in the platform database. Scoped to an organisation and optionally to specific workspaces. Displayed only once at creation time. API keys can be rotated or revoked without affecting user sessions.
- **Data Plane Service Tokens:** Short-lived JWTs (24-hour expiry, auto-rotated) issued to Data Plane Agents at registration time. These tokens are separate from user tokens and are scoped to allow only the gRPC endpoints used by Data Planes (desired state fetch, status report, heartbeat). They cannot be used to call user-facing REST endpoints.

**Authorisation model** is hierarchical RBAC:

- **Platform scope** — super-administrator role used internally for platform operations. Not exposed to customers.
- **Organisation scope** — `org:owner` (full control including billing), `org:admin` (manage workspaces and users, no billing), `org:member` (access to assigned workspaces only).
- **Workspace scope** — `workspace:admin` (full workspace control), `workspace:developer` (deploy and manage DAGs, create connections and variables), `workspace:viewer` (read-only).
- **Deployment scope** — Fine-grained access to individual deployments, used for CI/CD service accounts that should only be able to deploy to one specific deployment.

Roles are additive and inherited downward. An `org:owner` has implicit admin access to all workspaces and deployments in their organisation. Role assignments are stored in the platform database and cached in Redis for 5 minutes to reduce database load on high-frequency permission checks.

**JWT structure:** Access tokens carry `sub` (user ID), `org_id`, `roles` (array of role assignments with scope), `scopes` (permitted API scopes), `jti` (unique token ID for revocation), `iat`, and `exp`. Tokens are signed with RS256 using a 4096-bit RSA key pair. The public key is exposed at `https://api.airflowcloud.io/.well-known/jwks.json` for third-party validation.

**Token revocation:** Revoked tokens (on logout, password change, or security incident) are added to a Redis blocklist keyed by `jti`. Every request validates the token signature and checks the blocklist. Blocklist entries expire at the token's natural expiry time so the list does not grow unboundedly.

#### 3.5.2 Organisation Service

The Organisation Service manages the top-level tenant hierarchy. An Organisation is the billing and policy boundary — it corresponds to a company or team. Every user belongs to one or more Organisations.

When a new organisation is created, the service provisions the following automatically: a default workspace, a quota record defining the maximum number of deployments and total resource allocation for the organisation's plan tier, and a billing record linked to a Stripe customer ID. Organisation slugs are globally unique and URL-safe — they appear in API paths and as subdomain prefixes for customer Airflow UIs.

The Organisation Service also manages **teams** — logical groups of users within an organisation that can be assigned workspace or deployment roles collectively. When a user is added to a team, they inherit all the team's role assignments. Role changes on a team propagate to all members.

**Quotas** are enforced by the Organisation Service and checked by the Deployment Service before any resource creation. Quota limits cover: maximum number of active deployments, total CPU allocation across all deployments, total memory allocation, and total storage. Quota checks are optimistic (check-then-act) with a distributed lock held in Redis during the check-and-create window to prevent race conditions when multiple deployments are created simultaneously.

#### 3.5.3 Workspace Service

The Workspace Service manages workspaces within an organisation. A workspace is a logical grouping of deployments, typically corresponding to a team or environment (e.g., "data-engineering", "ml-platform", "production").

Each workspace has its own member list and role assignments, independent of the organisation member list. This allows an organisation administrator to give a user access to only specific workspaces. The workspace also holds shared configuration: default Airflow version for new deployments, allowed executor types, and CI/CD enforcement flag.

**CI/CD enforcement** is a workspace-level setting that, when enabled, prevents deployments from being modified through the UI or CLI by users — only service account API tokens with the `DEPLOYER` role can trigger deploys. This enforces the practice that all changes to production environments go through a pipeline.

#### 3.5.4 Deployment Service

The Deployment Service is the central orchestrator of deployment lifecycle. It holds the desired state for every deployment and is responsible for translating user intent into commands dispatched to the Data Plane.

**Desired state** is a JSON document stored in the platform database that describes exactly what the deployment should look like: Airflow version, executor type, resource profile (CPU/memory limits per component), environment variables, connection references, and variable references. Whenever a user updates a deployment, the Deployment Service writes the new desired state and publishes a command to the Data Plane Registry to synchronise.

**Deployment state machine:** Every deployment transitions through a well-defined set of states. `PENDING` means the request has been received and validated. `PROVISIONING` means the Data Plane is creating AWS infrastructure (RDS, ElastiCache, EFS). `CONFIGURING` means Helm is installing Airflow components. `STARTING` means Airflow pods are starting and health checks are running. `RUNNING` means the deployment is healthy and serving traffic. `UPDATING` means a configuration change is being applied. `SCALING` means worker count is being adjusted. `STOPPING` and `STOPPED` represent graceful shutdown. `ERROR` is a recoverable failure state that prompts an alert. `TERMINATING` and `TERMINATED` represent deletion. State transitions are recorded in the audit log with timestamps and the actor (user or system) that triggered them.

**Commands dispatched to the Data Plane** are one of: `CREATE`, `UPDATE`, `DELETE`, `RESTART`, or `SCALE`. Commands are published to the NATS `deployments.commands` stream with the deployment ID and the complete desired state document. The Data Plane Agent consumes these commands, applies them, and publishes status updates back to the `deployments.status` stream.

**Idempotency:** Every command carries a `command_id` (UUID) and a `desired_state_version` (monotonically incrementing integer). The Data Plane Agent tracks the last applied `desired_state_version` per deployment and ignores commands with a version equal to or lower than what it has already applied. This prevents reprocessing of duplicate NATS deliveries.

#### 3.5.5 Data Plane Registry

The Data Plane Registry tracks all Data Plane instances — both managed (AirflowCloud-operated) and hybrid (customer-operated). It is the routing layer that determines which Data Plane receives commands for a given deployment.

Each Data Plane record contains a unique ID, display name, type (`MANAGED`, `HYBRID`, `PRIVATE`), AWS region, gRPC endpoint, current health status (`HEALTHY`, `DEGRADED`, `OFFLINE`), current and maximum deployment capacity, last heartbeat timestamp, and cloud account metadata.

**Routing logic:** When the Deployment Service creates a command, it calls the Data Plane Registry to resolve the target Data Plane for the deployment. The registry looks up the Data Plane assigned at deployment creation time, verifies its health status is `HEALTHY`, and returns the gRPC endpoint. If the Data Plane is `DEGRADED`, the command is still forwarded but an alert is raised. If the Data Plane is `OFFLINE`, the command is queued in NATS until the Data Plane reconnects, rather than failing immediately.

**Capacity management:** The registry tracks how many deployments each Data Plane is currently running. When a new deployment is created, the registry assigns it to the Data Plane in the appropriate region with the most available capacity. In the future, rack-aware scheduling and custom affinity rules will be supported to allow enterprise customers to pin deployments to specific Data Planes.

**Health tracking:** Data Plane Agents send heartbeats to the registry every 15 seconds via the gRPC `Heartbeat` RPC. Each heartbeat carries the agent version, Kubernetes version, available cluster resources, and per-deployment health summaries. If no heartbeat is received for 60 seconds, the Data Plane is marked `DEGRADED`. If no heartbeat is received for 5 minutes, it is marked `OFFLINE` and all affected customers are notified.

#### 3.5.6 Event Bus (NATS JetStream)

AirflowCloud uses NATS JetStream as the internal event bus between the Control Plane and all Data Plane Agents. NATS JetStream provides durable, ordered, acknowledged message delivery with persistence to disk, meaning events are not lost if a consumer is temporarily disconnected.

NATS is deployed as a 3-node StatefulSet on the Control Plane EKS cluster, using EBS gp3 volumes for persistence. The cluster uses the NATS Raft-based quorum protocol for leader election and log replication.

**Key streams and their purpose:**

`deployments.commands` carries orchestration commands from the Deployment Service to Data Plane Commanders. This stream uses a work-queue retention policy — once a consumer acknowledges a message, it is deleted from the stream. Each command is acknowledged only after the Data Plane has begun executing it (not after it completes, since execution can take minutes). If the agent crashes mid-execution, NATS re-delivers the command to the next available agent consumer.

`deployments.status` carries status updates from Data Plane Agents to the Control Plane. This stream uses a limits retention policy, keeping the last 100 status updates per deployment subject. The Deployment Service subscribes to this stream and updates the platform database and triggers UI notifications when status changes arrive.

`agents.heartbeat` carries heartbeat signals from Data Plane Agents to the Data Plane Registry. This stream uses limits retention, keeping only the last heartbeat per agent subject. The registry uses these to maintain the health state of each Data Plane.

`alerts.platform` carries alert notifications from Alertmanager instances (on both Control and Data Planes) to the Control Plane Notification Service, which routes them to the appropriate customer channels.

Data Plane Agents connect to NATS over TLS using their mTLS client certificate. The NATS server validates the certificate against AirflowCloud's internal CA before allowing the connection. Agents always initiate the outbound connection to NATS — the NATS cluster never reaches into a customer's network.

#### 3.5.7 Billing Service

The Billing Service tracks resource consumption and generates invoices. It integrates with Stripe for payment processing.

Usage metrics are consumed from the Data Plane Metrics Collector, which sends hourly aggregated billing events to the `billing.usage` NATS stream. Each billing event contains the deployment ID, organisation ID, and resource consumption totals for the period: worker-hours (sum of worker pod uptime across all worker nodes), DAG run count, task execution count, and storage GB-hours (EFS and S3).

The Billing Service aggregates these events into monthly invoices and syncs usage to Stripe for metered billing. It also enforces spend limits — if an organisation's projected monthly bill exceeds a configured threshold, it can automatically pause new deployment creation and notify the organisation owner.

The Billing Service exposes usage dashboards to organisation owners through the API, showing cost breakdown by workspace, deployment, and resource type.

#### 3.5.8 Notification Service

The Notification Service routes alerts and platform events to customer-configured destinations. Supported channels are email (AWS SES), Slack (webhook), PagerDuty (Events API v2), and generic HTTP webhooks.

Customers configure notification channels at the workspace level. Alert routing rules determine which events trigger which channels — for example, a deployment entering `ERROR` state might trigger PagerDuty, while a billing threshold alert triggers email only.

The service subscribes to the `alerts.platform` NATS stream and also receives direct calls from the Deployment Service for state change notifications. Notifications are templated per event type and per channel — the Slack message format is different from the email format but carries the same information.

#### 3.5.9 Audit Logger

Every mutating action in the platform (user login, deployment creation, configuration change, role assignment, API key creation, etc.) produces an audit log entry. The Audit Logger is a middleware component that intercepts all API Gateway responses and writes audit entries to the platform database before returning the response to the caller.

Each audit entry records: organisation ID, workspace ID (if applicable), actor user ID or service account ID, actor IP address, action type (structured string like `deployment.create`), resource type, resource ID, before/after state snapshots for mutations, request timestamp, and response status. Entries are immutable once written.

Audit log entries are partitioned monthly in the platform database for query performance. Entries older than 90 days are exported to S3 (in Parquet format for Athena queries) and deleted from the database. S3 audit archives are retained for 7 years for compliance purposes.

### 3.6 Control Plane Data Layer

#### 3.6.1 Platform Database (RDS Aurora PostgreSQL)

The platform database is an Amazon RDS Aurora PostgreSQL 15 cluster deployed in Multi-AZ configuration with one writer instance and two reader instances. The writer instance handles all write operations from core services. Reader instances handle read-heavy queries such as deployment listing, metrics aggregation, and audit log retrieval.

Connection pooling is provided by a PgBouncer StatefulSet deployed alongside the API services. PgBouncer runs in transaction mode, allowing many application connections to share a small pool of actual Aurora connections. This prevents Aurora from being overwhelmed during traffic spikes — without PgBouncer, a surge of API requests would each open their own database connection, rapidly exhausting Aurora's connection limit. Each API Server pod has a PgBouncer sidecar configured with a pool of 20 server connections, giving a total Aurora connection count of roughly `20 × (API Server replicas)` regardless of how many concurrent user requests are in flight.

All data is encrypted at rest using KMS customer-managed keys (CMK). Automated backups are retained for 7 days with point-in-time recovery (PITR) enabled. Monthly snapshots are exported to S3 for long-term retention. Enhanced Monitoring and Performance Insights are enabled for query-level profiling.

**Key tables:** `organisations`, `workspaces`, `deployments` (desired state + synced actual status), `data_planes`, `users`, `organisation_members`, `workspace_members`, `api_keys`, `audit_logs`, `billing_events`, `notification_channels`, `alert_rules`.

The `deployments` table stores desired state as JSONB, allowing flexible schema evolution without database migrations for most configuration changes. The actual deployment status is synced from the Data Plane and stored as a separate `status` column and `endpoints` JSONB column, making clear which data is authoritative (desired = Control Plane, actual = Data Plane report).

#### 3.6.2 Session Cache and Rate Limiting (ElastiCache Redis)

ElastiCache Redis is used for short-lived, high-throughput data that does not need database durability:

- **JWT blocklist** — revoked token JTIs stored with TTL equal to the token's remaining lifetime
- **Rate limit counters** — per-user and per-organisation request counts, using Redis atomic increment operations
- **Session data** — server-side session state for WebSocket connections
- **Role assignment cache** — RBAC decisions cached per user for 5 minutes to reduce database load
- **Quota check locks** — distributed locks held during the check-and-create window for deployment creation

Redis is deployed as a 3-node cluster in Cluster mode for horizontal scalability. Encryption in transit (TLS) and at rest (AWS-managed key) are enabled.

#### 3.6.3 S3 Artifact Storage

The Control Plane uses S3 for durable object storage:

- **Audit log archives** — exported monthly from the database in Parquet format for long-term retention and Athena queries
- **Terraform state** — remote state for all infrastructure provisioned by the Data Plane's Terraform runner, stored in a versioned S3 bucket with DynamoDB state lock table
- **CLI release artifacts** — versioned binaries for all supported platforms and operating systems
- **Platform configuration snapshots** — periodic exports of platform configuration for disaster recovery

### 4.3 Data Plane Agent

The Data Plane Agent is the bridge between the Control Plane and the Data Plane's internal components. It is deployed as a Kubernetes Deployment in the `airflowcloud-system` namespace and is the only component in the Data Plane that communicates directly with the Control Plane.

**Outbound gRPC connection:** The Agent maintains a persistent, bidirectional gRPC stream to the Control Plane API Gateway. This stream is used for command delivery (Control Plane pushes commands to the Agent), status reporting (Agent pushes status updates to the Control Plane), and heartbeat signals. The connection is authenticated with mTLS (the Agent presents its client certificate) plus a JWT bearer token in the gRPC metadata. The Agent always initiates the connection outbound — it dials the Control Plane gRPC endpoint, never the reverse.

**Command consumption:** The Agent subscribes to the NATS `deployments.commands` stream for commands destined for deployments assigned to this Data Plane. When a command arrives, the Agent validates its `desired_state_version` against the locally tracked version, then dispatches it to the Deployment Controller. The Agent only acknowledges the NATS message after the Deployment Controller has accepted the command for processing (not after completion), ensuring at-least-once delivery even if the Agent restarts mid-execution.

**Status reporting:** The Agent publishes status updates to the NATS `deployments.status` stream whenever a deployment's state changes. Status updates include the deployment ID, the new state, a human-readable message, Kubernetes endpoint information (the Airflow webserver URL), and a timestamp. The Agent also sends a status update at startup to report the current actual state of all deployments it manages, ensuring the Control Plane re-synchronises after an Agent restart.

**Heartbeat:** Every 15 seconds, the Agent sends a heartbeat to the Control Plane's Data Plane Registry via the `agents.heartbeat` NATS stream. The heartbeat carries the agent version, Kubernetes version, total and available CPU and memory across all nodes, and a summary of each managed deployment's health.

**mTLS certificate lifecycle:** At registration time, the Agent receives an initial client certificate signed by AirflowCloud's internal PKI (implemented via HashiCorp Vault's PKI secrets engine). Certificates have a 30-day validity period. The Agent monitors its certificate's remaining lifetime and initiates rotation 7 days before expiry. Rotation involves calling the Control Plane's certificate rotation API with the current certificate as proof of identity, receiving a new certificate, swapping it into the gRPC connection, and confirming successful rotation. Old certificates are immediately revoked in Vault after successful rotation.

**Token rotation:** The Agent's JWT service token is automatically rotated every 24 hours. The Agent requests a new token from the Identity Service using its current valid token, receives the new token, updates its in-memory and Kubernetes Secret storage, and confirms the rotation. Tokens have a 30-minute overlap period during which both old and new tokens are valid to handle clock drift and slow rotation handoffs.

**Agent state machine:**

```
REGISTERING → (certificate issued, NATS connected) → HEALTHY
HEALTHY → (heartbeat missed for 60s) → DEGRADED
HEALTHY → (command received) → RECONCILING → HEALTHY
DEGRADED → (reconnected) → HEALTHY
DEGRADED → (missed for 5 min) → OFFLINE → (alert fires)
```

### 4.4 Deployment Controller

The Deployment Controller receives commands from the Data Plane Agent and executes them by orchestrating a sequence of infrastructure provisioning and Kubernetes resource management steps.

**Three-phase provisioning for CREATE:**

Phase 1 is **Infrastructure provisioning via Terraform**. The Deployment Controller invokes a Terraform workspace for the deployment, applying a module that creates the deployment's dedicated AWS resources: an RDS PostgreSQL instance for the Airflow metadata database, an ElastiCache Redis cluster for the Celery broker (if using CeleryExecutor), and an EFS file system for DAG storage (or an S3 bucket for S3-backed DAG storage). Terraform state is stored in S3 with a DynamoDB lock table, keyed by deployment ID, so it can be re-applied safely if the provisioning step is interrupted. Each resource is tagged with the deployment ID, workspace ID, and organisation ID for cost allocation. Terraform runs as an in-cluster Job (not a long-running process) — it is invoked, completes, and exits.

Phase 2 is **Kubernetes resource deployment via Helm**. Once Terraform completes and outputs the database endpoint, Redis endpoint, and EFS mount target, the Deployment Controller renders a Helm values file from the deployment's desired state and the Terraform outputs. It then runs `helm install` (for CREATE) or `helm upgrade --atomic` (for UPDATE) of the `airflowcloud-airflow` chart into the deployment's namespace. The `--atomic` flag on upgrades means that if the upgrade fails, Helm automatically rolls back to the previous release — the Deployment Controller does not need to implement its own rollback logic. The Helm release name is the deployment ID, and the namespace is `ac-dep-{deployment-id}`.

Phase 3 is **Validation**. After Helm reports the release as deployed, the Deployment Controller waits for all Airflow component pods (webserver, scheduler, workers, triggerer) to reach `Ready` state, verifies the webserver's `/health` endpoint returns HTTP 200, and checks that the scheduler is heartbeating by querying the Airflow metadata database's `job` table. Once all checks pass, it extracts the webserver's URL from the Kubernetes Ingress resource and reports the `RUNNING` status with endpoints to the Data Plane Agent.

**State management:** The Deployment Controller maintains a ConfigMap in the `airflowcloud-system` namespace for each deployment, recording the last successfully applied `desired_state_version`, the Terraform workspace status, and the Helm release revision. This ConfigMap is the local ground truth — if the Deployment Controller restarts, it reads these ConfigMaps to reconstruct its in-memory state without needing to query the Control Plane.

**Reconciliation loop:** The Deployment Controller runs a reconciliation loop every 60 seconds. For each managed deployment, it compares the `desired_state_version` in the ConfigMap with the latest version received from NATS. If they differ, the controller applies the delta. If they are the same, it checks whether the actual Kubernetes state (Helm release, pod count, pod health) matches the desired state — if drift is detected (e.g., a pod was manually deleted), it re-applies the Helm release to restore the desired state. This loop makes the system self-healing without requiring explicit reconcile commands from the Control Plane.

**Update flow:** For configuration changes (e.g., adding an environment variable, changing a resource profile), the Deployment Controller runs `helm upgrade --atomic` with the new values. Airflow component pods are replaced with a rolling update — the scheduler and webserver are updated one pod at a time, and workers are drained of running tasks before being replaced (using Airflow's graceful shutdown mechanism). Downtime is typically zero for webserver updates and less than 30 seconds for scheduler updates (during which new DAG runs are queued but not started).

**Delete flow:** Deletion is a two-step process. First, the Deployment Controller runs `helm uninstall` to delete all Kubernetes resources in the namespace. Then it runs `terraform destroy` to delete the RDS instance, ElastiCache cluster, and EFS file system. RDS has a final snapshot taken before deletion (configurable — can be disabled for development deployments). After both steps complete, the namespace is deleted and the status is reported as `TERMINATED`.

### 4.5 Health Monitor

The Health Monitor runs independent of the Deployment Controller. It continuously probes the health of every deployment managed by this Data Plane and maintains a per-deployment health status that the Data Plane Agent includes in its heartbeats.

**Probe schedule:** Health probes run every 30 seconds per deployment. The probe set for each deployment includes:

- **Webserver probe:** HTTP GET to the Airflow webserver's `/health` endpoint. Expects HTTP 200 within 5 seconds. A failed probe increments a failure counter; three consecutive failures change the component status to `UNHEALTHY`.
- **Scheduler probe:** Checks the Kubernetes pod status of the scheduler pod (must be `Running` with `Ready` condition true) and verifies that the scheduler's heartbeat in the Airflow metadata database has been updated within the last 60 seconds.
- **Worker probes:** Checks the pod status of each worker pod. Validates that the Celery queue depth (fetched from the Redis broker) is not growing unboundedly (a sign that workers are not picking up tasks).
- **Database probe:** Validates database connectivity by executing a lightweight query (`SELECT 1`) against the Airflow metadata RDS instance through a connection from the Deployment Controller's connection pool.
- **Broker probe:** Executes a Redis PING command against the ElastiCache endpoint to validate broker availability.

**Health aggregation:** Individual component health is aggregated into a deployment-level status. `HEALTHY` means all components pass. `DEGRADED` means at least one non-critical component is failing (e.g., one worker pod is in CrashLoopBackOff but others are healthy). `UNHEALTHY` means a critical component is failing (scheduler or webserver down — Airflow cannot function in this state).

**Auto-recovery:** The Health Monitor attempts automated recovery before alerting. For a failed worker pod, it deletes the pod (Kubernetes will reschedule it). For a failed scheduler pod, it first attempts a pod restart. If the scheduler remains unhealthy for more than 5 minutes, the Health Monitor escalates — it triggers a full `helm upgrade` to re-apply the last known-good Helm values, which will recreate all pods. If this does not resolve the issue within 10 minutes, the deployment is marked `ERROR` and an alert is sent via the Notification Service.

### 4.6 Secret Distributor

The Secret Distributor manages the lifecycle of secrets within each deployment's Kubernetes namespace. Secrets are never stored in plaintext in the Data Plane — they are always fetched from HashiCorp Vault (or AWS Secrets Manager in simpler configurations) and written to Kubernetes Secrets with encryption at rest (EKS Secrets envelope encryption using KMS).

**Secrets managed per deployment:**

- **Database credentials** — Username and password for the Airflow metadata RDS instance. Generated uniquely per deployment using Vault's database secrets engine, which issues short-lived dynamic credentials (rotated every 24 hours). The Deployment Controller configures Vault with the RDS master credentials at provisioning time, and all subsequent credential issuance happens through Vault's dynamic engine.
- **Celery broker credentials** — Redis AUTH password for the ElastiCache cluster, rotated every 30 days.
- **Fernet encryption key** — Used by Airflow to encrypt connection passwords and XCom values at rest in the metadata database. Generated once at deployment creation time and never rotated (rotating the Fernet key requires re-encrypting all connections). Stored in Vault's KV secrets engine.
- **Customer Airflow connections** — When a user creates an Airflow connection through the AirflowCloud API, the connection URI (including credentials) is stored in Vault at a path scoped to the deployment. The Secret Distributor syncs these to a Kubernetes Secret in the deployment namespace, and Airflow is configured to read connections from Kubernetes Secrets rather than the metadata database.
- **Container registry credentials** — ECR pull secrets, rotated every 12 hours (ECR tokens expire on a 12-hour cycle).

**Sync process:** The Secret Distributor runs as a recurring Job every 30 seconds. It compares the versions of all secrets in Vault against the versions stored in the Kubernetes Secrets in the deployment namespace. For any secret whose Vault version is newer than the Kubernetes Secret version, it fetches the new secret value from Vault and updates the Kubernetes Secret. Airflow picks up updated secrets on pod restart — for credential rotation, a rolling restart of the affected components is triggered automatically.

### 4.7 Metrics Collector

The Metrics Collector aggregates and forwards metrics from the Runtime Plane to two destinations: the Control Plane (for billing and platform-wide dashboards) and the local Prometheus instance (for operational monitoring).

**Airflow metrics** are emitted by Airflow's built-in StatsD client. The Deployment Controller configures each Airflow deployment to point its StatsD output at a per-deployment StatsD exporter sidecar, which translates StatsD metrics to Prometheus format and exposes them on a `/metrics` endpoint. Prometheus scrapes these endpoints every 15 seconds. Key Airflow metrics collected include: task duration by DAG and task ID, task failure counts, DAG parsing time, scheduler heartbeat age, executor queue depth and slot utilisation, pool slot usage, and DAG run duration by status.

**Infrastructure metrics** are collected via Prometheus's Kubernetes service discovery — it scrapes Node Exporter (CPU, memory, disk I/O per node), kube-state-metrics (pod status, deployment replica counts, resource requests and limits vs. actual usage), and the Airflow RDS instance via RDS Enhanced Monitoring metrics published to CloudWatch, then scraped by the CloudWatch exporter.

**Billing metrics** are computed by the Metrics Collector from the raw Prometheus data on an hourly schedule. Worker-hours are computed as the integral of worker pod count over time. Storage usage is computed from EFS and S3 metrics. Computed billing events are published to the `billing.usage` NATS stream for the Billing Service to consume.

**Forwarding to Control Plane Prometheus:** The local Prometheus instance in the Data Plane is configured with remote write to the Control Plane's Thanos Receive endpoint, sending a curated subset of metrics (deployment-level aggregates, health status metrics, billing metrics) for cross-deployment visibility in the AirflowCloud operations dashboard.

### 4.8 DAG Bundle Management

AirflowCloud supports two DAG delivery strategies, and the Deployment Controller manages both:

**Image-based DAG bundles (recommended):** The customer builds a Docker image that extends the AirflowCloud Airflow base image and includes their DAG files, plugins, and Python dependencies in the image layer. This image is pushed to ECR. The Deployment Controller updates the Airflow Helm values to use the new image tag and runs `helm upgrade`. All Airflow components restart with the new image, ensuring consistent DAG code across all pods. Image builds are triggered by the `airflowcloud deploy` CLI command, which builds the image using the customer's local Docker daemon, authenticates to ECR using the customer's credentials, pushes the image, and then calls the Deploy API to trigger the Helm upgrade. A DAG parse check runs before the push — if any DAG fails to import, the build is rejected before the image is pushed.

**Git-sync based DAG delivery:** A `git-sync` sidecar container runs alongside the Airflow scheduler and worker pods. It continuously polls a configured Git repository for changes and syncs DAG files to a shared volume that all Airflow components read from. This approach does not require an image build for DAG changes, making it faster for teams with frequent DAG iterations. However, it does not version Python dependencies — dependency changes still require an image rebuild. Git-sync is configured with a repository URL, branch, polling interval (default 60 seconds), and deploy key (SSH private key stored as a Kubernetes Secret by the Secret Distributor).

**EFS vs. S3 for DAG storage (Git-sync mode):** EFS is used when DAG files are synced to a shared filesystem that all pods mount. S3 is used as an alternative where git-sync writes directly to an S3 bucket and Airflow's DAG bag reads from S3 using the S3 remote logs plugin. S3-backed DAG storage is preferred for large deployments with many DAG files because it eliminates the need to provision and manage EFS throughput.

### 5.3 Airflow Components

**Webserver (Airflow API Server in Airflow 3.x):** Runs the Airflow UI and REST API. Deployed as 1 replica for Starter/Pro tiers, 2 replicas in active-active mode for Enterprise tier. The webserver is stateless — it reads state from the metadata database and does not hold any local state. It is exposed through a Kubernetes Ingress resource (using the AWS Load Balancer Controller to provision an ALB) at `https://{deployment-slug}.airflow.airflowcloud.io`. Authentication on the Airflow UI is handled by Airflow's built-in auth using credentials stored in the metadata database, or by SSO using the customer's IdP if configured.

**Scheduler:** Parses DAG files, evaluates schedules and triggers, creates task instances, and submits them to the executor. Deployed as 1 replica for standard deployments, 2 replicas in High Availability mode for Enterprise tier (using Airflow's built-in HA scheduler with database-level leadership election). The scheduler pod runs with the highest pod priority class to ensure it is not evicted by resource pressure that would instead evict lower-priority worker pods. The scheduler's DAG parsing performance is monitored via the `dag_processing.total_parse_time` metric — if parsing time exceeds 60 seconds for the full DAG bag, a warning alert fires.

**Workers (CeleryExecutor):** Execute task instances assigned by the scheduler via the Celery message queue. Workers are the most resource-intensive and variable component — they scale based on task queue depth. Each worker pod runs a Celery worker process, and each worker can run multiple tasks concurrently up to its configured `worker_concurrency` limit (default: 4 tasks per worker). Workers are deployed on Spot Instance node groups in AWS to reduce cost. Worker pods have a PodDisruptionBudget that allows at most 50% of workers to be unavailable simultaneously, preventing all workers from being evicted at once during a Spot interruption event. When a Spot interruption notice arrives (2-minute warning), the worker gracefully finishes its current tasks (up to 2 minutes) and then exits.

**Workers (KubernetesExecutor):** When a deployment uses the KubernetesExecutor, each task runs as an individual short-lived Kubernetes Pod rather than in a long-running Celery worker. The scheduler creates a pod per task, monitors it, and cleans it up on completion. There are no persistent worker pods — only the scheduler and webserver are long-running. This executor is better suited for workloads with heterogeneous resource requirements per task, since each task pod can request its own CPU and memory.

**Triggerer:** Handles deferrable operators — tasks that pause execution while waiting for an external event (e.g., a sensor waiting for an S3 file or an HTTP endpoint to become available). The triggerer runs asynchronous event loops to monitor these conditions, consuming minimal resources while waiting. When a condition is met, the triggerer signals the scheduler to resume the task. Deployed as 1 replica per deployment.

### 5.4 Worker Autoscaling (KEDA)

Worker autoscaling is managed by KEDA (Kubernetes Event-Driven Autoscaling) rather than the standard Kubernetes HPA. KEDA is specifically designed for event-driven workloads and can scale based on external metrics, including message queue depth.

For CeleryExecutor deployments, KEDA creates a `ScaledObject` resource for each deployment's worker Deployment. The scaler type is `aws-sqs-queue` (if using SQS-backed Celery) or a custom scaler using the Redis LIST length (if using Redis-backed Celery). The scale-up threshold is set so that one additional worker is added for every 5 pending tasks in the queue. The scale-down threshold is more conservative — a worker is removed only if the queue has been empty for at least 5 consecutive minutes, preventing rapid scale-up/scale-down oscillation.

Minimum worker count is 1 (to keep at least one worker ready for immediate task execution). Maximum worker count is configured per deployment based on the resource profile (default: 10 for medium, 20 for large). KEDA also manages scale-to-zero for development deployments — when no tasks have run for 30 minutes, workers scale to 0 and the Airflow webserver is hibernated, saving costs.

Node-level autoscaling for the Runtime Plane EKS cluster is managed by Karpenter. Karpenter watches for pods in `Pending` state (caused by insufficient cluster resources) and automatically provisions new EC2 instances of the appropriate type and size. Karpenter prefers Spot instances for worker node groups and falls back to On-Demand if Spot capacity is unavailable. It also consolidates the cluster by removing underutilised nodes when workloads can be bin-packed onto fewer nodes.

### 5.5 Namespace Isolation

Each deployment's Kubernetes namespace is strictly isolated from all other namespaces through a combination of controls:

**ResourceQuota** limits the total CPU, memory, and number of pods that can be created in the namespace. This prevents a single deployment from consuming all cluster resources and starving other deployments.

**LimitRange** sets default and maximum CPU and memory requests and limits for every pod created in the namespace. This ensures that pods without explicit resource requests do not run without limits.

**NetworkPolicy** implements a default-deny posture for the namespace. Ingress to namespace pods is allowed only from the ALB (for webserver traffic) and from other pods within the same namespace (for Airflow component-to-component communication). Egress is allowed to the namespace's RDS and ElastiCache endpoints, the EFS mount target, the ECR registry, and external internet (for tasks that call external APIs). Cross-namespace traffic between customer deployments is blocked at the network level.

**IRSA (IAM Roles for Service Accounts):** Each deployment has a dedicated IAM role bound to the Kubernetes Service Account for its namespace. This role grants access to only the specific Secrets Manager paths, S3 prefixes, and other AWS resources belonging to that deployment. A bug or security issue in one deployment's code cannot access another deployment's secrets because the IAM role boundary prevents it.

**Pod Security Standards:** All namespaces in the Runtime Plane cluster are configured with the `restricted` pod security standard, preventing privilege escalation, running as root, host path mounts, and other dangerous pod configurations.

### 5.6 Airflow Version Management and Upgrades

AirflowCloud maintains a curated set of supported Airflow versions, based on the official Apache Airflow release schedule. Each supported version is available as a base Docker image published to AirflowCloud's ECR registry with security patches applied. Customers select their Airflow version at deployment creation time.

**Minor version upgrades** (e.g., 2.8.0 → 2.8.1) are offered as optional updates in the UI and CLI. The upgrade process runs `helm upgrade` with the new Airflow image tag. Because minor versions are backward compatible, no database migration is required.

**Major version upgrades** (e.g., 2.x → 3.x) require a more careful process. The customer is notified 30 days in advance of end-of-support for their current major version. The upgrade wizard in the UI:

1. Validates that all DAGs in the deployment are compatible with the target version (using the Airflow upgrade check tool, run as a Job in the customer's namespace).
2. Creates a pre-upgrade snapshot of the Airflow metadata database.
3. Runs the Airflow database migration (`airflow db upgrade`) as a Job before restarting any components.
4. Upgrades the Helm chart to the new version with an `--atomic` flag (rolls back if any component fails to start).
5. Monitors for 10 minutes to confirm stability before marking the upgrade complete.

If the upgrade fails at any step, the database snapshot is restored and the deployment is rolled back to the previous version automatically.

### 5.7 Backup and Disaster Recovery

**Airflow Metadata Database:** Each deployment's RDS instance has automated daily backups with 7-day PITR retention. Backups are stored in S3 in the same AWS region. For Enterprise deployments, cross-region backup replication is enabled, storing backups in a secondary region (`us-west-2`) for regional disaster recovery. The target RPO (Recovery Point Objective) is 1 hour for standard deployments and 5 minutes for Enterprise (using Aurora with enhanced backup frequency).

**DAG Storage:** EFS file systems are backed up daily using AWS Backup. S3-backed DAG storage is inherently durable (11 nines of durability). Cross-region replication can be enabled for Enterprise deployments.

**Deployment Configuration:** Desired state for all deployments is stored in the platform database (Control Plane), which is backed up with PITR. Even if the entire Runtime Plane is lost, all deployment configurations can be replayed from the Control Plane to recreate all deployments from scratch.

**RTO targets:** For Runtime Plane failures (EKS node failure, pod crash), the Health Monitor's auto-recovery achieves RTO of less than 5 minutes through pod rescheduling. For a full Runtime Plane EKS cluster failure, deploying a new cluster and recreating all deployments from the Control Plane's desired state achieves RTO of 30–60 minutes, limited primarily by RDS restore time. For Enterprise deployments with a warm standby cluster, RTO is under 10 minutes.

### 6.3 DAG Deployment Flow (Git-Sync)

In git-sync mode, DAG changes do not trigger a Helm upgrade. Instead, the git-sync sidecar continuously polls the configured repository. When a new commit is pushed to the monitored branch:

1. The git-sync sidecar detects the new commit within the configured polling interval (default: 60 seconds).
2. It clones or fetches the new commit and updates the shared volume that the scheduler, workers, and webserver all read from.
3. The Airflow scheduler detects the changed DAG files and reloads the affected DAGs automatically (using the `file_modification_time` check built into Airflow's DagBag).
4. No pod restart is required unless the changes include new Python packages not already in the base image.

The trade-off is that git-sync provides faster DAG iteration (changes visible within ~60 seconds) but does not provide reproducible builds — two deployments at different times may have different transitive dependency versions if `requirements.txt` changes. Teams are encouraged to use image-based deployment for production environments where reproducibility matters.

## 7. Network Architecture

### 7.1 AWS Network Topology

AirflowCloud uses two VPCs in the primary region (`us-east-1`) — one for the Control Plane and one for the Data/Runtime Planes. These are connected via VPC Peering to allow internal communication between the planes without traversing the public internet.

```
AWS Region: us-east-1
│
├── Control Plane VPC (10.0.0.0/16)
│   ├── Public Subnets (3 AZs: 10.0.1.0/24, 10.0.2.0/24, 10.0.3.0/24)
│   │   └── ALB, NAT Gateways
│   ├── Private App Subnets (3 AZs: 10.0.11.0/24, 10.0.12.0/24, 10.0.13.0/24)
│   │   └── EKS Node Groups (Control Plane services)
│   └── Private DB Subnets (3 AZs: 10.0.21.0/24, 10.0.22.0/24, 10.0.23.0/24)
│       └── RDS Aurora (platform DB), ElastiCache Redis
│
├── Data/Runtime Plane VPC (10.1.0.0/16)
│   ├── Public Subnets (3 AZs: 10.1.1.0/24, 10.1.2.0/24, 10.1.3.0/24)
│   │   └── ALB (Airflow UI endpoints), NAT Gateways
│   ├── Private Subnets — Data Plane (3 AZs: 10.1.11.0/24-10.1.13.0/24)
│   │   └── Data Plane Agent, Deployment Controller, Health Monitor
│   ├── Private Subnets — Runtime Plane (3 AZs: 10.1.31.0/24-10.1.33.0/24)
│   │   └── Airflow pods (all customer namespaces)
│   └── Private DB Subnets (3 AZs: 10.1.41.0/24-10.1.43.0/24)
│       └── Per-deployment RDS instances, ElastiCache clusters
│
└── VPC Peering: Control Plane ↔ Data Plane (non-transitive)
```

**VPC Endpoints** are configured in both VPCs to allow AWS API calls without traversing the public internet or NAT Gateway: ECR API and DKR endpoints, S3 gateway endpoint, Secrets Manager interface endpoint, KMS interface endpoint, and CloudWatch Logs interface endpoint. This eliminates NAT Gateway costs for AWS API traffic and reduces latency.

### 7.2 DNS Architecture

All public DNS is managed in Amazon Route53 under the `airflowcloud.io` hosted zone:

- `api.airflowcloud.io` — A record pointing to the Control Plane ALB (API Server REST)
- `grpc.airflowcloud.io` — A record pointing to the Control Plane ALB (API Server gRPC)
- `app.airflowcloud.io` — CNAME pointing to CloudFront distribution (Web UI)
- `{deployment-slug}.airflow.airflowcloud.io` — CNAME records created per-deployment pointing to the Runtime Plane ALB

Private DNS hosted zones are configured within each VPC to resolve internal service names (e.g., `platform-db.internal.airflowcloud.io` resolves to the RDS writer endpoint).

### 7.3 Security Groups

Security groups enforce least-privilege network access between components. Every resource has its own security group with rules defined by role, not by CIDR.

**Control Plane ALB SG:** Inbound 443/tcp from `0.0.0.0/0` (public internet). Outbound to Control Plane App SG on ports 8080 and 9090.

**Control Plane App SG (API Server pods):** Inbound 8080 and 9090 from Control Plane ALB SG only. Outbound 5432 to Control Plane DB SG, 6379 to Control Plane Redis SG, 443 to 0.0.0.0/0 (AWS VPC endpoints and external calls), 4222 to NATS SG.

**NATS SG:** Inbound 4222/tcp (NATS client port) from Control Plane App SG and Data Plane Agent SG. Inbound 6222 (cluster routing) and 8222 (monitoring) from NATS SG only.

**Data Plane Agent SG:** No inbound (outbound-only connections to Control Plane). Outbound 443 to Control Plane ALB SG (gRPC over HTTPS), 4222 to NATS SG, all ports to Runtime Plane SG.

**Runtime Plane (Airflow pods) SG:** Inbound from Data Plane Agent SG (management). Inbound 8080 from Runtime Plane ALB SG (Airflow UI). Outbound 5432 to per-deployment RDS SG, 6379 to per-deployment Redis SG, 2049 to EFS SG, 443 to 0.0.0.0/0.

**Per-deployment RDS SG:** Inbound 5432 from Runtime Plane SG only. No outbound.

**Per-deployment ElastiCache SG:** Inbound 6379 from Runtime Plane SG only. No outbound.

### 7.4 Airflow UI Access

Customers access their Airflow UI through a deployment-specific URL: `https://{deployment-slug}.airflow.airflowcloud.io`. DNS for this URL resolves to the Runtime Plane ALB, which routes traffic to the Kubernetes Ingress for the specific deployment namespace. The Ingress is created by the Deployment Controller at deployment creation time using the AWS Load Balancer Controller, which automatically provisions an ALB listener rule and target group for each deployment's webserver pods.

TLS termination occurs at the ALB using the wildcard certificate `*.airflow.airflowcloud.io`. Traffic between the ALB and the Airflow webserver pods within the cluster uses HTTP (not HTTPS) since it is internal to the VPC.

## 8. Security Architecture

### 8.1 Security Layers

Security is implemented in six independent layers, so that a failure in any one layer does not expose the system.

**Layer 1 — Perimeter:** AWS WAF v2 filters malicious traffic before it reaches the application. CloudFront provides DDoS protection via AWS Shield Standard. AWS Shield Advanced is available for Enterprise accounts.

**Layer 2 — Authentication and Authorisation:** Every request requires a valid identity (JWT, API key, or mTLS certificate). RBAC is enforced on every API handler before any business logic executes. The Identity Service is the sole authority for all authentication decisions.

**Layer 3 — Network:** VPC isolation separates Control Plane from Data/Runtime Planes. Private subnets ensure no resource is directly reachable from the internet. Security groups enforce least-privilege at the network level. Kubernetes NetworkPolicies enforce namespace-level isolation.

**Layer 4 — Data:** All data is encrypted at rest using KMS CMK. All data is encrypted in transit using TLS 1.3 (no TLS 1.2 fallback in production). Secrets are never stored in plaintext and are managed through HashiCorp Vault with automatic rotation.

**Layer 5 — Runtime:** Kubernetes pod security standards (`restricted` profile) prevent privilege escalation. IRSA ensures each pod can access only its designated AWS resources. EKS Secrets envelope encryption protects Kubernetes Secrets at rest. GuardDuty monitors for anomalous behaviour at the EKS audit log and S3 data event levels.

**Layer 6 — Supply chain:** ECR container image scanning on push blocks images with CRITICAL CVEs. SBOM is generated for every release. Container images are signed using AWS Signer. Dependencies are scanned and automatically updated via Dependabot.

### 8.2 Secrets Management

AirflowCloud uses HashiCorp Vault as the central secrets management system for all credentials across the platform. Vault is deployed in HA mode (3-node cluster with Raft-based storage backend) on the Control Plane EKS cluster. Vault data is stored on encrypted EBS volumes.

**Vault secret engines:**

The **KV (Key-Value) engine** stores static secrets with version history: Fernet encryption keys per deployment, customer Airflow connections (encrypted at the application layer by Airflow before storage), internal API keys, and platform-level configuration.

The **Database engine** issues dynamic, short-lived credentials for RDS instances. When a component needs database credentials, it calls Vault with its service identity (Kubernetes service account token for in-cluster components, or AWS IAM identity for EC2-based components). Vault authenticates the identity, checks the policy, and generates a new username and password with a 24-hour TTL directly in the target RDS instance. Credentials are automatically revoked when they expire. No static database password is ever stored.

The **PKI engine** issues and manages X.509 certificates for mTLS communication between the Control Plane and Data Plane Agents. The root CA certificate is stored offline (not in Vault) and used only to sign the intermediate CA that Vault manages. Intermediate CA certificates are rotated every 6 months. Leaf certificates issued to agents have a 30-day validity period with automated 7-day pre-expiry rotation.

**Vault authentication methods:**

- **Kubernetes auth:** EKS pods authenticate to Vault by presenting their Kubernetes service account JWT token. Vault validates the token against the EKS OIDC endpoint. Policies are assigned based on the service account name and namespace.
- **AWS IAM auth:** Used for Terraform and other AWS-native processes. Vault validates the IAM identity using the STS `GetCallerIdentity` API.
- **AppRole:** Used for CI/CD systems that run outside Kubernetes or AWS. Role IDs are stored as CI/CD secrets; secret IDs are generated per pipeline run and immediately revoked after use.

### 8.3 mTLS for Control Plane — Data Plane Communication

All gRPC communication between the Control Plane and Data Plane Agents uses mutual TLS (mTLS). Both sides present certificates and validate each other.

The Control Plane presents a certificate signed by AirflowCloud's public CA (validated by the Data Plane Agent's trust store, which contains the CA bundle). The Data Plane Agent presents a certificate signed by AirflowCloud's internal CA (validated by the Control Plane using Vault's certificate verification). This means that even if an agent token is compromised, an attacker cannot impersonate a legitimate Data Plane Agent without also having a valid client certificate signed by AirflowCloud's internal CA.

## 9. Infrastructure as Code

### 9.1 Approach

All AirflowCloud infrastructure is defined as code using Terraform, organised with Terragrunt for DRY multi-environment configurations. Every infrastructure change goes through a pull request review and automated plan verification before being applied. Manual console changes are blocked via AWS Config rules and SCPs.

### 9.2 Repository Structure

The infrastructure repository is structured to separate reusable modules from environment-specific configurations. Modules define the building blocks (EKS cluster, RDS instance, VPC, IAM roles). Environment configurations (prod, staging, dev) reference these modules with environment-specific variable values. Terragrunt handles the DRY configuration by providing shared remote state configuration and common variable injection without duplicating Terraform code across environments.

Terraform state is stored in S3 in the `airflowcloud-prod` account, encrypted with KMS, with a DynamoDB table providing state locking to prevent concurrent applies. State is partitioned by environment and component (e.g., `prod/us-east-1/eks-control/terraform.tfstate`).

### 9.3 Per-Deployment Infrastructure

The infrastructure for each customer deployment (RDS, ElastiCache, EFS) is provisioned by the Deployment Controller using Terraform programmatically. The controller maintains a Terraform workspace per deployment, identified by the deployment ID. Workspaces are isolated and cannot interfere with each other. Terraform state for per-deployment infrastructure is stored in the same S3 bucket under a `deployments/{deployment-id}/` prefix.

The Deployment Controller invokes Terraform by spawning a Terraform binary as a subprocess (run as a Kubernetes Job), passing the deployment's parameters as Terraform variables and the workspace-specific backend configuration. Terraform outputs (RDS endpoint, Redis endpoint, EFS mount target) are read back by the controller and used to construct the Helm values file for the subsequent Airflow installation.

### 9.4 GitOps for Platform Applications

AirflowCloud platform applications (API Server, NATS, Prometheus, Grafana, Loki, Vault) are deployed to EKS using Flux CD in a GitOps pattern. Flux watches the infrastructure repository's `apps/` directory and automatically applies Helm releases and Kubernetes manifests when changes are merged to the `main` branch. This means application deployments are reviewed via pull request, automatically tested in staging before production, and fully auditable through git history.

Airflow deployment Helm releases are managed by the Deployment Controller (not Flux). The Deployment Controller is the authoritative reconciler for customer workloads and does not use GitOps — it applies Helm releases imperatively in response to commands from the Control Plane.

## 10. Observability

### 10.1 Metrics Stack

**Prometheus** is deployed per EKS cluster (one on the Control Plane cluster, one on the Runtime Plane cluster). Each Prometheus instance uses Kubernetes service discovery to automatically scrape metrics from all pods that expose a `/metrics` endpoint, annotated with `prometheus.io/scrape: "true"`.

**Thanos** provides long-term metrics storage and cross-cluster querying. Each Prometheus instance is configured with the Thanos sidecar, which uploads metric blocks to S3 every 2 hours. The Thanos Querier component on the Control Plane cluster can query metrics from both the Control Plane and Runtime Plane Prometheus instances simultaneously, providing a unified view of the entire platform. Metrics are retained in S3 for 1 year.

**Grafana** is deployed on the Control Plane cluster and queries Thanos as its data source. Two dashboards are maintained: a platform operations dashboard (visible to AirflowCloud SREs) showing all deployments across all customers, and a customer dashboard (visible to organisation owners) showing their own deployments' metrics. The customer dashboard is filtered by `org_id` label, enforced at the Grafana datasource level.

**Key Airflow metrics tracked per deployment:**

- DAG processing total parse time (alerts at > 60 seconds)
- Scheduler heartbeat age (alerts at > 30 seconds — indicates scheduler is stuck)
- Task queue depth (used by KEDA for autoscaling decisions)
- Executor open slots vs. queued tasks
- DAG run success and failure rates
- Task instance duration by DAG and task ID
- Pool slot utilisation

### 10.2 Logging Stack

**Fluent Bit** is deployed as a DaemonSet on all EKS clusters. It tails all container log files on every node, adds Kubernetes metadata (namespace, pod name, container name, deployment ID label), and forwards logs to OpenSearch. Fluent Bit is configured with a buffer to handle OpenSearch unavailability — logs are held in memory (and optionally on disk) for up to 5 minutes before dropping.

**OpenSearch** (deployed on the Control Plane cluster, backed by EBS gp3 volumes and an S3 snapshot repository) provides log indexing, full-text search, and retention management. Logs are indexed per deployment — each deployment gets an index named `airflow-{deployment-id}-YYYY.MM.DD`. Indices are retained for 30 days on hot storage (EBS) and then transitioned to warm storage (S3 snapshots) for an additional 60 days. After 90 days, logs are deleted.

**OpenSearch Dashboards** provides the log search UI. Customers access their deployment's logs through the AirflowCloud UI (which proxies OpenSearch queries filtered by deployment ID), or through the `airflowcloud deployment logs` CLI command.

**Structured logging** is enforced for all AirflowCloud platform components (API Server, Deployment Controller, Health Monitor, etc.). All log entries are JSON-structured with fields: `timestamp`, `level`, `service`, `trace_id`, `deployment_id` (where applicable), `org_id` (where applicable), and `message`. This enables precise log correlation and filtering.

### 10.3 Distributed Tracing

OpenTelemetry SDK is used in all AirflowCloud platform services for distributed tracing. Traces are exported to AWS X-Ray in Phase 1, with a migration to a vendor-neutral OTEL collector in Phase 3. Each API request generates a trace that spans the API Gateway, Identity Service, Deployment Service, and (asynchronously correlated via trace ID) the Data Plane Agent's command execution. This allows SREs to trace a slow deployment creation from the user's API call all the way to the Helm install completion.

Trace IDs are propagated in API responses as the `X-Trace-Id` header, allowing customers to reference a specific request when reporting issues to support.

### 10.4 Alerting

**Alertmanager** on the Control Plane receives alerts from both the Control Plane and Runtime Plane Prometheus instances (remote-written to the Control Plane). It deduplicates, groups, and routes alerts to the appropriate notification channel.

**Platform-level alerts** (visible to AirflowCloud SREs only): Control Plane API latency > 500ms for 5 consecutive minutes, NATS stream lag > 1,000 messages, Vault sealed or unreachable, EKS node group at 90% capacity, RDS storage > 80% full.

**Deployment-level alerts** (visible to customers via the Notification Service): Deployment status transitions to `ERROR`, scheduler heartbeat age > 30 seconds, task failure rate > 20% in the last hour, worker CPU or memory > 90% for 10 minutes, DAG parsing time > 60 seconds.

Alert silencing rules allow customers to silence non-critical alerts during maintenance windows via the API or UI.

## 11. Extensibility

### 11.1 Extension Architecture Principle

The three-plane architecture (Control Plane, Data Plane, Runtime Plane) is the stable contract. Extensions to new deployment modes work by changing where the Data Plane and Runtime Plane run, not by changing the Control Plane. The gRPC contract between the Control Plane and Data Plane Agent is versioned and backward-compatible, meaning an older Agent version can still communicate with a newer Control Plane within the same major version.

### 11.2 Hybrid Mode Extension

In Hybrid mode, the Data Plane Agent Helm chart is installed by the customer in their own EKS cluster. The agent connects outbound to the Control Plane's gRPC endpoint. From the Control Plane's perspective, a Hybrid Data Plane is identical to a Managed one — it registers, sends heartbeats, and processes commands through the same interfaces.

Additional work required for Hybrid mode:

- The Agent registration flow in the Control Plane UI (already described in Section 6.6)
- AWS PrivateLink VPC Endpoint Service, so customers who require private connectivity can avoid the agent connecting over the public internet
- Documentation and Terraform module for customer IAM setup (the agent needs permissions to create RDS instances, ElastiCache clusters, and manage EKS namespaces in the customer's account)
- Secret backend pluggability in Config Syncer (customers use their own Vault or AWS Secrets Manager rather than AirflowCloud's Vault)

No changes to the Control Plane services are required for Hybrid mode — it is purely a Data Plane and network configuration change.

### 11.3 Remote Execution Extension

Remote Execution is a future mode where the Airflow Scheduler and Metadata DB remain on AirflowCloud's infrastructure but task execution happens in the customer's environment. This is enabled by Airflow 3's Task Execution API, which decouples workers from direct database access.

In Remote Execution mode, a lightweight Worker Agent (not the full Data Plane Agent) is installed in the customer's environment. It heartbeats to the AirflowCloud API Server advertising available task slots, receives task assignments via the Task Execution API, executes tasks locally (with access to customer data, secrets, and network), and reports task results back via HTTP without the task data ever leaving the customer's environment.

Work required: Task Execution API proxy in the AirflowCloud API Server, Worker Agent Helm chart, DAG Processor Agent (serialises DAG graph without sending Python code to AirflowCloud), and PrivateLink support for private worker connectivity.

### 11.4 Private Cloud Extension

Private Cloud mode packages the entire AirflowCloud Control Plane for installation in a customer's own AWS account (or GCP or Azure account). A `airflowcloud-installer` CLI handles bootstrapping the Control Plane on the customer's Kubernetes cluster. An offline license key (signed JWT with embedded feature flags and expiry) is validated locally without requiring network connectivity to AirflowCloud.

The same Terraform modules, Helm charts, and application code are used in Private Cloud mode as in the public cloud — only the license validation mode and the absence of outbound telemetry to AirflowCloud differ.

Air-gap support is provided by the installer's `generate-image-bundle` command, which produces a tarball of all required container images that the customer can load into their private registry without internet access.

### 11.5 Multi-Cloud Extension

The Data Plane Agent and all Data Plane components are written without AWS SDK dependencies. AWS SDK usage is confined to the cloud provider abstraction layer (Terraform modules and the Secret Distributor's backend implementations). To support GCP or Azure as a Data Plane host, new Terraform modules and a new secrets backend implementation are added without modifying any core Data Plane or Control Plane logic.

The Control Plane itself remains on AWS in all current extension paths. A multi-cloud Control Plane (running simultaneously on AWS, GCP, and Azure for regional data residency) is a future Phase 6 capability that requires Control Plane federation design.

### 11.6 Extension Points Summary

| Extension | What Changes | What Stays the Same |
|---|---|---|
| Hybrid mode | Data Plane runs in customer VPC | Control Plane, gRPC contract, all core services |
| Remote Execution | Thin Worker Agent in customer env | Control Plane, full Data Plane for scheduling |
| Private Cloud | Full platform in customer account | Same code, same Helm charts, same gRPC contract |
| Multi-cloud Data Plane | New Terraform modules + secrets backend | Agent code, Control Plane, gRPC contract |
| New executor types (Dask, Ray) | New executor Helm values + resource profiles | Data Plane reconciliation logic |
| Custom deployment sizes | New resource profiles in Deployment Service | Everything else |
| New Airflow versions | New base images in ECR | Everything else |

### 12.3 Glossary

| Term | Definition |
|---|---|
| **Control Plane** | Central management layer hosting the API, identity, billing, audit, and deployment intent (desired state). Always operated by AirflowCloud. |
| **Data Plane** | Infrastructure orchestration layer that provisions resources and manages Airflow deployment lifecycle. Runs in AirflowCloud VPC (managed) or customer VPC (hybrid/private). |
| **Runtime Plane** | The actual Airflow workload environment — webserver, scheduler, workers, triggerer, and their supporting databases and storage. Runs within the Data Plane's Kubernetes cluster. |
| **Data Plane Agent** | The component running in the Data Plane that maintains the outbound gRPC connection to the Control Plane, receives commands, and dispatches them to the Deployment Controller. |
| **Deployment Controller** | Component in the Data Plane responsible for orchestrating Terraform (infrastructure) and Helm (Kubernetes resources) for each customer Airflow deployment. |
| **Deployment** | A single Airflow environment — all Airflow components (webserver, scheduler, workers, triggerer) plus their supporting infrastructure (RDS, ElastiCache, EFS), running in a dedicated Kubernetes namespace. |
| **Workspace** | A logical grouping of deployments within an organisation, typically corresponding to a team or environment. |
| **Organisation** | The top-level tenant — corresponds to a company or team. All billing, user management, and quota enforcement happens at the organisation level. |
| **Desired State** | The JSON document stored in the Control Plane platform database describing what a deployment should look like. The source of truth for what the Data Plane should create and maintain. |
| **Actual State** | The current observed state of a deployment, reported by the Data Plane Agent from inspecting Kubernetes resources. Periodically synced back to the Control Plane. |
| **IRSA** | IAM Roles for Service Accounts — an AWS mechanism that binds a Kubernetes service account to an IAM role, allowing pods to call AWS APIs using the role's permissions without static credentials. |
| **KEDA** | Kubernetes Event-Driven Autoscaling — a Kubernetes component that scales workloads based on external event sources such as queue depth, rather than only CPU/memory metrics. |
| **Karpenter** | An AWS-native Kubernetes node autoscaler that provisions EC2 instances dynamically in response to pending pods, replacing the older Cluster Autoscaler. |
| **mTLS** | Mutual TLS — a TLS configuration where both the client and the server present certificates and verify each other's identity, rather than only the server presenting a certificate. |
